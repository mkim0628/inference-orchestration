## 한 줄 요약

EPIC은 임의 prefix를 가진 요청들 사이에서도 **불변 청크의 KV를 재계산 없이 재사용**할 수 있도록, 청크를 position 0부터 인코딩해 캐싱한 뒤(Compile), 각 청크의 첫 k≤32 토큰만 재계산해 attention sink 효과를 무력화(Link)함으로써 CacheBlend의 O(0.15·N²) 링크 비용을 **O(kN)** 으로 낮춘 PIC 시스템이다.

## 1. 문제 정의

표준 prefix 기반 KV 캐시(vLLM/SGLang RadixAttention, Anthropic prompt caching 포함)는 "두 요청의 prefix가 byte-identical"일 때만 재사용 가능하다. 그러나 실제 워크로드에서는:

- **Few-shot learning**: 시스템 프롬프트는 요청마다 다르지만 예시 블록은 동일.
- **RAG**: 검색된 문서 청크는 동일하나 prefix(질문, 시스템 명령)가 매번 다름.
- **Multi-document QA / Tool use**: 여러 immutable 문서가 임의 순서·임의 prefix와 결합.

이 모든 경우 prefix가 한 토큰만 달라져도 캐시가 무효화되어 **prefill을 통째로 재실행**해야 한다. EPIC은 "특정 청크의 KV는 그 청크 자신의 토큰들로만 결정되는 것이 아니라 prefix와 절대 위치(RoPE)에도 의존한다"는 사실 때문에 단순히 KV를 가져다 붙이면 attention 분포가 망가진다는 점을 직시하고, **재계산을 어떻게 최소화할 것인가**를 푼다.

## 2. 핵심 아이디어

청크를 position 0부터 isolated prefill로 미리 계산해 저장(Compile)하고, 다른 요청에서 재사용할 때는 **"각 청크의 처음 k(≤32)개 초기 토큰만 다시 계산"** 한다. 초기 토큰들이 attention sink로서 자기 청크 내부 토큰들의 attention을 비정상적으로 흡수하던 능력을 제거하면, 나머지 토큰의 KV는 그대로 재사용해도 정확도 손실이 거의 없다 — 이 link 단계 알고리즘이 LegoLink(v1) / AttnLink(v2)다.

## 3. 알고리즘 / 아키텍처 상세

EPIC의 두 핵심 컴포넌트:

### 3.1 KVSplit — 청크 분할기
- 사용자가 의미·길이·고정 토큰 수 등 분할 정책을 주입할 수 있는 customizable splitter.
- 의미(semantic) 분할이 고정 512-token 분할 대비 정확도-성능 Pareto frontier 우위.
- 분할된 immutable 청크 단위로 KV가 캐싱·재조합된다.

### 3.2 Compile Step (오프라인 또는 첫 사용 시)
1. 각 immutable 청크 c_i를 **position id 0..|c_i|-1** 로 LLM에 prefill만 수행.
2. 산출 KV 텐서를 청크 콘텐츠 해시(또는 사용자 지정 키) 기준으로 디스크/HBM에 저장.
3. 청크 간 prefix 의존성 없음 → **임의 순서·임의 조합으로 재사용 가능**.
4. 이 단계에서 RoPE는 위치 0에 앵커링되어 적용된다.

### 3.3 Link Step (요청 도착 시) — LegoLink / AttnLink
요청이 [system prompt] + [chunk_a] + [chunk_b] + [chunk_c] + [user query] 로 구성되어 있다고 하자.

```
# Pseudocode
def link(prefix_tokens, chunk_list, k=32):
    # 1. 각 청크의 캐시된 KV를 가져와 글로벌 위치로 재배치
    global_pos = len(prefix_tokens)
    kv_blocks = []
    recompute_token_ids = []          # link cost는 여기에만 비례
    recompute_positions = []
    for i, chunk in enumerate(chunk_list):
        cached_kv = KVStore.get(chunk.hash)
        # RoPE 재적용: position 0 기반 KV → global_pos 기반으로 회전 보정
        rotated_kv = rope_shift(cached_kv, delta=global_pos)
        kv_blocks.append(rotated_kv)
        if i > 0:                     # 첫 청크는 스킵 (자연스러운 sink 위치)
            # 처음 k개 토큰만 재계산 후보로 등록
            recompute_token_ids.extend(chunk.tokens[:k])
            recompute_positions.extend(range(global_pos, global_pos + k))
        global_pos += len(chunk)

    # 2. prefix + 사용자 쿼리는 항상 정상 prefill
    prefix_kv = prefill(prefix_tokens, positions=range(0, len(prefix_tokens)))

    # 3. 선택적 재계산: 각 청크 내 처음 k개 토큰만 다시 attention 계산
    #    이 토큰들의 query는 새 위치에서 prefix + 모든 이전 청크의 KV를 봄
    fixed_kv = recompute_attention_for(
        token_ids=recompute_token_ids,
        positions=recompute_positions,
        keys_visible=[prefix_kv, *kv_blocks],
    )
    overwrite_first_k_per_chunk(kv_blocks, fixed_kv, k)

    # 4. 사용자 쿼리 토큰의 prefill은 결합된 KV 위에서 마지막에 수행
    return concat(prefix_kv, *kv_blocks)
```

핵심 디테일:
- **k는 청크당 ≤32**, 첫 청크는 자기 자신이 attention sink 위치에 있으므로 재계산 불필요.
- **정적 attention 희소성(static attention sparsity)** 가정: attention sink의 영향은 청크 시작부 매우 좁은 토큰 윈도우에 집중되며, 이 윈도우는 모델·레이어·헤드별로 정적이라 사전에 추정 가능.
- v1 논문은 이 알고리즘을 **LegoLink**(레고 조각의 접합부 처리에 비유)로, v2/ICML 게재본은 **AttnLink**로 부른다 — 본질 동일.

### 3.4 시스템 통합
- 구현: Python 약 2K LoC 위에 **vLLM 0.4.1** 기반.
- 메인 메모리 KV 풀 + GPU 슬롯의 2계층 저장. PagedAttention의 블록 단위와 정렬해 청크별 KV를 가변 길이 블록 시퀀스로 저장.

## 4. 이론적 근거 및 가정

| 가정 | 근거 |
|------|------|
| 정적 attention sparsity | StreamingLLM·Attention Sink 라인의 후속 — 시퀀스 시작부 1~32 토큰이 attention 질량의 큰 비율을 흡수. 이 패턴은 입력 분포에 거의 무관(추정). |
| RoPE를 임의 delta로 회전해도 KV의 의미는 보존 | RoPE는 dot product에 회전만 곱하므로 query·key가 같은 양만큼 회전하면 attention score 보존. 이는 PIC 라인의 공통 가정. |
| 청크 내부 토큰 간 attention은 첫 k 토큰을 고치면 나머지는 prefix-independent | 어텐션 sink 토큰을 정정하면, 이후 토큰의 KV가 보던 sink 분포가 정상화되어 실효 정확도 회복 (실험적). |
| immutable 청크의 콘텐츠 해시가 안정적 | 사용자가 KVSplit 정책을 일관되게 적용 |

## 5. 복잡도 분석

| 항목 | 값 | 비고 |
|------|-----|------|
| Compile (per chunk, one-time) | O(C²·d) | C = chunk size, d = hidden dim. 표준 prefill. |
| Link 시간 | **O(k·N)** (k≤32) | N = 결합 컨텍스트 길이. CacheBlend는 O(0.15·N²). |
| Link 메모리 | O(N·d·layers) | KV 블록 자체는 그대로 유지. 재계산 토큰 k·M 개분의 추가 query/attention 임시 버퍼. |
| 저장 추가 메모리 | O(unique_chunks · avg_chunk_len · d · layers) | 일반 prefix cache보다 크지만 prefix 다양성 폭증을 흡수. |
| TTFT 점근 | N에 대해 **선형** | CacheBlend는 quadratic, full recompute는 quadratic. |

| Cache Eviction Rate | 청크 LRU/LFU 관리 필요 | 별도 정책 필요. EPIC 자체는 eviction을 깊게 다루지 않음(추정). |

## 7. 관련/경쟁 기술 비교

| 기법 | 핵심 차이 | 강점 | 약점 |
|------|----------|------|------|
| **EPIC (LegoLink/AttnLink)** | 청크 0-기반 prefill + 첫 k≤32 토큰 재계산 | O(kN) 링크, prefix 무관 재사용, RAG·few-shot에 직접 효용 | k 추정 캘리브레이션 필요, RoPE 모델만 적용 명확, 정확도 미세 손실 가능 |
| **CacheBlend (Yao et al. 2024)** | 청크 결합 후 attention deviation 상위 15% 토큰 재계산 | 일반적 적용성, 정확도 보존 모델별 자동 조정 | 링크 비용 O(0.15·N²) — 컨텍스트가 길어지면 비용 폭증 |
| **RadixAttention (SGLang)** | prefix 트리 기반 정확 매칭 | 무손실 재사용, 트리로 다중 prefix 공유 | byte-identical prefix 한정 — RAG에서 hit-rate 매우 낮음 |
| **PagedAttention (vLLM)** | 블록 단위 페이징, hash 기반 prefix 매칭 | 메모리 단편화 제거, 표준 LLM 서빙 인프라 | 자체적으로 PIC 미지원 — prefix 매칭만 |
| **H2O / SnapKV** | Heavy-hitter eviction 기반 압축(Activity C) | KV 메모리 절감, accuracy delta 작음 | 재사용 자체를 늘리지 않음. 직교 |
| **CacheGen** | KV를 압축해 네트워크 전송 | 멀티노드 KV 마이그레이션 가속 | PIC와는 직교 — 결합 가능 |
| **ProphetKV (2602.02579, 트렌드 2026-05-06)** | 사용자 쿼리 관련성 기반 재계산 예산 배분 | RULER +8.8~24.9%, LongBench +18.6~50.9% over EPIC | 추가 쿼리 임베딩 단계 필요. EPIC 위에 query-aware 재계산을 얹는 형태 |
| **QCFuse (2604.08585)** | 의미 요약 앵커로 쿼리 강화 + 레이어 어텐션 기반 토큰 선택 | 캐시 융합 베이스라인 대비 TTFT −40% | 앵커 사전계산·레이어 분석 비용 |
| **MEPIC (2512.16822)** | EPIC + 메모리 효율(메인 메모리 KV 풀) | EPIC의 GPU 메모리 압박 완화 | 후속작이라 인용 — EPIC 자체 한계 보완 |

EPIC vs 우리 코드베이스의 `SegmentedHashCache`: 둘 다 청크 단위 hash 키를 사용하지만 우리 구현은 **재계산 없이 단순 fixed-position-offset 또는 위치 무관 저장**을 가정한다. EPIC은 이 위에 **부분 재계산(첫 k 토큰)** 이라는 정확도 보정 단계를 더한 것이라고 보면 정확하다.

### CacheStore 인터페이스 호환성

`/home/user/inference-orchestration/src/cache/base.py`의 `CacheStore`는 `put(key, value)`, `get(key) -> Optional[Tensor]`, `evict()`, `hit_rate()`, `memory_bytes()`, `reset_stats()`만 노출한다.

- **호환 부분**: 청크별 KV는 `put(chunk_hash, kv_tensor)`로 자연스럽게 저장 가능. `get`에서 캐시 hit 시 KV 반환 → engine이 link step 수행 → 표준 인터페이스만으로 충분.
- **확장 필요 부분**: link step에서 "어떤 토큰을 재계산할 것인가"는 캐시가 아닌 engine 책임. `CacheStore` 인터페이스를 건드릴 필요 없음. 다만 청크 길이 메타데이터를 함께 저장해야 하므로 **`put` 호출 전 KV 텐서에 `[chunk_len, layers, heads, head_dim]` 형식을 강제**하거나 별도 `put_segment(chunk_hash, kv, length)` 헬퍼를 `epic_pic_cache.py`에 추가한다(`SegmentedHashCache.put_segment` 패턴이 이미 `segmented.py`에 존재).
- **Eviction 정책**: EPIC 청크는 OS-page 캐시처럼 LRU + 인기도 가중. `SegmentedHashCache._importance` 메커니즘을 그대로 쓸 수 있다.

### 예상 이점
1. **Non-contiguous cache hit rate**: 본 연구 목표(≥30%)를 RAG/few-shot에서 직접 달성. 현재 `SegmentedHashCache`는 hit 발생 시에도 정확도 검증 단계가 없는데, EPIC은 정확도 보정을 명시적으로 결합한다.
2. **Throughput**: 보고 7× ↑은 RAG 시나리오 한정이지만, 일반 워크로드에서도 +20% 목표는 안정적으로 달성 가능(추정).
3. **TTFT**: 컨텍스트 길이가 길수록 CacheBlend 대비 격차 확대 — 본 연구의 "long-context" 목표와 일치.
4. **압축과의 결합**: 청크별 KV에 `TriAttentionCodec`을 적용하면 메모리 절감 30% 목표(C)와 PIC 히트율 목표(B)를 동시 달성하는 B+C cross가 자연스러움.

### 예상 비용/리스크
- **k 캘리브레이션**: 모델·레이어별 attention sink 강도가 다를 수 있어 모델 등록 시 캘리브레이션 데이터셋이 필요하다. 잘못 잡으면 정확도 −2%p 이상 손실 가능(추정).
- **RoPE 회전 보정**: pre-RoPE/post-RoPE 텐서 선택을 모델 어텐션 백엔드와 정합해야 한다. `tri_attention_codec.py`가 이미 pre-RoPE를 가정하고 작성되어 있어 충돌 위험 있음 — 청크 KV는 pre-RoPE 형태로 저장하고 재사용 시 새 position으로 RoPE 적용하는 방식이 깨끗하다.
- **메모리 풋프린트 증가**: prefix 캐시보다 청크 캐시는 1.5~3× 더 클 수 있다(추정). C 결합으로 상쇄 필요.
- **첫 청크 가정 위반**: 시스템 프롬프트가 매번 다르면 "첫 청크는 재계산 불필요" 가정이 깨진다 → prefix 토큰 길이가 chunk size 미만이면 첫 청크도 k 토큰 재계산해야 안전.
- **다중 청크 재배치 시 cross-chunk attention sink**: 청크 b와 c 사이에서도 sink가 발생하므로 청크 b의 마지막 토큰들이 청크 c의 첫 k 토큰에 의해 부분적으로 영향 받음. EPIC은 이걸 첫 k 재계산으로 흡수한다고 본다(추정).

### 다른 Activity와의 시너지/충돌
- **B + C(시너지)**: 청크 KV를 `TriAttentionCodec` 또는 `NQKVCodec`으로 양자화 저장 → 동일 메모리 예산에서 더 많은 unique 청크 캐시 가능. 트렌드 리포트(2026-05-06) Cross-1 라인과 자연스럽게 결합.
- **A + B(시너지)**: 청크 인기도 기반 prefix-aware batching — 인기 청크를 공유하는 요청을 같은 배치로 묶어 KV 풀 GPU 적재 비용 분할.
- **B + B(충돌 가능)**: `QueryCentricRecomputeCache`(ProphetKV)와 EPIC은 같은 위치(link step) 점거. ProphetKV는 query-relevance 기반 토큰 선택, EPIC은 첫 k 토큰 고정 선택. 결합하면 **"첫 k(EPIC) ∪ top-q% query-relevant(ProphetKV)"** 합집합 재계산 후보가 되며 예산 초과 위험. 합집합 → 예산 캡 적용 순서로 정리해야 함.
- **B + B(시너지)**: `InfoFlowChunkReorderCache`와는 정보 흐름 기반 청크 순서 결정 후 EPIC AttnLink로 결합 — 자연스러움.

## 9. vLLM 이식 관점

### 최신 vLLM 내 유사 기능 유무
- vLLM 본체에는 **PIC가 정식 통합되어 있지 않다**. RadixAttention과 prefix caching만 제공.
- 다만 LMCache(2510.09665)가 PIC-style 청크 재사용을 부분적으로 지원하기 시작했으며 vLLM과 통합 모듈을 공유한다 — 우리 이식 시 LMCache 인터페이스를 차용 가능(검토 필요).
- EPIC 원본 구현은 vLLM 0.4.1 기반이며, 우리 환경의 vLLM v0.20.x과는 architecture 차이가 크다(v1 엔진, paged 블록 풀, async KV connector).

### 손대야 할 모듈 후보
1. **Block manager** (`vllm/v1/core/kv_cache_manager.py`, `block_pool.py`): 청크 hash 키를 prefix block hash와 별도 namespace로 등록. `vllm_integration/block_manager_patch.py` 확장.
2. **Attention backend** (`vllm/v1/attention/backends/`): prefill 시 `query_start_loc`을 사용해 "각 청크의 첫 k 토큰"을 별도 mask로 처리하는 부분 재계산 경로. `vllm_integration/attention_backend_patch.py`의 `VllmQueryCentricAttentionWrapper` 패턴 재사용.
3. **Scheduler** (`vllm/v1/core/sched/scheduler.py`): PIC 청크 hit 정보를 prefill budget 계산에 반영해 batch composition 결정. `vllm_integration/scheduler_patch.py` 확장.
4. **Model runner** (`vllm/v1/worker/gpu_model_runner.py`): RoPE 적용 시 청크별 position offset을 정확히 계산.
5. **KV connector** (`vllm/v1/core/kv_cache_connector/`): PIC 청크를 메인 메모리/디스크에 오프로드/로드.

### 이식 난이도
**Medium-High**. 이유:
- 알고리즘 자체는 단순(첫 k 토큰 재계산)하나 vLLM v1의 paged 블록 모델과 청크 단위 KV 저장을 정합시키는 작업이 비-trivial.
- RoPE 위치 재할당이 attention backend(FlashAttention, XFormers, Triton)별로 정확히 동일하게 동작하도록 검증 필요.
- 원 EPIC 구현이 v0.4.1 기반 → 코드 직접 포팅은 거의 불가능, 알고리즘만 차용해 v1 architecture에서 재구현이 현실적.
- 단, AttnLink/LegoLink 자체 로직은 100~300 LoC로 압축 가능(추정).

## 10. 열린 질문 / 후속 실험 제안

1. **k의 동적 결정**: 원 논문은 k≤32 고정. 모델·태스크별로 첫 k 토큰의 sink 강도를 사전 캘리브레이션 + 동적 조정하면 정확도-비용 곡선 추가 개선 여지. 우리 실험: Llama-3-8B/Qwen2.5-7B/Mistral-7B에서 k∈{4,8,16,32,64} 스윕하여 RULER·LongBench accuracy/TTFT 곡선 측정.
2. **EPIC × 압축 결합 (B+C cross)**: 청크 KV를 `TriAttentionCodec` 또는 `NQKVCodec`으로 양자화 저장한 뒤 link step에서 첫 k 토큰만 풀-precision으로 재계산. 가설: 메모리 −30% 동시에 PIC hit-rate ≥30% 달성 가능. 실측 후 우리 목표 지표 표 갱신.
3. **EPIC × Query-Centric 결합 (B+B cross)**: AttnLink의 "첫 k 토큰 재계산"과 ProphetKV의 "query-relevance top-q% 재계산"을 합집합/예산캡 결합. 합집합이 예산을 초과할 때 어떤 우선순위(positional vs semantic)가 정확도에 더 중요한지 ablation. RAG long-context 워크로드에서 측정.

추가 검증이 필요한 추정(`(추정)` 표시):
- (추정) AttnLink가 비-RoPE 모델(ALiBi, T5-style)에서도 작동하는지는 논문에서 명시되지 않음.
- (추정) Cross-chunk attention sink 효과는 첫 k 토큰 재계산만으로 완전 제거된다고 주장되나, 청크 수가 매우 많아지면(>20) 누적 오차 가능.
- (추정) EPIC 공식 GitHub 저장소 공개 여부는 검색 결과에서 명확히 확인되지 않았다 — 저자 페이지(derekhjh.com)와 ICML 미러를 직접 확인 필요.
