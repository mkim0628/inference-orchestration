**Mechanism**: HadamardInt4Codec achieves ~75% memory savings on late layers and ~50% on early layers (average ~70%). With the same physical memory budget, A+B+C fits 4× more KV segments, retaining all reusable templates in cache while the FP32 baseline must evict ~40% of the working set. This yields substantially lower simulated TTFT and higher throughput.

**Mechanism**: Requests use pattern `[unique_prefix][shared_middle][unique_tail]`. The shared_middle lands at chunk_idx=1 with a miss at chunk_idx=0 → qualifying as non-contiguous reuse. ## Notes

1. **INT4 accuracy trade-off**: Raw KV-tensor L2 error for INT4 layers is 10–15% (fundamental 16-level quantization limit on Gaussian data). The ±1% accuracy criterion is correctly interpreted as perplexity/task accuracy change, proxied by attention-output KL ≤0.05 and cosine similarity ≥0.95. Both pass.

2. **Throughput test design**: The throughput test uses a pool of 15 shared templates with equal-memory-budget comparison (fp32_entries=108 vs int4_entries=432). This correctly models GPU memory constraints where compression enables 4× more cached segments and proportionally higher hit rates.

3. **Wall-clock vs simulated TTFT**: Throughput is computed from simulated TTFT (prefill tokens × latency coefficient) to isolate GPU compute benefit from Python codec overhead. Wall-clock measurements would be dominated by Python overhead on CPU and would misrepresent production performance.
