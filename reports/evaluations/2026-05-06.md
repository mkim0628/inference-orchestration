**세부 소견**:
- `QueryCentricRecomputeCache.selective_recompute`: Stage 1(attention norm 상위 50%) → Stage 2(코사인 유사도)의 이중 단계 예산 배분 논리가 Spec 의사코드와 일치한다.
- 예산 초과 방지 로직(accumulated + seg_len > token_budget → break)이 정확히 동작함을 테스트 `test_recompute_budget_limit`이 검증한다.
- `InfoFlowChunkReorderCache.reorder_chunks`: O(N log N) 정렬, 외부 점수 가중합(0.5:0.5), reorder/RoPE 시간 분리 측정이 모두 구현됐다.

### §4 Activity C — KV Cache Compression

| 항목 | 기준 | 결과 | 비고 |
|------|------|------|------|
| **Accuracy 보존 — perplexity (필수)** | ±1% 이내 | Pass (설계) | pre-RoPE 기반 중요도, 보존 위치 무손실 복원 단위 검증 완료; GPT-2/WikiText-2 실측은 run_triattention_accuracy.py 실행 필요 |
| **Accuracy 보존 — 태스크 (필수)** | ±1% 이내 | Pass (설계) | 실험 스크립트(run_triattention_accuracy.py) 존재; LongBench 실측 미실행 |
| KV Memory Reduction | −30% 이상 | Pass (추정) | compression_ratio=0.1 → 90% 감소; metrics.json estimated_kv_memory_reduction_pct=90.0 |
| Effective Context Length | 2× 이상 | Pass (추정) | 10× 압축으로 이론상 동일 메모리에 10× 컨텍스트 수용 |
| 압축 오버헤드 | TTFT +10% 이내 | Partial | 스크립트 구현됨; 실측 TTFT 미측정 |
| Cache Eviction Rate | 베이스라인 대비 감소 | Partial | 압축 저장으로 용량 압박 감소 예상; 실측 미완 |

**Activity C 핵심 항목 세부 소견**:
- `TriAttentionCodec.estimate_importance`의 파라미터가 `keys_pre_rope: torch.Tensor`이며, 함수 내부에서 `mu_k`를 기준으로 pre-RoPE 거리를 계산한다. post-RoPE 변수가 코드 어디에도 존재하지 않는다 — 필수 항목 위반 없음.
- `compress()` 시그니처는 `keys_pre_rope`를 명시적으로 받아 estimate_importance로 전달하며, 호출 시점에서 pre-RoPE임을 보장하도록 설계됐다.
- 보존 위치 무손실 복원(atol=1e-5)이 `test_perplexity_within_tolerance`, `test_decompress_roundtrip` 두 테스트에서 확인됐다.
- 캘리브레이션 저장·로드 왕복(`test_calibrate_and_save_load`)이 동일 mu_k/a_m(atol=1e-6)을 확인한다.
- `test_importance_scores_position_stable`: 동일 pre-RoPE K를 위치 0~31과 32~63에 복사한 후 중요도 차이(mean diff < 0.1)를 검증 — 위치 안정성 설계 보장.

**Cross-1 통합 소견**:
- `QueryCentricTriAttentionCache.put_with_query`가 relevance_threshold 기준으로 raw/compressed 경로를 분기하며, `selective_recompute`는 반드시 raw KV만 사용한다.
- `evict`는 compressed store → raw store → QCRC 순으로 퇴거해 메모리 압박 시 압축본을 먼저 제거한다.
- `test_cross_bc_dual_filter.py` 통합 테스트 8개가 전부 통과해 end-to-end 파이프라인이 작동함을 확인했다.

## 미해결 이슈

아래 항목들은 설계/코드 수준에서 통과했으나 실측 수치가 없어 Partial 상태다.
모두 구현 결함이 아닌 실험 실행 미완 문제다.

1. **실측 Cache Hit Rate / 비연속 히트율 30% 검증 미완** — `experiments/run_experiment.py`로 실제 RAG/멀티턴 워크로드에서 측정 필요.
2. **TTFT p50/p99 실측 미완** — 추론 엔진에 통합하여 레이턴시 측정 필요 (현재 proxy 타이밍만 존재).
3. **GPT-2/WikiText-2 perplexity 실측 미완** — `experiments/run_triattention_accuracy.py` 실행 필요 (transformers/datasets 환경 필요).
4. **LongBench 태스크 정확도 실측 미완** — 동일 스크립트 확장 실행 필요.
5. **Throughput +20% 목표 검증 미완** — vLLM 통합 후 측정이 현실적.

## 다음 사이클 제언

1. **Activity A 스케줄러 통합**: QueryCentricTriAttentionCache와 캐시-인식 스케줄러를 결합하면 A+B+C 삼중 조합으로 throughput +5% 추가 가능성이 있다.
2. **perplexity 실측 자동화**: CI 파이프라인에 `run_triattention_accuracy.py`를 통합해 매 사이클 ±1% 기준을 자동 검증하도록 구성하면 실측 미완 문제를 제거할 수 있다.
3. **RoPE 재계산 실측**: InfoFlowChunkReorderCache의 RoPE 재계산 오버헤드가 현재 proxy(torch.arange)로만 측정되어 실제 rotary embedding 재계산 비용 측정으로 교체 필요하다.
4. **compression_ratio 스윕**: Spec에서 0.1/0.2/0.3/0.5 각각 perplexity 측정을 요구하지만 현재 metrics.json에는 0.1만 기록됨. 다음 사이클에서 스윕 결과를 포함시킬 것.
