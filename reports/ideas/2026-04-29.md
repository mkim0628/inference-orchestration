## 변화 요약
- SAW-INT4(arXiv 2604.19157)가 Hadamard 회전 기반 INT4 양자화로 near-BF16 정확도를 달성해, 이전 사이클의 INT8 혼합 정밀도 전략 대비 압축률이 −68.8% → −75%+ 이상으로 도약 가능한 신규 알고리즘이 등장했다.
- KV Packet(arXiv 2604.13226)이 soft-token adapter 학습으로 재계산 FLOPs ≈ 0인 위치-독립 KV 재사용을 실현해, Activity B의 현재 구현(선택적 재계산 방식)의 핵심 병목을 우회하는 새로운 경로를 열었다.
- llm-d-kv-cache 오픈소스 + vLLM Q2 2026 스케줄러 재설계 로드맵 등록으로 Activity A 구현의 기반 인프라와 통합 포인트가 명확해졌다. 전날 SUMMARY.md에서 다음 우선순위 1위로 지목된 Activity A를 이번 사이클에 추가 가능.

## 이번 사이클 초점
**A+B+C** — KV Cache-aware Scheduling + Non-Contiguous KV Cache Reuse + KV Cache Compression 전체 통합

**선택 이유**:
1. 이전 사이클(B+C)에서 비연속 히트율 30.3%, 메모리 −68.8%, 정확도 ±0.72% 달성으로 B+C 기반이 확립됨. 이 위에 A를 추가하는 것이 자연스러운 다음 단계.
2. Activity A의 "cache-hit-weighted 라우팅 스케줄러"는 이전 사이클에서 다음 우선순위 1위로 권장된 항목.
3. C 측에서 SAW-INT4(INT4 + Hadamard 회전)로 교체하면 메모리 목표 −30%를 2.5× 초과 달성 가능.
4. B 측에서 KV Packet adapter를 추가하면 히트 시 TTFT를 추가로 줄여 throughput +20% 목표에 근접 가능.
5. A+B+C 통합으로 처리량·메모리·정확도 세 지표를 동시에 측정 가능 — evaluation_criteria.md §1·§2·§3·§4·§5 전 섹션 적용.

## Activity A 아이디어

### 아이디어 A-1: 비연속 히트율 인식 캐시-가중 스케줄러
- **가설**: 기존 ContiguousCache에 접두사 미일치 요청이 들어와도, 세그먼트 해시 인덱스를 통해 부분 히트(partial hit)를 감지하고 히트 예상 세그먼트 수를 가중치로 요청을 재정렬하면 전체 캐시 히트율과 처리량이 향상된다.
- **접근 방법**:
  1. `CacheAwareScheduler` 클래스 구현 (`src/scheduler/cache_aware_scheduler.py`)
  2. 각 요청의 청크 해시를 `CompressedSegmentCache` 인덱스와 매칭 → 예상 히트 세그먼트 수 계산
  3. 우선순위 스코어 = hit_segments / total_segments × (1 − queue_wait_penalty)
  4. 동일 배치 내에서 히트율 높은 요청을 먼저 처리. 대기 시간 상한 = 2× 평균 대기(공정성 보장)
  5. `src/engine/batch_runner.py`에 스케줄러 훅 추가
- **대상 환경**: 단일 노드
- **멀티 노드 고려사항**: 이번 사이클은 단일 노드에 집중. 멀티 노드 확장은 llm-d-kv-cache 스타일 KVEvents를 차기 사이클에 추가.
- **예상 효과**: 처리량 +15~20%, 스케줄링 오버헤드 TTFT +3% 이내
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 구현 타겟)

### 아이디어 A-2: 퇴거-인식 선점 회피 스케줄러
- **가설**: vLLM Q2 2026 로드맵에서 "과도한 선점(preemption) 방지"가 이슈로 등록된 것처럼, 압축된 세그먼트가 퇴거되기 직전에 해당 세그먼트를 공유하는 요청을 우선 스케줄링하면 퇴거율을 줄일 수 있다.
- **접근 방법**: LRU 캐시에서 eviction 예정 엔트리를 모니터링 → 해당 세그먼트를 참조하는 대기 요청을 우선순위 큐 상위로 이동.
- **대상 환경**: 단일 노드
- **멀티 노드 고려사항**: 해당 없음 (단일 노드 LRU 기준)
- **예상 효과**: 캐시 퇴거율 −15~25%, TTFT 오버헤드 +2% 이내
- **구현 난이도**: low
- **우선순위**: 2

## Activity B 아이디어

### 아이디어 B-1: KV Packet 스타일 Adapter 브릿지 (재계산 없는 PIC 강화)
- **가설**: 현재 구현의 `CompressedSegmentCache`가 캐시 히트 시 위치 불일치로 발생하는 어텐션 오류를 경량 trainable adapter로 보정하면, 재계산 없이 비연속 세그먼트 정확도를 추가로 개선할 수 있다.
- **접근 방법**:
  1. `SegmentAdapter` 모듈: 각 세그먼트 경계에 2-layer MLP soft-token adapter (dim=64) 추가
  2. self-supervised distillation: 재계산 KV(정답)와 cached KV(입력) 간 L2 loss로 학습
  3. adapter weights는 청크 크기별로 공유 (위치 독립성 유지)
  4. 추론 시: cached KV → adapter → corrected KV → attention 계산
- **예상 효과**: 비연속 히트 시 어텐션 오차 −50% (현재 INT8 max 0.72% → 0.36% 이하), TTFT 추가 감소
- **구현 난이도**: high
- **우선순위**: 3 (B-2 다음으로 시도)

### 아이디어 B-2: 청크 경계 동기화 (ChunkKV 연계)
- **가설**: Activity C의 ChunkKV가 의미 청크 단위로 퇴거를 결정하는 것처럼, Activity B도 동일한 청크 경계(128 토큰)를 공유하면 퇴거 결정과 재사용 결정이 일치하여 B+C 통합 효율이 높아진다.
- **접근 방법**: `CompressedSegmentCache`의 chunk_size를 128(현재 64에서 확장), ChunkKV 스타일 청크 중요도 스코어를 LRU 퇴거 정책에 반영.
- **예상 효과**: 비연속 히트율 +3~5%p (청크 경계 정렬로 미스 감소), LRU 퇴거율 −10%
- **구현 난이도**: low
- **우선순위**: 2 (A-1 다음으로 구현)

## Activity C 아이디어

### 아이디어 C-1: SAW-INT4 스타일 Hadamard 회전 기반 INT4 양자화
- **가설**: 기존 INT8 혼합 정밀도(후반 레이어) 대신 Hadamard 회전 + INT4 양자화를 적용하면, 추가 오버헤드 없이 메모리를 −75%로 줄이면서 near-BF16 정확도를 유지할 수 있다.
- **접근 방법**:
  1. `HadamardInt4Codec` 클래스: `encode(kv, layer_idx)` → Hadamard 회전 → clip → int4 quantize, `decode(compressed, layer_idx)` → int4 dequantize → inverse Hadamard
  2. 블록 크기 B=64인 블록-대각 Hadamard 행렬 (SAW-INT4 참조)
  3. 초기 N/3 레이어는 FP16 유지 (PM-KVQ 혼합 정밀도 전략 유지)
  4. `CompressionCodec`의 `encode/decode` 인터페이스 교체로 기존 `CompressedSegmentCache`와 호환
- **예상 효과**: 메모리 −75% (현재 −68.8% → −75%), accuracy delta ±0.5% 이내
- **accuracy-preserving 근거**: SAW-INT4 논문이 Qwen3·Llama 계열에서 near-BF16 정확도를 실증. Hadamard 회전이 outlier를 분산시켜 INT4의 치명적 quantization error를 방지. 초기 레이어 FP16 유지로 어텐션 싱크 토큰 정보 보전. 검증: WikiText-2 perplexity 측정 + HellaSwag 정확도 비교 (허용 ±1%).
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 구현 타겟)

### 아이디어 C-2: ChunkKV 의미 청크 퇴거 통합
- **가설**: 청크 내 모든 토큰의 어텐션 스코어 합산으로 청크 중요도를 계산하고 청크 단위로 퇴거하면, 토큰 단위 퇴거 대비 의미 정보가 보존되어 accuracy delta가 줄어든다.
- **접근 방법**: `CompressedSegmentCache.evict()` 메서드에 청크 중요도 계산 로직 추가. 가장 낮은 중요도 청크를 퇴거.
- **예상 효과**: 퇴거 시 accuracy delta −30% 감소 (청크 경계 보존), 퇴거율 변화 없음
- **accuracy-preserving 근거**: 의미 단위 청크 보존으로 문맥 연속성 유지. ChunkKV 논문이 공격적 압축 비율에서 기존 토큰 퇴거 대비 10% 성능 향상 실증.
- **구현 난이도**: low
- **우선순위**: 2

- **구현 난이도**: medium (각 컴포넌트는 독립 구현 가능, 통합 인터페이스 추가만 필요)

- **예상 복합 효과**: 메모리 −75%, 어텐션 오차 ≤0.3%, TTFT 추가 감소
- **구현 난이도**: high
- **우선순위**: 4 (다음 사이클 후보)
