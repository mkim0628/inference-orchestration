## 변화 요약
- **멀티노드 스케줄링 구체화**: FlowKV(2504.03775)와 PrefillaaS(2604.15039)가 P/D 분리 환경에서 KV 전송 오버헤드와 load-aware 라우팅의 구체적 구현 방식을 실증했다. 이전 사이클의 Activity A는 단일노드에만 머물렀으므로, 오늘 사이클에서 멀티노드 stub 구현이 현실적 타겟이 된다.
- **ARKV tri-state**: retain/compress/evict 3-상태 분류가 기존 LRU+INT4 대비 adaptive하게 압축 강도를 조절한다. 이전 사이클에는 없던 알고리즘.
- **KV Packet adapter 재확인**: SUMMARY.md의 "다음 우선순위 4위" 항목이 오늘 트렌드에서도 핵심 미완성 B 업그레이드로 재확인됨. self-supervised distillation 경로가 명확.

**선택 이유**:

4. FlowKV/PrefillaaS 기반 멀티노드 스케줄러가 Activity A의 disaggregated prefill 요구사항을 충족.

## Activity A 아이디어

### 아이디어 A-1: FlowKV 스타일 멀티노드 Load-Aware 스케줄러 (신규)
- **가설**: Prefill 노드와 Decode 노드를 분리하고, 비연속 캐시 세그먼트의 예상 히트율과 현재 노드 부하를 결합한 라우팅 스코어로 요청을 배분하면, 단일노드 대비 TTFT를 줄이고 전체 처리량을 +20% 이상 달성할 수 있다.
- **접근 방법**:
  1. `MultiNodeScheduler` 클래스 (`src/scheduler/multi_node_scheduler.py`): prefill 노드 풀 + decode 노드 풀 시뮬레이션
  2. 라우팅 스코어 = cache_hit_score(request) / estimated_transfer_cost(prefill_node → decode_node)
  3. PrefillaaS 스타일 "compress_before_transfer" 플래그: INT4 KV를 전송 전 압축하여 대역폭 절감
  4. 단일노드 `CacheAwareScheduler`와 동일 인터페이스 (`BaseScheduler`) 상속으로 기존 테스트 재활용
  5. 멀티노드 stub: 실제 네트워크 없이 전송 비용을 latency 파라미터로 시뮬레이션
- **대상 환경**: 멀티노드 (P/D 분리 시뮬레이션)
- **예상 효과**: 처리량 +20~25% (prefill 분리로 decode 병목 제거), TTFT p50 오버헤드 +5% 이내
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 A-2: compress-before-transfer 정책 스케줄러
- **가설**: KV를 전송 전 INT4로 압축하면 대역폭 요구가 4× 감소. 이를 스케줄러 결정 변수로 형식화하면 고-latency 네트워크에서 PrefillaaS 대비 추가 이득.
- **접근 방법**: `MultiNodeScheduler`에 `compress_policy={"threshold_bytes": X, "codec": HadamardInt4Codec}` 옵션 추가. 전송 비용이 임계값 초과 시 자동 압축.
- **예상 효과**: 멀티노드 네트워크 대역폭 -75%, decode 노드 메모리 추가 절감
- **구현 난이도**: low (A-1 위에 플래그 추가)
- **우선순위**: 2

## Activity B 아이디어

### 아이디어 B-1: KV Packet Adapter 통합 (SUMMARY 4위 미완성 항목)
- **가설**: `CompressedSegmentCache` 비연속 히트 시 위치-불일치 오류를 경량 MLP adapter로 보정하면, 재계산 없이 어텐션 정확도를 FP16 수준으로 복원할 수 있다.
- **접근 방법**:
  1. `SegmentAdapter` 클래스 (`src/cache/segment_adapter.py`): 2-layer MLP (hidden=64), 세그먼트 경계마다 1개 어댑터
  2. 학습: `(cached_kv, target_recomputed_kv)` 쌍으로 L2 loss self-supervised distillation (CPU 학습, 500 step)
  3. 추론: `cached_kv → adapter.forward(cached_kv) → corrected_kv → attention`
  4. adapter weights를 `CompressedSegmentCache`에 청크 크기별로 저장 (위치-독립)
  5. `record_attention_score()` 기존 메서드와 통합
- **예상 효과**: 비연속 히트 시 어텐션 KL divergence 0.007 → 0.003 이하, TTFT 재계산 비용 0 FLOPs
- **구현 난이도**: high
- **우선순위**: 3

### 아이디어 B-2: MEPIC 스타일 NoPE 저장 실험
- **가설**: RoPE를 적용하지 않은 상태로 KV를 저장하면 청크를 임의 위치에 배치해도 정확도 손실이 없다.
- **접근 방법**: `PositionFreeSegmentCache` 서브클래스: 저장 시 RoPE 미적용, 로드 시 fused RoPE 주입. 커널 수정 불필요한 근사 버전.
- **예상 효과**: 위치-독립 히트율 +5%p
- **구현 난이도**: high (커널 수준 변경)
- **우선순위**: 4 (다음 사이클 후보)

## Activity C 아이디어

### 아이디어 C-1: ARKV Tri-State 프레임워크 통합 (신규)
- **가설**: 모든 KV 토큰을 동일하게 INT4 양자화하는 대신, attention heavy-hitter 스코어에 따라 retain(FP16)/compress(INT4)/evict(삭제) 3-상태로 분류하면, 정확도를 보존하면서 메모리를 추가 절감할 수 있다.
- **접근 방법**:
  1. `TriStateCompressor` 클래스 (`src/cache/tri_state_compressor.py`)
  2. `score(kv, attn_weights) → {"retain": [], "compress": [], "evict": []}` 분류
  3. retain: 상위 20% heavy-hitter 토큰 → FP16 유지
  4. compress: 중간 40% → `HadamardInt4Codec` 적용
  5. evict: 하위 40% → 삭제 (재계산 필요 시 flag 설정)
  6. `CompressedSegmentCache`의 `evict()` 메서드를 `TriStateCompressor`로 교체
- **정확도 보존 계획**: retain tier가 attention sink 토큰을 보호. compress tier의 INT4는 이미 KL<0.007 검증. evict는 attention score가 낮은 토큰만 대상. WikiText-2 perplexity ±1% 기준 통과 후 배포.
- **예상 효과**: 메모리 −75% → −80%+ (evict tier 추가), 정확도 delta ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 C-2: Adaptive Bit-Width (Don't Waste Bits 참조)
- **가설**: 세그먼트 중요도(Activity B의 attention score)를 비트폭 결정에 재활용하면, 중요 세그먼트는 INT8, 덜 중요한 세그먼트는 INT4/INT2로 동적 압축 가능.
- **접근 방법**: `AdaptiveBitWidthCodec`: score > threshold_high → INT8, threshold_low < score ≤ threshold_high → INT4, score ≤ threshold_low → INT2.
- **예상 효과**: 평균 비트폭 3.2bit (기존 4bit 대비 −20% 추가 절감)
- **구현 난이도**: medium
- **우선순위**: 3

## 크로스 Activity 시너지 아이디어

### Cross-1: A(멀티노드) + B(KV Packet) + C(ARKV tri-state)
- **조합 근거**: MultiNodeScheduler가 ARKV compress/retain 상태를 메타데이터로 수신 → retain KV는 로컬 노드 우선 배치, compress KV는 원격 전송 시 추가 INT4 적용, evict KV는 재계산 flag. KV Packet adapter가 비연속 세그먼트를 정확도 손실 없이 브릿지.
- **예상 복합 효과**: 처리량 +20~25%, 메모리 −80%+, 정확도 ±0.5% 이내, TTFT 오버헤드 ≤5%
- **구현 난이도**: medium (각 컴포넌트 독립 구현 후 인터페이스 연결)
- **우선순위**: 1 (이번 사이클 전체 타겟)

### Cross-2: compress-before-transfer + ARKV
- **조합 근거**: PrefillaaS 스타일로 전송 전 ARKV compress-tier KV만 선택적 INT4 전송. retain-tier는 FP16 그대로 전송.
- **예상 효과**: 대역폭 −60% (compress tier만 INT4), 정확도 손실 최소화
- **우선순위**: 2
