- **Activity C 경량 양자화 신규 분기 등장**: NQKV(정규분포 분위수)와 FireQ(RoPE-인식 2단계 평활화)가 이전 사이클의 Hadamard/PolarQuant/QJL 계열과 전혀 다른 이론적 기반에서 INT4/FP8을 달성한다. 특히 FireQ는 RoPE와 INT4의 충돌이라는 새로운 문제 차원을 명시적으로 해소하며, NQKV는 변환 오버헤드 없이 분포 적합만으로 INT4를 달성해 CPU 시뮬 환경에서도 실제 GPU 처리량 검증 경로를 단순화한다.

## Activity A 아이디어

### 아이디어 A-1: Helium 쿼리 플랜 연산자 추상화 — LLM 호출을 일급 연산자로 취급하는 캐시-인식 워크플로우 스케줄러
- **가설**: 에이전틱 워크플로우를 데이터베이스 쿼리 플랜으로 모델링하고, 각 LLM 호출을 일급 연산자(first-class operator)로 취급해 워크플로우 전체의 KV 캐시 재사용 계획을 쿼리 최적화 원리로 수립하면, 이전 사이클의 DAGTopologyScheduler(DAG BFS + 위상 분석 기반 KV 보존 결정)와 달리 "쿼리 플랜 내 연산자 간 중복 KV를 선제적으로 공유"하는 계획 레벨 최적화가 가능해 1.56× 이상 속도 향상을 달성할 수 있다.
- **접근 방법**:
  1. `QueryPlanScheduler` 클래스 (`src/scheduler/query_plan_scheduler.py`): `BaseScheduler` 상속
  2. 워크플로우 등록: `register_workflow(plan_tree)` — 플랜 트리는 `{op_id, op_type(llm_call/tool_call/branch), input_refs, kv_reuse_hints}` 노드로 구성
  3. 플랜 최적화: 동일 입력을 공유하는 연산자 서브트리를 탐지해 공통 KV 프리캐싱(proactive caching) 결정 → KV 재계산 없이 연산자 간 공유
  4. 연산자 실행 순서 최적화: 공통 KV를 생성하는 연산자를 의존하는 연산자보다 먼저 배치(CSE: Common Subexpression Elimination 원리 적용)
  5. 캐시 미스 예측: 플랜 분석 시점에 캐시 히트 예상 연산자와 미스 예상 연산자를 사전 분류, 미스 예상 연산자는 GPU 슬롯 여유 시간에 선제 배치
  6. DAGTopologyScheduler와의 차별점: DAGTopologyScheduler가 DAG 위상 순서 기반으로 KV 보존만 결정하는 반면, QueryPlanScheduler는 쿼리 최적화 원리(CSE, 선제 캐싱 계획)로 KV 생성 순서 자체를 재배열
  7. 폴백: 플랜 구조 미지 요청 → `DAGTopologyScheduler`(이전 사이클 구현)로 위임
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 공통 KV를 생성하는 연산자를 모든 후속 연산자가 접근 가능한 노드에 배치(Helium의 "KV 상태 지역성-인식 배치"). 멀티노드에서는 공통 KV를 단일 노드에 집중 생성 후 RDMA로 브로드캐스트하는 전략이 각 노드에서 독립 재계산보다 효율적.
- **예상 효과**: 처리량 +30~45% (Helium 실증 1.56× 기반, DAGTopologyScheduler 대비 추가 +10~15%p), 캐시 히트율 +15~25%p (공통 서브트리 선제 캐싱), Scheduling overhead TTFT +4% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 A-2: TokenCake 공간+시간 이중 스케줄러 — DAG 에이전트 캐시 경쟁 방지 + 도구 호출 선제 오프로딩
- **가설**: 에이전틱 DAG에서 에이전트 간 KV 캐시 메모리 경쟁(공간 문제)과 도구 호출 대기 중 GPU 메모리 낭비(시간 문제)를 단일 스케줄러로 해결하는 대신, 공간 스케줄러와 시간 스케줄러를 분리 설계하면 각 문제를 독립적으로 최적화할 수 있다. 이전 사이클의 DAGTopologyScheduler가 KV 보존 결정만 수행한 것과 달리, 이 설계는 메모리 파티셔닝(공간)과 오프로딩 타이밍(시간)을 동시에 제어한다.
- **접근 방법**:
  1. `SpatialCachePartitioner` 클래스 (`src/scheduler/spatial_cache_partitioner.py`): 중요 에이전트 KV 보호
  2. 동적 메모리 파티션: DAG 상의 각 에이전트에 "보호 파티션(protected partition)" 할당 — 크리티컬 패스 에이전트는 보호 파티션에서 퇴거 불가
  3. `TemporalOffloadScheduler` 클래스 (`src/scheduler/temporal_offload_scheduler.py`): 도구 호출 중 오프로딩 제어
  4. 도구 호출 지연 예측: 도구 종류별 평균 지연(DB 쿼리 ~100ms, API ~500ms 등) 프로파일링 → 지연 > threshold_offload 시 비중요 에이전트 KV 선제 CPU 오프로딩
  5. 자동 재로드: 도구 호출 완료 예상 시점 - prefetch_window 전에 CPU → GPU 예측적 재로드
  6. DAGTopologyScheduler와의 차별점: DAGTopologyScheduler가 위상 순서 기반 KV 보존 확률을 계산하는 단일 계층인 반면, 이 설계는 메모리 파티셔닝과 오프로딩 타이밍을 독립 제어하는 이중 계층 아키텍처
  7. Continuum(TTL 핀닝)과의 차별점: Continuum이 단일 턴 재진입 TTL을 관리하는 반면, 이 설계는 DAG 전체 에이전트 KV 생명주기를 공간/시간 양 차원에서 동시 최적화
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 보호 파티션을 NVLink 클러스터의 단일 노드 HBM에 집중 배치해 크로스-노드 KV 접근을 최소화. 도구 호출 오프로딩 대상을 로컬 CPU DRAM 우선, 용량 초과 시 RDMA 원격 DRAM 순서로 계층화.
- **예상 효과**: 처리량 +20~30% (도구 호출 대기 시간 동안 GPU 활용 최대화), 캐시 퇴거율 −40~50% (보호 파티션 에이전트 KV 보존), TTFT p50 오버헤드 +3% 이내
- **구현 난이도**: medium
- **우선순위**: 3

## Activity B 아이디어

### 아이디어 B-1: TokenDance Diff-Aware Storage — 에이전트 간 KV 마스터+블록-희소 차분 저장
- **가설**: 멀티에이전트 All-Gather 패턴에서 매 라운드마다 에이전트들이 공유 출력 블록을 포함하는 유사 프롬프트를 처리할 때, 공통 KV를 단일 마스터 복사본으로 저장하고 개별 에이전트의 차이만 블록-희소 차분(block-sparse diff)으로 인코딩하면, 이전 사이클의 모든 세그먼트 캐시(SegmentedHashCache, SemanticSegmentCache, WorkloadAwareTTLCache)가 각 에이전트의 KV를 독립적으로 저장하는 것과 달리 17.5× KV 저장 절감과 2.7× 동시 에이전트 지원이 가능하다.
- **접근 방법**:
  1. `DiffAwareSegmentStore` 클래스 (`src/cache/diff_aware_store.py`): `CacheStore` 인터페이스 상속
  2. 마스터 KV 등록: `register_master(master_kv, group_id)` → 에이전트 그룹의 공통 KV를 마스터로 저장
  3. 차분 인코딩: `put_agent_kv(agent_id, group_id, agent_kv)` → 마스터와의 블록-희소 차분 계산
      - 블록 단위(기본 64 토큰): 마스터 블록과 에이전트 블록 간 L2 거리 > diff_threshold이면 차분 블록으로 저장, 이하이면 마스터 참조 포인터만 저장
      - 희소성: 실제 변경된 블록만 저장 → 저장 공간 ∝ 변경 비율
  4. 차분 복원: `get_agent_kv(agent_id, group_id)` → 마스터 KV + diff 블록 병합 → 에이전트 전체 KV 재구성
  5. 그룹 관리: 그룹 내 모든 에이전트의 마스터 참조를 LRU로 관리 → 그룹 마스터 퇴거 시 모든 에이전트 차분도 함께 퇴거
  6. 기존 `SemanticSegmentCache`와 스택: 의미 유사도 검색 → 히트된 세그먼트가 에이전트 그룹에 속하면 `DiffAwareSegmentStore`로 라우팅
  7. FAISS 병목 우회: 검색 공간을 마스터 복사본(그룹 수)으로 제한 → N>10K 병목 문제 구조적 해소 (SUMMARY.md 미해결 항목 직접 해결)
- **예상 효과**: 비연속 히트율 +20~25%p (에이전트 간 공통 KV 즉시 히트), KV 저장 공간 −85~94% (17.5× 절감 기준, 멀티에이전트 시나리오), TTFT −30~50% (마스터 공유로 재계산 제거), 동시 에이전트 지원 2.7× 확장
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 B-2: LRAgent 기저/어댑터 KV 분해 캐싱 — LoRA KV를 공유 기저 + 저랭크 어댑터 성분으로 분리
- **가설**: 멀티-LoRA 에이전트 환경에서 각 에이전트가 동일 프롬프트에 대해 독립 KV를 생성하는 낭비를 제거하기 위해, KV를 공유 사전학습 기저 성분(base)과 어댑터 의존 저랭크 성분(adapter-dependent)으로 분해하면, 기저 성분은 에이전트 간 공유하고 어댑터 성분만 저랭크로 저장해 이전 사이클의 SignVQSegmentCache나 SemanticSegmentCache가 처리하지 못한 "LoRA 어댑터 다양성에 따른 구조적 KV 다름"을 정확히 반영하면서 SUMMARY.md의 approx_sign Stage 2 미발동 구조 문제를 원천 해소할 수 있다.
- **접근 방법**:
  1. `LoRADecomposedKVCache` 클래스 (`src/cache/lora_decomposed_kv.py`): `CacheStore` 인터페이스 상속
  2. KV 분해: `decompose_kv(kv_tensor, lora_adapter_id)` → 기저 성분(FP16, 공유) + 어댑터 성분(저랭크 r×d, 어댑터별)
      - 기저 성분 추정: KV 텐서를 공유 기저 행렬(base_kv)과 잔차로 분리. base_kv는 동일 레이어에서 여러 어댑터 요청의 평균 KV로 온라인 업데이트
      - 어댑터 성분: `adapter_kv = kv_tensor - base_kv` → SVD로 랭크-r 근사(r = d_head × 0.25)
  3. 공유 기저 캐시: `BaseKVCache`: 레이어별 기저 KV를 단일 사본 저장, 어댑터 무관하게 모든 요청에 재사용
  4. 어댑터 성분 캐시: `AdapterKVCache`: 어댑터 ID + 세그먼트 해시 키로 저랭크 성분 저장
  5. 런타임 재구성: `get_kv(segment_id, adapter_id)` → `base_kv + reconstruct(adapter_delta_r)` → 전체 KV 복원 (FlashAttention 시 저랭크 성분을 별도 구체화 없이 직접 반영)
  6. FASTLIBRA 유효성 추적 통합: `validity_tracker`로 base_kv 업데이트 시 무효화된 어댑터 성분을 자동 표시 (46.5% 무효 KV 실측 문제 해소)
  7. 기존 `LoRAAwareRouter`(A, 이전 사이클)와 연동: 라우팅 키에 `(adapter_id, segment_hash)` 복합 키 대신 `(adapter_id, base_segment_hash, rank_r)` 3중 키 사용
- **예상 효과**: 비연속 히트율 +15~20%p (기저 성분 즉시 재사용), KV 저장 공간 −60~70% (기저 공유 + 저랭크 어댑터), TTFT −25~40% (기저 재사용으로 재계산 감소)
- **구현 난이도**: medium
- **우선순위**: 2

## Activity C 아이디어

### 아이디어 C-1: NQKV 정규분포 기반 블록-분위수 양자화 — 변환 오버헤드 없는 Normal Float INT4
- **가설**: KV 캐시 원소들이 정규분포를 따른다는 관찰에 기반해 블록-분위수 양자화(per-block quantile quantization)를 적용하면, 이전 사이클의 모든 양자화 기법(Hadamard 회전+INT4, PolarQuant+QJL 랜덤 회전, CapKV 레버리지 스코어)이 공통으로 필요로 하는 "행렬 변환(회전 행렬 적용, SVD 등)" 오버헤드 없이 이론적 최적 양자화 오류로 INT4를 달성할 수 있다. 이는 SUMMARY.md에서 "실제 GPU 처리량 미검증"과 "실제 perplexity 미측정" 두 미해결 항목을 해소하기 위한 가장 단순한 구현 경로다.
- **접근 방법**:
  1. `NQKVCodec` 클래스 (`src/cache/nqkv_codec.py`): `CompressionCodec` 인터페이스 구현
  2. 블록 분위수 계산: `encode_kv(kv_block)` → 블록 내 원소의 정규분포 파라미터(μ, σ) 추정 → 4비트 Normal Float 분위점에 매핑
      - Normal Float 분위점: `{-∞, Φ^{-1}(1/16), Φ^{-1}(2/16), ..., Φ^{-1}(15/16), +∞}` 15개 임계값으로 16개 구간 구성
      - 블록 크기: 기본 64 원소 (채널 단위 또는 토큰 단위 선택 가능)
  3. 분포 파라미터 저장: 블록당 (μ, σ) 2개 FP16 스칼라만 추가 저장 (전체 오버헤드 << 1%)
  4. 역양자화: `decode_kv(compressed_block, mu, sigma)` → Normal Float → FP16 복원
  5. 기존 `CompressionCodec` 인터페이스 drop-in 교체 (기존 `CompressedSegmentCache` 호환)
  6. **GPU 처리량 검증 경로**: NQKV가 변환 없이 Pure Python → Triton → CUDA 순으로 커널 구현 가능 → torch.cuda.Event TTFT/TBT 실측 스크립트 작성 (SUMMARY.md 미해결 항목 1 직접 해결)
  7. **실제 perplexity 검증**: GPT-2 + WikiText-2 perplexity를 NQKVCodec ON/OFF로 직접 비교 (SUMMARY.md 미해결 항목 5 직접 해결)
- **accuracy-preserving 근거**: NQKV가 정보이론적으로 최적 양자화 오류를 달성함을 실증(OPT 모델 기준 제로-샷 태스크 정확도 저하 무시 가능). 정규분포 가정의 타당성: KV 캐시 K/V 벡터는 LayerNorm 직후 생성되므로 정규분포에 근사. 블록-분위수 방식이 채널별 이상치(outlier)를 블록 내 분포 파라미터로 흡수해 별도 이상치 처리 불필요. 검증: GPT-2 WikiText-2 perplexity ±1% (직접 실측) + LLaMA-3 제로-샷 태스크 5개 정확도 비교.
- **예상 효과**: 메모리 −72~75% (4비트, FP32 대비), accuracy delta ±0.5% 이내, 2× 배치 크기 또는 4× 컨텍스트 길이 지원 (NQKV 실증 기준), 처리량 +9× (NQKV 9.3× 기반)
- **구현 난이도**: low
- **우선순위**: 1

### 아이디어 C-2: FireQ RoPE-인식 2단계 아웃라이어 평활화 — pre-RoPE 채널 쌍 정규화 + post-RoPE 표적 스케일링
- **가설**: RoPE(Rotary Position Embedding)가 적용된 Key 벡터를 INT4로 양자화하면 RoPE가 생성하는 이상치(outlier)로 인해 정확도가 급격히 저하된다. pre-RoPE 단계에서 채널 쌍별 정규화(채널 회전 쌍의 분산 균형)와 post-RoPE 단계에서 표적 스케일링(잔여 이상치 개별 처리)의 2단계 아웃라이어 평활화를 적용하면, 이전 사이클의 HadamardInt4Codec(Hadamard 회전이 RoPE 이상치를 분산시키지만 채널 쌍 구조를 고려하지 않음)과 TurboQuantCodec(PolarQuant 랜덤 회전이 RoPE와 구조적으로 간섭 가능)이 다루지 못한 "RoPE-INT4 충돌"을 명시적으로 해소할 수 있다.
- **접근 방법**:
  1. `FireQCodec` 클래스 (`src/cache/fireq_codec.py`): `CompressionCodec` 인터페이스 구현
  2. Stage 1 — pre-RoPE 채널 쌍 정규화: RoPE 회전 쌍 (채널 i, 채널 i+d/2)의 분산을 동등하게 스케일링 → 회전 전 이상치 균형
      - 스케일 팩터: `s_i = sqrt(var(K[:, i]) / var(K[:, i + d/2]))` — 캘리브레이션 배치에서 추정
  3. Stage 2 — post-RoPE 표적 스케일링: RoPE 적용 후 여전히 이상치로 남은 채널을 탐지 → 해당 채널만 추가 스케일링
      - 이상치 기준: `|K_rotated[:, c]| > outlier_threshold` (기본값: 3σ)
  4. INT4 양자화: 평활화된 Key를 블록-스케일 INT4로 양자화 (`encode_kv(kv, layer_idx, rope_applied=True)`)
  5. Value는 FP8로 양자화 (FireQ 원논문 혼합 정밀도 설계: Key INT4 + Value FP8)
  6. 스케일 팩터 저장: 레이어별 pre-RoPE 스케일 + post-RoPE 이상치 채널 마스크를 모델 초기화 시 1회 캘리브레이션
  7. **GPU 처리량 검증 경로**: Triton 커널로 FlashAttention-3 통합 경로 구현 시 Hopper GPU에서 1.26× 프리필 가속 달성 가능. CPU 환경에서는 NumPy 기반 검증 후 CUDA 이식.
  8. 기존 `TurboQuantCodec`과의 차별점: FireQ는 RoPE 구조를 명시적으로 인식해 채널 쌍 기반 처리를 수행; TurboQuantCodec은 랜덤 회전으로 이상치 분산을 균일화하지만 RoPE 쌍 구조를 고려하지 않음.
- **accuracy-preserving 근거**: FireQ가 Llama2-7B/Llama3-8B에서 INT4+FP8 혼합 정밀도로 QServe 대비 1.26~1.68× 가속을 달성하면서 정확도 손실이 무시 수준임을 실증. 2단계 평활화의 정확도 보존 메커니즘: Stage 1이 RoPE 회전으로 증폭된 이상치를 회전 전에 제거하고, Stage 2가 잔여 이상치를 표적 처리해 INT4 양자화 오류 곡선을 정규분포에 근접시킴. 검증: WikiText-2 perplexity ±1% + LLaMA-3-8B MMLU + LongBench 서브태스크 3개.
- **예상 효과**: 메모리 −75% (Key INT4 + Value FP8 혼합), 처리량 +26% 프리필 가속 (FireQ 실증, GPU 환경), accuracy delta ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 C-3: TailorKV 레이어별 오프로딩+양자화 하이브리드 — 단일 소비자급 GPU 128K 컨텍스트 지원
- **가설**: 레이어별 특성에 따라 "전역 정보 유지가 필요한 레이어"는 선택적 토큰 로딩(중요 토큰만 DRAM에서 GPU로 로드)으로, "소수 토큰에 집중하는 레이어"는 전체 토큰 1~2비트 공격적 양자화로 처리하는 하이브리드 전략을 적용하면, 이전 사이클의 TwoDimBudgetScheduler(레이어 × 단계 2D 예산 행렬)가 양자화 비트폭만 조정한 것과 달리 오프로딩과 양자화를 레이어별로 교대 적용해 단일 RTX 3090 GPU(24GB)에서 128K 컨텍스트 처리가 가능하다.
- **접근 방법**:
  1. `TailorKVManager` 클래스 (`src/cache/tailorkv_manager.py`): `CacheStore` 인터페이스 상속
  2. 레이어 특성 프로파일링: 소수 캘리브레이션 배치에서 레이어별 어텐션 분산(entropy) 계산
      - 낮은 entropy → 소수 토큰 집중 레이어 → "양자화 레이어" 분류
      - 높은 entropy → 전역 정보 보존 레이어 → "선택적 로딩 레이어" 분류
  3. 양자화 레이어 처리: `NQKVCodec`(C-1) 1~2비트 공격적 양자화 → GPU HBM 내 압축 저장
  4. 선택적 로딩 레이어 처리: 전체 KV를 CPU DRAM에 저장 → 어텐션 계산 시 상위-k 중요 토큰(어텐션 스코어 기준)만 선택적으로 GPU로 로드
  5. 레이어 분류 임계값: `entropy_threshold` 설정 파라미터로 노출 (기본 TailorKV 논문 설정 참조)
  6. `WorkloadAwareTTLCache`(이전 사이클 B)와 연동: TTL 만료 임박 레이어 KV를 우선 CPU 오프로딩 대상으로 설정
  7. TwoDimBudgetScheduler와의 차별점: TwoDimBudgetScheduler가 레이어 × 단계 2D 예산 행렬(양자화 비트폭 할당)인 반면, TailorKVManager는 레이어별로 오프로딩 전략(선택적 로딩) 또는 양자화 전략(1~2비트 NF)을 완전히 다른 메커니즘으로 적용
- **accuracy-preserving 근거**: TailorKV(ACL Findings 2025)가 Llama-3.1-8B 128K 컨텍스트를 단일 RTX 3090(24GB)에서 82ms/token으로 처리하면서 정확도를 유지함을 실증. 두 전략의 상보적 설계: 양자화 레이어는 소수 토큰에 집중하므로 높은 압축에서도 실질 정보 손실 낮음; 선택적 로딩 레이어는 중요 토큰을 온전히 로드하므로 전역 정보 보존. 검증: LongBench + RULER 128K+ long-context + WikiText-2 perplexity ±1%.
- **예상 효과**: 메모리 −70~80% (오프로딩+압축 결합), 유효 컨텍스트 길이 4~8× 확장(128K 단일 GPU 달성), accuracy delta ±1% 이내, 처리량 82ms/token (단일 RTX 3090 기준)
- **구현 난이도**: medium
- **우선순위**: 3

### 아이디어 C-4: Lookahead Q-Cache — pseudo query로 프리필-디코딩 어텐션 불일치 해소 기반 KV 퇴거
- **가설**: 기존 KV 퇴거 정책(H2O, SnapKV, LookaheadKV ICLR 2026 등)이 프리필 단계의 어텐션 스코어를 사용해 실제 디코딩 쿼리와 불일치하는 문제를, 응답 접두사를 경량으로 사전 생성한 "pseudo query"를 관측 창으로 사용해 해소하면, 이전 사이클의 HybridEvictionPolicy(CAOTE+StructKV)나 RedundancyAwareEvictionPolicy가 "어텐션 출력 기여도 + 중복성"으로 퇴거를 결정한 것과 달리 "실제 디코딩 단계에서의 어텐션 패턴"을 프리필 시점에 미리 반영해 퇴거 결정 품질이 향상된다.
- **접근 방법**:
  1. `LookaheadQCacheEviction` 클래스 (`src/cache/lookahead_q_cache.py`): `CacheStore` 인터페이스 내 eviction policy로 구현
  2. pseudo query 생성: 메인 추론 전 경량 샘플링(greedy decoding, max_new_tokens=8~16)으로 응답 접두사 생성 → 응답 접두사 토큰을 pseudo query로 사용
  3. 관측 창 어텐션: pseudo query와 프리필 KV 간 어텐션 스코어 계산 → 실제 디코딩 쿼리 패턴 시뮬레이션
  4. 퇴거 결정: pseudo query 어텐션 스코어 기반 토큰 중요도 재정렬 → 기존 프리필 어텐션 스코어(α=0.5) + pseudo query 스코어(α=0.5) 가중 평균
  5. LookaheadKV(ICLR 2026)와의 차별점: LookaheadKV가 파라미터-효율 모듈(PEM)로 미래 응답을 예측하는 반면, 이 접근은 실제 응답 접두사(8~16 토큰)를 경량 생성해 사용 — 추가 학습 없음
  6. 비용-효율 트레이드오프: pseudo query 생성 비용(8~16 토큰 greedy 디코딩) ≪ 퇴거 정확도 향상에 따른 재계산 비용 절감
  7. `RedundancyAwareEvictionPolicy`(이전 사이클 C)와 결합 가능: pseudo query 스코어로 1차 필터링 후 중복성 스코어로 2차 정렬
- **accuracy-preserving 근거**: Lookahead Q-Cache(EMNLP 2025)가 프리필-디코딩 불일치를 pseudo query로 해소해 기존 KV 퇴거 방법보다 일관적으로 우수한 정확도를 달성함을 실증. pseudo query 어텐션 패턴이 실제 디코딩 패턴을 더 정확하게 반영하므로 중요 토큰이 잘못 퇴거되는 false negative를 줄임. 검증: LongBench + RULER + WikiText-2 perplexity ±1% + LLaMA-3-8B MMLU.
- **예상 효과**: 퇴거 정확도 향상으로 동일 KV 예산에서 perplexity −5~10% 추가 감소, 메모리 −30~50% (퇴거 품질 향상으로 더 공격적 예산 가능), accuracy delta ±0.7% 이내
- **구현 난이도**: low
- **우선순위**: 4

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: B+C — DiffAwareSegmentStore + NQKVCodec (차분 저장 + 변환-무료 INT4 압축)
- **조합 근거**: DiffAwareSegmentStore(B-1)는 에이전트 간 KV 차분(diff)만 저장해 저장 공간을 17.5× 줄이지만, 마스터 KV 자체의 압축은 다루지 않는다. NQKVCodec(C-1)으로 마스터 KV를 INT4로 추가 압축하면, (마스터 INT4 압축) + (차분 희소 저장) 두 효과가 승수적으로 결합되어 총 저장 공간이 FP16 독립 저장 대비 50~70× 감소한다. 변환 오버헤드가 없는 NQKVCodec이 차분 저장의 경량성과 잘 어울린다.
- **구체 설계**:
  1. `CompressedDiffStore` 클래스 (`src/cache/compressed_diff_store.py`): `DiffAwareSegmentStore` 상속
  2. 마스터 KV 저장: `register_master()` 시 NQKVCodec으로 INT4 압축 → 압축된 마스터 저장
  3. 차분 저장: 차분 블록만 FP16 원본으로 저장 (차분은 규모가 작으므로 압축 오버헤드 없음)
  4. 복원: `get_agent_kv()` → 마스터 INT4 역양자화 + 차분 블록 병합 → 에이전트 KV 재구성
  5. 정확도 보존: 차분 블록을 원본 FP16으로 유지하므로 에이전트별 특수성 정보 손실 없음
- **예상 복합 효과**: 처리량 +30~40%, 메모리 −92~96% (17.5× 차분 절감 × 4× INT4 압축), 비연속 히트율 +25~30%p, accuracy delta ±0.6% 이내
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 Cross-2: B+C — LoRADecomposedKVCache + FireQCodec (기저 INT4+Value FP8 비대칭 계층 압축)
- **조합 근거**: LoRADecomposedKVCache(B-2)가 KV를 공유 기저 성분과 저랭크 어댑터 성분으로 분해할 때, 공유 기저 성분에 FireQCodec(C-2)의 INT4 압축을 적용하고 어댑터 성분은 FP8 고정밀도로 유지하면, "기저는 고압축, 어댑터는 고정밀도"라는 비대칭 계층 압축 파이프라인이 실현된다. 기저는 에이전트 간 공유되어 압축 비용이 1회만 발생하고, 어댑터 성분은 소규모 저랭크이므로 FP8 유지 비용이 낮다.
- **구체 설계**:
  1. 기저 성분: `FireQCodec.encode(base_kv, layer_idx, rope_applied=True)` → RoPE-인식 INT4 압축
  2. 어댑터 성분(저랭크): FP8 또는 FP16으로 원본 정밀도 유지 (랭크-r, r ≤ d_head/4이므로 저장 비용 최소)
  3. 복원: `FireQCodec.decode(base_compressed) + adapter_delta` → 전체 KV 재구성
  4. 어댑터 성분 정확도 우선: FireQ의 RoPE-인식 평활화가 기저 성분의 RoPE 이상치를 처리하므로, 어댑터 성분(잔차)의 이상치 규모가 기저 대비 작아 FP8으로도 충분한 정밀도 유지
- **예상 복합 효과**: 처리량 +25~35%, 메모리 −80~85% (기저 INT4 압축 + 어댑터 저랭크 저장), 비연속 히트율 +18~22%p, accuracy delta ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 Cross-3: A+B+C — QueryPlanScheduler + DiffAwareSegmentStore + NQKVCodec (에이전틱 증분 KV 전파 통합)
- **조합 근거**: Helium의 교훈(트렌드 리포트 A+B 시너지 섹션)을 직접 구현한다. QueryPlanScheduler(A-1)가 워크플로우 쿼리 플랜에서 에이전트 간 공통 서브트리를 탐지하고, DiffAwareSegmentStore(B-1)가 공통 KV를 마스터로 저장 후 차분만 전파하며, NQKVCodec(C-1)이 마스터를 INT4로 압축한다. 세 기법이 파이프라인으로 결합되면 에이전트 단위 "증분적 KV 전파"가 완성된다: 쿼리 플랜이 공통 KV를 식별 → 마스터+차분 구조로 효율 저장 → INT4 압축으로 전송 비용 최소화.
- **구체 설계**:
  1. 쿼리 플랜 분석: `QueryPlanScheduler`가 공통 서브트리를 탐지 → `DiffAwareSegmentStore.register_master(group_id)` 호출
  2. 에이전트 KV 생성 시: `DiffAwareSegmentStore.put_agent_kv()` → 차분 계산 → 차분 블록 저장
  3. 마스터 KV: `NQKVCodec.encode()` → INT4 압축 저장 (한 번만 압축)
  4. 멀티노드 KV 전파: 다른 노드의 에이전트에 마스터 INT4 KV + 차분 블록만 전송 (FP16 전체 KV 전송 대비 50~70× 트래픽 감소)
  5. 복원: 수신 노드에서 마스터 역양자화 + 차분 병합 → 에이전트 전체 KV 재구성
- **예상 복합 효과**: 처리량 +40~55%, 메모리 −92~96%, 비연속 히트율 +30~35%p (전체 히트의 60%+ 목표), 노드 간 KV 전송 트래픽 −95~98%, accuracy delta ±0.6% 이내
- **구현 난이도**: high
- **우선순위**: 3

## 이번 사이클 최우선 구현 타겟

**Cross-1(B+C): DiffAwareSegmentStore(B-1) + NQKVCodec(C-1) + CompressedDiffStore 통합**, 보조로 FireQCodec(C-2) 구현

구현 순서:
1. **C-1** `NQKVCodec`: 블록-분위수 Normal Float INT4 양자화. `CompressionCodec` 인터페이스 drop-in. 단순 구현(low 난이도)으로 빠른 배포. **GPT-2 + WikiText-2 perplexity 직접 측정 스크립트 포함** (SUMMARY.md 미해결 항목 5 해소).
2. **B-1** `DiffAwareSegmentStore`: 마스터+블록-희소 차분 저장. `CacheStore` 인터페이스 구현. FAISS N>10K 병목 우회 구조 내장 (SUMMARY.md 미해결 항목 2 해소).
3. **Cross-1** `CompressedDiffStore`: B-1 + C-1 통합. 마스터 INT4 압축 + 차분 FP16 저장 파이프라인 연결.
4. **C-2** `FireQCodec`: RoPE-인식 2단계 평활화 + INT4+FP8 혼합. 캘리브레이션 스크립트 포함. **torch.cuda.Event TTFT 실측 스크립트 포함** (SUMMARY.md 미해결 항목 1 해소).

이유:
- NQKVCodec은 변환 없는 단순 분위수 양자화로 구현 난이도 low이며 실제 perplexity 직접 측정이 가장 용이한 구조 → SUMMARY.md 미해결 항목 5 직접 해소.
- DiffAwareSegmentStore는 FAISS 검색 공간 자체를 마스터 수(에이전트 그룹 수)로 축소해 N>10K 병목을 구조적으로 우회 → SUMMARY.md 미해결 항목 2 직접 해소.
- FireQCodec은 RoPE-INT4 충돌이라는 새로운 차원의 문제를 해결하며 Hopper GPU 특화 GEMM 커널로 GPU 처리량 실측 경로를 제공 → SUMMARY.md 미해결 항목 1 직접 해소.
- Cross-1 통합이 92~96% 메모리 절감으로 목표 −30%를 10× 이상 초과 달성하고 비연속 히트율 +25~30%p 향상으로 연구 목표를 폭넓게 커버.
