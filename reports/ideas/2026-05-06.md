## 이번 사이클 초점
**B+C (쿼리 중심 비연속 재사용 + pre-RoPE 안정적 압축)**

**선택 이유**:
1. 이전 사이클(2026-05-05)이 B+C(DiffAwareSegmentStore + NQKVCodec + FireQCodec)를 최우선으로 구현했으므로, 이번에는 완전히 새로운 B+C 기반인 "쿼리 중심 재계산 + pre-RoPE 중요도 추정"으로 전환해 비연속 히트율 목표(≥30%)와 압축 정확도 목표(±1%)를 더 강력하게 달성한다.

3. TriAttention의 vLLM/SGLang 통합 오픈소스가 6단계 vLLM 이식 후보로 바로 적용 가능하며, AIME25 32K CoT 검증이 추론 모델에서의 정확도 ±1% 달성 근거를 강화한다.
4. Activity A의 OnlineLatencyOptimalScheduler와 ContiguousChunkIOScheduler는 이론 완전성과 시스템 설계 원칙으로 이번 사이클 B+C 구현의 배경 인프라로 병행 설계한다.

## Activity A 아이디어

### 아이디어 A-1: OnlineLatencyOptimalScheduler — 준-온라인/완전-온라인 이론 최적 KV 메모리 인식 배칭
- **가설**: 응답 길이를 사전에 알 수 없는 온라인 환경에서 KV 캐시 메모리 예산 제약 하에 평균 레이턴시를 최소화하는 문제를, 준-온라인(semi-online) 모델에서는 다항시간 최적 알고리즘으로, 완전 온라인(fully online) 확률적 도착 모델에서는 상수 후회(constant regret) 알고리즘으로 해결하면, 이전 사이클의 모든 경험적 A 기법(DAGTopologyScheduler, QueryPlanScheduler, SpatialCachePartitioner 등)이 이론 최적성을 보장하지 못하는 반면 "메모리 예산 위반 없는 수학적 레이턴시 최적화"가 가능하다.
- **접근 방법**:
  1. `OnlineLatencyOptimalScheduler` 클래스 (`src/scheduler/online_latency_optimal.py`): `BaseScheduler` 상속
  2. 준-온라인 모드: 프롬프트 도착 시 길이 상한 힌트(optional)를 받아 다항시간 최적 배치 결정. 힌트 미제공 시 완전-온라인 모드로 전환.
  3. 완전-온라인 모드: Online Scheduling(2502.07115) 알고리즘 — 확률적 도착 모델에서 누적 레이턴시 후회 최소화. 실측 요청 트레이스 히스토그램으로 도착 분포 온라인 추정.
  4. KV 메모리 예산 제약 통합: `available_kv_budget` 파라미터로 현재 사용 가능한 KV 슬롯 수를 실시간 주입. 배치 결정 시 예산 위반 방지.
  5. ProphetKV 재계산 예산 연동: 비연속 재사용에서 "20% 재계산 예산"을 KV 메모리 예산 모델에 추가 제약으로 통합 — "KV 메모리 예산 + 재계산 비용 예산" 이중 제약 최적화.
  6. 기존 DAGTopologyScheduler와 병렬 운용: DAG 구조 있는 워크플로우는 DAGTopologyScheduler, 일반 요청은 OnlineLatencyOptimalScheduler로 처리.
  7. 이전 A 기법과의 차별점: 이전 사이클의 모든 A 기법이 경험적 휴리스틱(DAG 위상, 의미 히트율, TTL, 공간/시간 분리)인 반면, 이 설계는 준-온라인 최적성과 완전-온라인 상수 후회를 수학적으로 보장한다.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드 환경에서 KV 메모리 예산을 노드별이 아닌 클러스터 전체로 모델링. 노드 간 KV 전송 비용을 레이턴시 목적 함수에 추가 항으로 반영. 완전-온라인 모드에서 분산 도착 프로세스를 각 노드별 서브-프로세스 합으로 분해.
- **예상 효과**: 처리량 +20~30% (이론 최적 배치로 KV 메모리 낭비 제거), TTFT p50 오버헤드 ≤+5% (이론 상수 후회 보장), KV 메모리 예산 위반 0%

- **우선순위**: 2

### 아이디어 A-2: ContiguousChunkIOScheduler — 세그먼트 경계 I/O 블록 정렬 + 레이어 간 비동기 프리패칭 KV 오프로딩 스케줄러
- **가설**: KV 캐시 오프로딩 시스템과 비연속 세그먼트 프루닝 알고리즘 사이의 세분화 불일치(granularity mismatch) 문제를, 세그먼트 경계를 I/O 블록 경계(16~64 토큰 ContiguousChunk 단위)와 일치시키고 레이어 간 비동기 프리패칭으로 I/O-컴퓨팅 순차 의존성을 제거하면, 이전 사이클의 모든 Activity A 기법이 알고리즘 수준 최적화에 집중한 것과 달리 Re-Prefill 단계 3.85× 가속이라는 시스템-레벨 성능 향상을 달성할 수 있다.
- **접근 방법**:
  1. `ContiguousChunkIOScheduler` 클래스 (`src/scheduler/contiguous_chunk_io.py`): `BaseScheduler` 상속
  2. ContiguousChunk 단위 정의: `chunk_size` 파라미터(기본 16 토큰, ContiguousKV 권장값). 세그먼트 경계를 청크 경계와 강제 정렬(padding 허용).
  3. I/O-프루닝 정렬: 프루닝 알고리즘이 선택한 중요 토큰을 청크 단위로 반올림 → 중요 청크 인덱스(chunk-wise importance index) 생성. 읽기 증폭(read amplification) 없는 비연속 세그먼트 로드.
  4. 레이어 간 비동기 프리패칭: 레이어 L의 중요 청크 인덱스와 레이어 L+1의 인덱스 유사성 활용 → GPU가 레이어 L 어텐션 계산 중 CPU/NVMe가 레이어 L+1 청크 로드(I/O-컴퓨팅 오버랩).
  5. `SegmentedHashCache`와 연동: 세그먼트 해시 키를 청크 경계 정렬된 토큰 범위로 재정의 → 세그먼트 로드 시 ContiguousChunk 단위 I/O 사용.
  6. vLLM v0.9 비동기 KV Connector 통합 경로: 비동기 KV Connector의 전송 단위를 ContiguousChunk로 설정해 레이어 간 비동기 프리패칭을 vLLM에서 직접 활용.
  7. 이전 A 기법과의 차별점: 이전 사이클의 A 기법이 모두 요청/워크플로우 단위 스케줄링 알고리즘인 반면, 이 설계는 I/O 블록 정렬이라는 시스템 하드웨어 계층 최적화를 스케줄러 설계 원칙으로 통합한다.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드 Disaggregated Prefill(P/D 분리) 환경에서 디코드 측의 Re-Prefill 단계(오프로딩된 KV 로드 후 첫 토큰)가 ContiguousChunk 단위 비동기 프리패칭으로 3.85× 가속됨. 노드 간 KV 전송 시에도 ContiguousChunk 단위를 RDMA 전송 단위로 정렬해 단편화 없는 고속 전송.
- **예상 효과**: Re-Prefill 단계 처리량 +2.85~3.85× (ContiguousKV 실증 기준), 비연속 세그먼트 로드 지연 −70~80% (읽기 증폭 제거), Scheduling overhead TTFT +3% 이내
- **구현 난이도**: medium
- **우선순위**: 3

## Activity B 아이디어

### 아이디어 B-1: QueryCentricRecomputeCache — 사용자 쿼리 관련성 기반 이중 단계 비연속 KV 재계산 예산 배분
- **가설**: 비연속 세그먼트 재사용 시 "전역 중요도(어텐션 스코어)" 기준으로 재계산 예산을 배분하면 "쿼리 비관련 고현저성 토큰"이 제한된 예산을 낭비하는 밀어내기 효과(crowding-out effect)가 발생하는데, 사용자 쿼리와의 의미 관련성을 기준으로 재계산 대상 토큰을 우선화하는 이중 단계 재계산 파이프라인(ProphetKV 원리)을 적용하면, 이전 사이클의 모든 B 기법(정확 해시, 의미 유사도/DHD, TTL, 차분 저장, LoRA 분해, 모델 지시형 퇴거)이 쿼리 관련성을 재계산 기준으로 명시적으로 활용하지 않은 것과 달리 20% 재계산으로 96~101% 정확도 보존이 가능하다.
- **접근 방법**:
  1. `QueryCentricRecomputeCache` 클래스 (`src/cache/query_centric_recompute.py`): `CacheStore` 인터페이스 상속
  2. Stage 1 — 쿼리 관련성 스코어링: 캐시된 비연속 세그먼트별로 "사용자 쿼리 임베딩"과의 코사인 유사도 계산 → 관련성 상위 K% 세그먼트를 "재계산 후보"로 표시 (K=20, ProphetKV 실증 상한).
  3. Stage 2 — 레이어별 어텐션 메트릭 융합: 재계산 후보 세그먼트 내에서 레이어별 어텐션 메트릭(누적 어텐션 스코어)으로 토큰 단위 재계산 우선순위 결정 → 헤드-와이즈 편차 > 임계값인 토큰만 실제 재계산.
  4. 재계산 예산 제한: `recompute_budget_ratio` 파라미터(기본 0.20) — 전체 토큰의 최대 20%만 재계산 허용. 예산 초과 시 쿼리 관련성 하위 세그먼트 재계산 스킵.
  5. 쿼리 임베딩 생성: 사용자 쿼리 토큰의 평균 K 벡터(추가 모델 불필요, 기존 인코딩 재활용).
  6. `SegmentedHashCache` + `SemanticSegmentCache` 위에 재계산 레이어로 추가: 정확 해시 히트 → 쿼리 관련성 스코어 → 이중 단계 재계산 순으로 계층적 처리.
  7. 이전 B-1 SemanticSegmentCache(DHD)와의 차별점: SemanticSegmentCache가 세그먼트 간 L2 편차로 재계산 여부를 결정하는 반면, QueryCentricRecomputeCache는 사용자 쿼리와의 관련성으로 재계산 예산 배분 자체를 결정 — 예산 배분 주체가 "세그먼트 내부 편차"에서 "쿼리 외부 관련성"으로 전환.
- **예상 효과**: 비연속 히트율 +20~30%p (쿼리 관련 세그먼트 정밀 선택), TTFT −30~50% (재계산 최소화), accuracy delta ±0.8% 이내 (20% 재계산으로 96~101% 보존), 재계산 비용 −80% (전체 재계산 대비)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 B-2: InfoFlowChunkReorderCache — 정보 흐름 기반 비연속 세그먼트 최적 배치 순서 결정
- **가설**: 비연속 세그먼트를 재사용할 때 단순히 "어떤 세그먼트를 가져올 것인가"만이 아니라 "어떤 순서로 배치할 것인가"가 정확도에 유의미한 영향을 주는데, 어텐션-정규(attention-norm) 신호를 기반으로 "의미적으로 관련되고 정보 전파에 구조적으로 유리한 위치"를 식별해 비연속 세그먼트를 최적 순서로 재배치하면(InfoFlowKV 원리), 이전 사이클의 모든 B 기법이 원래 순서 보존 또는 단순 위치-독립 연결에 머문 것과 달리 정보 흐름 기반 배치 최적화라는 새 차원을 통해 추가 정확도 향상이 가능하다.
- **접근 방법**:
  1. `InfoFlowChunkReorderCache` 클래스 (`src/cache/infoflow_reorder.py`): `CacheStore` 인터페이스 상속
  2. 정보 흐름 신호 계산: `compute_infoflow_score(segment, query_embedding)` → 어텐션-정규(attention score × norm) 벡터를 쿼리로부터 계산. 이 신호는 RoPE 기하학 하에서 위치 편향 없이 계산.
  3. 청크 재순서화: 재사용할 비연속 세그먼트 집합에 대해 정보 흐름 스코어로 최적 배치 순서 결정 → "정보 전파에 유리한 세그먼트"를 컨텍스트 앞쪽에 배치.
  4. 위치 재할당: 재순서화된 세그먼트에 새로운 연속 위치 인덱스 할당(RoPE 재계산 포함). 기존 위치-독립 캐싱(PIC) 기법과의 차이: PIC는 위치 무관 저장인 반면, 이 기법은 재사용 시 정보 흐름 기반 새 위치를 적극 할당.
  5. `QueryCentricRecomputeCache`(B-1)와 직렬 결합: B-1이 재계산 예산을 배분한 후 B-2가 재사용 세그먼트 순서를 최적화 → 예산 배분 + 배치 최적화 이중 강화.
  6. 재순서화 비용 관리: 세그먼트 수가 N인 경우 O(N log N) 정렬. N ≤ 50인 일반적 RAG 시나리오에서 오버헤드 무시 가능.
  7. 이전 B 기법과의 차별점: 이전 사이클의 모든 B 기법(DHD, TTL, 차분, LoRA, 모델 지시형)이 세그먼트 "선택·저장·퇴거"에 집중한 반면, 이 기법은 선택된 세그먼트의 "배치 순서"라는 완전히 새로운 차원을 최적화한다.
- **예상 효과**: 비연속 히트율 +10~15%p (정보 흐름 기반 순서 최적화로 attention 품질 향상), accuracy delta −3~8% 추가 감소 (정보 흐름 최적 순서 적용 시), TTFT +2% 이내 (재순서화 오버헤드)
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 B-3: QCFuseSemanticAnchorCache — 의미 요약 앵커 기반 쿼리 강화 캐시 융합
- **가설**: 기존 캐시 융합(CacheBlend/EPIC) 방법이 로컬 관점의 토큰 선택에 의존해 사용자 쿼리의 전역 맥락을 활용하지 못하는 문제를, 의미 요약 앵커(semantic summary anchors)로 쿼리 표현을 강화하고 가장 중요한 트랜스포머 레이어의 어텐션 분포를 기준으로 쿼리 관련 토큰만 선택 재계산하면(QCFuse 원리), 이전 사이클의 SemanticSegmentCache/DHD가 "세그먼트 간 의미 유사도"로 히트 결정한 것과 달리 "캐시 융합 파이프라인 내 전역 쿼리 맥락"을 레이어 어텐션 분포 기반으로 활용해 TTFT를 기존 캐시 융합 기준선 대비 40% 추가 감소시킬 수 있다.
- **접근 방법**:
  1. `QCFuseSemanticAnchorCache` 클래스 (`src/cache/qcfuse_anchor.py`): `CacheStore` 인터페이스 상속
  2. 의미 요약 앵커 생성: 각 비연속 세그먼트에서 "의미 요약 앵커"(세그먼트 평균 임베딩)를 사전 계산 → 앵커 집합으로 쿼리 표현 강화(쿼리 임베딩 + 앵커 가중합).
  3. 중요 레이어 어텐션 분포 계산: 경량 사전 패스로 상위 K 레이어(트랜스포머 레이어 중 어텐션 엔트로피 최고)의 어텐션 분포 계산 → 쿼리 관련 토큰 상위-M개 선택.
  4. 선택적 재계산: 선택된 M개 토큰만 실제 재계산 → 나머지는 캐시된 KV 직접 재사용.
  5. 이전 SemanticSegmentCache와의 차별점: SemanticSegmentCache가 세그먼트 임베딩 유사도로 히트 여부를 결정하는 반면, QCFuseSemanticAnchorCache는 앵커 강화 쿼리 + 레이어 어텐션 분포라는 2단계 선택으로 "어느 토큰을 재계산할 것인가"를 더 정밀하게 결정한다.
  6. `QueryCentricRecomputeCache`(B-1)와의 차별점: B-1이 재계산 예산을 "쿼리-세그먼트 관련성"으로 세그먼트 단위 배분하는 반면, QCFuseSemanticAnchorCache는 "앵커 강화 쿼리 + 레이어 어텐션"으로 토큰 단위 재계산을 결정한다.
- **예상 효과**: 비연속 히트율 +15~20%p, TTFT −40% 추가 감소 (기존 캐시 융합 기준선 대비), accuracy delta ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 3

## Activity C 아이디어

### 아이디어 C-1: TriAttentionCodec — pre-RoPE 삼각함수 시리즈 기반 위치-안정적 KV 중요도 추정 + 윈도우 프루닝
- **가설**: 기존 KV 중요도 추정이 post-RoPE 쿼리를 사용해 토큰 위치에 따라 쿼리 벡터가 회전하므로 대표 쿼리 수가 제한되어 불안정한 KV 선택이 발생하는 근본 문제를, pre-RoPE 공간에서 Q와 K 벡터가 고정된 비제로 중심 주변에 집중됨을 관찰하고 이 집중이 결정하는 선호 거리를 삼각함수 시리즈로 표현해 key 중요도를 위치-안정적으로 추정하면(TriAttention 원리), 이전 사이클의 모든 C 기법(TurboQuantCodec/PolarQuant+QJL, NQKVCodec/정규분포 분위수, FireQCodec/RoPE-인식 2단계, TailorKVManager, RedundancyAwareEviction, PackKV, DMS+RL 등)이 post-RoPE 또는 레이어/어텐션 스코어 기반으로 머문 것과 달리 AIME25 32K CoT에서 10.7× KV 감소 + 전체 어텐션 동등 정확도를 달성할 수 있다.
- **접근 방법**:
  1. `TriAttentionCodec` 클래스 (`src/cache/tri_attention_codec.py`): `CompressionCodec` 인터페이스 구현
  2. pre-RoPE 중심 벡터 추정: 각 레이어·헤드별로 Q/K 벡터의 pre-RoPE 공간 비제로 중심(`mu_q`, `mu_k`)을 캘리브레이션 배치에서 추정 → 레이어별 중심 파일로 저장.
  3. 삼각함수 시리즈 중요도 추정: `importance_score(k_i) = Σ_m [a_m × cos(m × d(k_i, mu_k))]` — 중심까지의 거리를 삼각함수 급수로 표현. 계수 `{a_m}`은 캘리브레이션으로 학습.
  4. 윈도우 기반 프루닝: 생성 토큰 128개마다 프루닝 트리거. 중요도 스코어 하위 토큰을 퇴거.
  5. 압축률 파라미터: `compression_ratio` (기본 0.1 = 10× 압축, TriAttention AIME25 실증 기준).
  6. vLLM/SGLang 통합 경로: TriAttention 오픈소스(WeianMao/triattention)의 vLLM 통합 코드를 참조해 6단계 vLLM 이식 직접 활용.
  7. 이전 C 기법과의 차별점: 이전 사이클의 모든 C 기법이 post-RoPE 공간의 어텐션 스코어·중복성·주파수·분위수 기반인 반면, TriAttentionCodec은 pre-RoPE 공간의 기하학적 집중 패턴을 삼각함수로 정식화해 RoPE 회전으로 인한 불안정을 근본 해소한다.
- **예상 효과**: 메모리 −90% (10.7× KV 감소, TriAttention 실증), 처리량 +2.5× (TriAttention 실증), accuracy delta ±0% (AIME25 전체 어텐션 동등), 추론 모델(CoT) 특화 강점
- **accuracy-preserving 근거**: TriAttention(arXiv 2604.04921)이 AIME25 32K CoT에서 전체 어텐션과 동등한 추론 정확도를 유지하면서 10.7× KV 감소를 달성함을 실증. 기존 기준선들이 같은 효율 예산에서 약 50% 정확도에 그치는 것과 대비적으로 검증됨. pre-RoPE 안정성의 메커니즘: post-RoPE 쿼리는 위치마다 다르게 회전해 대표성이 제한되지만, pre-RoPE 공간의 중심 벡터는 위치-불변이므로 삼각함수 시리즈 추정이 안정적. 검증: AIME25 정확도 ±1% + LLaMA-3 WikiText-2 perplexity ±1% + LongBench 3개 서브태스크.
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 C-2: KVSculptDistillationStore — L-BFGS 기반 증류 KV 최적화로 비연속 세그먼트 요약 저장
- **가설**: 기존 KV 압축이 원본 KV 쌍을 선택하거나 결합하는 반면, 연속 임베딩 공간에서 각 레이어의 어텐션 동작을 보존하도록 Key를 L-BFGS로 최적화하고 Value를 최소자승으로 닫힌 형태로 풀면(KVSculpt 원리), 이전 사이클의 모든 C 기법(선택·결합·양자화·퇴거·저랭크)과 본질적으로 다른 "표현 가능한 공간을 넘어 최적 KV를 탐색"하는 증류 패러다임으로 KL 발산을 기존 대비 3.5~4.1× 감소시킬 수 있으며, 특히 다수의 비연속 세그먼트를 소수의 증류 KV로 요약 저장하는 "세그먼트 증류"에 적용하면 Activity B의 비연속 재사용과 Activity C의 압축이 동시에 달성된다.
- **접근 방법**:
  1. `KVSculptDistillationStore` 클래스 (`src/cache/kvsculpt_distillation.py`): `CacheStore` 인터페이스 상속
  2. 증류 KV 최적화: `distill_segments(segment_list, target_n_keys)` → L-BFGS로 `target_n_keys`개의 증류 Key 벡터를 어텐션 KL 발산 최소화 목표로 최적화.
     - Key 최적화: `minimize KL(A_orig || A_distilled)` where A = softmax(QK^T / sqrt(d))
     - Value 최적화: 최소자승 닫힌 형태 `V_distilled = (K_distilled^T K_distilled)^{-1} K_distilled^T V_orig`
  3. 적응 예산 할당: `pilot_compress(n_segments, budget)` — 소수 파일럿 압축으로 세그먼트별 압축 난이도 추정 → 어려운 세그먼트에 더 많은 증류 KV 예산 할당.
  4. 세그먼트 증류 비율: `distillation_ratio` 파라미터(기본 0.1 = 10개 세그먼트 → 1개 증류 KV로 요약). 비연속 세그먼트 N개를 N/10개 증류 KV로 압축.
  5. 복원 경로: 증류 KV로 어텐션 계산 → 원본 세그먼트 복원 불필요(어텐션 출력 직접 근사).
  6. `QueryCentricRecomputeCache`(B-1)와 연동: 쿼리 관련성 높은 세그먼트는 원본 KV 보존, 낮은 세그먼트는 증류 KV로 요약 → 예산 배분 기반 선택적 증류.
  7. 이전 C 기법과의 차별점: 이전 사이클의 모든 C 기법이 "원본 KV를 선택·조합·양자화"하는 반면, KVSculptDistillationStore는 "원본 KV와 무관한 공간에서 최적 KV를 직접 탐색"한다. 특히 "단일 원본 토큰이 표현할 수 없는 위치에 있는 증류 키"라는 개념이 완전히 새롭다.
- **예상 효과**: 메모리 −85~90% (세그먼트 증류 N→N/10 + 각 증류 KV FP16), KL 발산 −72~76% (기존 선택+최소자승 대비 3.5~4.1× 감소), accuracy delta ±0.5% 이내
- **accuracy-preserving 근거**: KVSculpt(arXiv 2603.27819)가 Qwen2.5-1.5B-Instruct 2048토큰 컨텍스트에서 어텐션 스코어 퇴거+최소자승 대비 KL 발산 3.5~4.1× 감소를 실증. 적응 예산 할당이 추론 비용 없이 1.3× KL 추가 감소. L-BFGS 최적화가 어텐션 행동 보존을 직접 목적 함수로 설정하므로 accuracy-preserving의 이론적 보장이 가장 직접적. 검증: WikiText-2 perplexity ±1% + LongBench QA/요약 + 증류 전후 어텐션 KL 발산 직접 측정.
- **구현 난이도**: high
- **우선순위**: 2

### 아이디어 C-3: AdaptiveKVCompressibility — KV-CoRE 기반 K/V 비대칭 + 언어/도메인 적응형 압축 비율 선택
- **가설**: Key가 Value보다 일관되게 저랭크 압축 가능성이 높고, 교차-언어 변동이 교차-도메인 변동보다 크다는 체계적 증거(KV-CoRE 벤치마크)를 바탕으로, Key와 Value에 비대칭 압축 비율을 적용하고 입력 언어/도메인을 감지해 동적으로 압축 비율을 선택하면, 이전 사이클의 모든 C 기법이 K/V에 동일 압축률을 적용하거나 레이어별로만 차별화한 것과 달리 "K>V 비대칭 + 언어/도메인 적응"이라는 두 차원의 최적화로 동일 정확도 예산에서 더 높은 압축률을 달성할 수 있다.
- **접근 방법**:
  1. `AdaptiveKVCompressibilityCodec` 클래스 (`src/cache/adaptive_kv_compressibility.py`): `CompressionCodec` 인터페이스 구현
  2. K/V 비대칭 압축: Key에 더 높은 압축률(기본 8비트→4비트 또는 저랭크 더 낮은 rank), Value에 낮은 압축률(8비트 유지 또는 더 높은 rank) → KV-CoRE 실증 기반.
  3. 언어 감지기: 경량 언어 식별(첫 32 토큰 N-gram 또는 langdetect 라이브러리) → 언어별 압축 난이도 프로파일(`lang_compressibility_profile.yaml`)에서 압축 비율 조회.
  4. 도메인 감지기: 요청 헤더 또는 토큰 분류기로 도메인 분류 (코드/수학/자연어/다국어) → 도메인별 압축 프로파일 조회.
  5. 적응형 압축 비율: `compress_ratio_k = profile[lang][domain]['key_ratio']`, `compress_ratio_v = profile[lang][domain]['value_ratio']`. 프로파일 초기값은 KV-CoRE 논문 Figure 3/4 참고.
  6. 온라인 프로파일 갱신: 실제 perplexity 모니터링으로 압축 비율이 ±1% 이내를 위반하면 해당 언어/도메인 압축률 자동 하향 조정.
  7. 기존 `TriAttentionCodec`(C-1)와 직교 결합: TriAttention으로 중요도 기반 토큰 선택 후, 선택된 KV에 비대칭 압축 비율 적용 → 이중 최적화.
- **예상 효과**: 메모리 −40~60% (K/V 비대칭 + 언어별 최적 압축), accuracy delta ±0.8% 이내 (적응형 비율로 언어별 한계 준수), 다국어 서비스에서 기존 균일 압축 대비 정확도 +5~15%p
- **accuracy-preserving 근거**: KV-CoRE(arXiv 2602.05929)가 Key가 Value보다 일관되게 압축 가능성 높음을 5개 영어 도메인 + 16개 언어에서 체계적으로 실증. K/V 비대칭 처리가 정확도 손실을 줄이는 이유: Value는 어텐션 가중합의 직접 입력이므로 Key보다 정밀도 요구가 높음. 언어별 적응형 비율이 과압축 위험 방지. 온라인 perplexity 모니터링이 ±1% 위반 즉시 감지·보정. 검증: KV-CoRE 벤치마크 5개 도메인 + 한국어/일본어/중국어 포함 다국어 WikiText-2 perplexity ±1%.
- **구현 난이도**: low
- **우선순위**: 3

### 아이디어 C-4: HybridHeadTypeCodec — 헤드 유형별 차별 압축 (정적 헤드 텍스트-사전 프루닝 + 동적 헤드 청크-단위 검색)
- **가설**: 멀티모달 및 장문 컨텍스트 LLM에서 어텐션 헤드를 텍스트-중심 어텐션(text-centric attention)으로 정적(static) 헤드와 동적(dynamic) 헤드로 분류하고, 정적 헤드에는 텍스트-사전 프루닝(text-prior pruning)으로 고압축, 동적 헤드에는 청크-단위 검색(chunk-wise retrieval)으로 정밀 재사용을 적용하면(HybridKV 원리), 이전 사이클의 모든 C 기법이 헤드 유형 구분 없이 균일한 압축·퇴거 정책을 적용한 것과 달리 KV 캐시 메모리 최대 7.9× 감소 + 동등 정확도를 달성할 수 있다.
- **접근 방법**:
  1. `HybridHeadTypeCodec` 클래스 (`src/cache/hybrid_head_type.py`): `CompressionCodec` 인터페이스 구현
  2. 헤드 유형 분류: 소수 캘리브레이션 배치에서 각 헤드의 어텐션 분포 엔트로피 측정. 낮은 엔트로피(= 소수 텍스트 토큰에 집중) → 정적 헤드, 높은 엔트로피(= 분산된 어텐션) → 동적 헤드.
  3. 정적 헤드 처리: 텍스트-사전 기반 토큰 중요도(텍스트 위치 + 레이어별 사전 어텐션 패턴)로 상위-K 토큰만 보존 → 고압축(나머지 퇴거).
  4. 동적 헤드 처리: ContiguousChunkIOScheduler(A-2)의 청크 단위와 정렬된 청크-단위 검색(chunk-wise retrieval) → Activity B의 비연속 세그먼트 재사용과 직접 통합.
  5. 계층적 예산 할당: K/V에 독립 예산 설정. `budget_key_static`, `budget_key_dynamic`, `budget_value_static`, `budget_value_dynamic` 4개 파라미터.
  6. `SegmentedHashCache` + `QueryCentricRecomputeCache`(B-1) 연동: 동적 헤드의 청크-단위 검색 결과를 B-1의 쿼리 관련성 필터에 통과시켜 재계산 예산 배분.
  7. 이전 C 기법과의 차별점: 이전 사이클의 FireQCodec(RoPE-인식 채널 쌍 기반), NQKVCodec(정규분포 분위수 기반), TurboQuantCodec(PolarQuant+QJL 전체 헤드 동일 처리)과 달리 헤드 유형에 따라 압축 전략 자체가 다르다.
- **예상 효과**: 메모리 −87% (7.9× 감소, HybridKV 실증), 처리량 디코딩 +1.52× (HybridKV 실증), accuracy delta ±0.5% 이내
- **accuracy-preserving 근거**: HybridKV(arXiv 2604.05887)가 KV 캐시 메모리 최대 7.9× 감소, 디코딩 1.52× 가속, 전체 캐시 MLLM과 동등하거나 우수한 성능을 실증. 정적 헤드는 이미 소수 토큰에 집중하므로 나머지 토큰 퇴거의 정확도 영향 최소. 동적 헤드는 청크-단위 정밀 검색으로 원본 KV 재사용 → 압축 오류 없음. 두 전략의 상보적 설계가 헤드 유형별 정확도 요구를 충족. 검증: HybridKV 원논문 멀티모달 벤치마크 + 텍스트 전용 LongBench + WikiText-2 perplexity ±1%.
- **구현 난이도**: medium
- **우선순위**: 4

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: B+C — QueryCentricRecomputeCache + TriAttentionCodec (쿼리 중심 선택 + pre-RoPE 안정적 압축 이중 필터링)
- **조합 근거**: ProphetKV·QCFuse·InfoFlowKV의 쿼리 중심 재계산 선택(B-1)에 TriAttention의 pre-RoPE 안정적 중요도 추정(C-1)을 결합하면, "쿼리 관련성(ProphetKV 기반)" × "pre-RoPE 위치-안정적 중요도(TriAttention 기반)"의 이중 필터링으로 더 정밀한 비연속 세그먼트 선택이 가능하다. 트렌드 리포트의 B+C 시너지 포인트를 직접 구현한다.
- **구체 설계**:
  1. `DualFilterSegmentSelector` 모듈 (`src/cache/dual_filter_selector.py`)
  2. 1단계 — 쿼리 관련성 필터: `QueryCentricRecomputeCache`의 쿼리-세그먼트 유사도 스코어로 후보 세그먼트 상위 40% 선택
  3. 2단계 — pre-RoPE 중요도 필터: `TriAttentionCodec`의 삼각함수 시리즈 중요도 스코어로 선택된 후보 내에서 토큰 단위 중요도 정렬 → 상위 20% 토큰만 보존 (전체의 20% 재계산 예산 내에서)
  4. 결합: 쿼리 관련성이 낮은 세그먼트는 TriAttention으로 고압축, 쿼리 관련성이 높은 세그먼트는 선택적 재계산 후 원본 품질 보존
  5. 재계산 예산: 전체 토큰의 최대 20% (ProphetKV 실증 상한)
- **예상 복합 효과**: 처리량 +25~35%, 메모리 −85~90%, 비연속 히트율 +25~35%p, accuracy delta ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 Cross-2: B+C — InfoFlowChunkReorderCache + KVSculptDistillationStore (최적 순서 재배치 + 저장 효율 극대화)
- **조합 근거**: InfoFlowChunkReorderCache(B-2)가 비연속 세그먼트를 정보 흐름 기반 최적 순서로 재배치하고, KVSculptDistillationStore(C-2)가 재배치된 세그먼트를 소수의 증류 KV로 요약 저장한다. 정보 흐름 기반 최적 순서가 확정된 후 증류 최적화를 적용하면, 임의 순서에서 증류하는 것보다 어텐션 KL 발산이 낮은 증류 KV를 생성할 수 있다(순서가 어텐션 행동에 영향을 주기 때문).
- **구체 설계**:
  1. 재순서화 먼저: `InfoFlowChunkReorderCache.reorder(segments, query)` → 최적 순서 세그먼트 목록 반환
  2. 증류 적용: `KVSculptDistillationStore.distill_segments(reordered_segments, target_n_keys)` → 최적 순서에서 L-BFGS 증류
  3. 저장: 증류 KV를 최적 순서 위치 메타데이터와 함께 저장 → 재사용 시 위치 재할당 없이 즉시 활용
  4. 정확도 개선 원리: 최적 순서에서의 어텐션 행동을 보존하도록 증류하므로, 임의 순서 대비 KL 발산 추가 감소 예상
- **예상 복합 효과**: 처리량 +20~30%, 메모리 −88~92%, 비연속 히트율 +18~25%p, accuracy delta ±0.5% 이내
- **구현 난이도**: high
- **우선순위**: 3

### 아이디어 Cross-3: A+B+C — ContiguousChunkIOScheduler + QueryCentricRecomputeCache + TriAttentionCodec (시스템-레벨 I/O 정렬 + 쿼리 중심 선택 + pre-RoPE 압축 완전 통합)
- **조합 근거**: 트렌드 리포트의 A+B+C 시너지 포인트를 직접 구현한다. ContiguousChunkIOScheduler(A-2)가 I/O 블록 경계 정렬로 Re-Prefill 단계를 3.85× 가속하고, QueryCentricRecomputeCache(B-1)가 쿼리 관련 20% 토큰만 재계산하며, TriAttentionCodec(C-1)이 나머지 80%의 캐시된 KV를 pre-RoPE 기반으로 10.7× 압축한다. 세 기법이 파이프라인으로 결합되면 "시스템 I/O 효율 + 재계산 예산 최적 배분 + 최고 압축률"이 동시에 달성된다.
- **구체 설계**:
  1. 요청 수신 → ContiguousChunkIOScheduler가 청크 경계 정렬 + 레이어 간 비동기 프리패칭으로 캐시된 KV 로드
  2. 쿼리 도착 → QueryCentricRecomputeCache가 쿼리 관련성 스코어 계산 → 상위 20% 세그먼트 재계산 후보 표시
  3. TriAttentionCodec이 재계산 후보가 아닌 세그먼트(80%)에 pre-RoPE 삼각함수 중요도 추정 적용 → 중요도 하위 토큰 퇴거 (10.7× 압축 목표)
  4. ContiguousChunk 단위 I/O와 TriAttentionCodec 128 토큰 프루닝 윈도우 경계 정렬: 프루닝 윈도우 = ContiguousChunk 크기의 배수 → I/O-프루닝 정렬 유지
  5. 멀티노드: ContiguousChunk 단위 RDMA 전송 → 수신 노드에서 즉시 TriAttentionCodec 중요도 스코어 적용
- **예상 복합 효과**: 처리량 +45~60%, 메모리 −88~92%, 비연속 히트율 +30~40%p (전체 히트의 60%+ 목표), Re-Prefill 단계 3.85× 가속, accuracy delta ±0.5% 이내
- **구현 난이도**: high
- **우선순위**: 2

## 이번 사이클 최우선 구현 타겟

**Cross-1(B+C): QueryCentricRecomputeCache(B-1) + TriAttentionCodec(C-1) + DualFilterSegmentSelector 통합**, 보조로 InfoFlowChunkReorderCache(B-2) 및 AdaptiveKVCompressibilityCodec(C-3) 구현

구현 순서:
1. **C-1** `TriAttentionCodec`: pre-RoPE 중심 벡터 캘리브레이션 + 삼각함수 시리즈 중요도 추정 + 128 토큰 윈도우 프루닝. `CompressionCodec` 인터페이스 drop-in. **AIME25/LongBench accuracy 및 WikiText-2 perplexity 검증 스크립트 포함**.
2. **B-1** `QueryCentricRecomputeCache`: 쿼리 임베딩 기반 세그먼트 관련성 스코어링 + 이중 단계 재계산 파이프라인 + 20% 예산 제한. `CacheStore` 인터페이스 구현.
3. **Cross-1** `DualFilterSegmentSelector`: B-1 쿼리 관련성 + C-1 pre-RoPE 중요도 이중 필터링 통합. 40% 쿼리 필터 × 20% 재계산 예산 결합 파이프라인.
4. **B-2** `InfoFlowChunkReorderCache`: 어텐션-정규 신호 기반 청크 재순서화 + RoPE 재계산 포함. `CacheStore` 인터페이스 구현.
5. **C-3** `AdaptiveKVCompressibilityCodec`: K/V 비대칭 압축 + 언어/도메인 적응형 비율 선택. `CompressionCodec` 인터페이스 drop-in. 단순 구현(low 난이도).

이유:
- TriAttentionCodec은 이전 사이클의 모든 C 기법이 도달하지 못한 10.7× KV 감소 + 전체 어텐션 동등 정확도를 실증한 신규 SOTA로, Activity C의 메모리 목표 −30%를 10× 이상 초과 달성하는 가장 강력한 단일 기법이다.
- QueryCentricRecomputeCache는 이전 사이클 B 기법들이 쿼리 관련성을 재계산 기준으로 활용하지 않은 공백을 채우며, 20% 재계산으로 96~101% 정확도 보존이라는 실증 상한이 accuracy-preserving 제약 달성을 직접 보장한다.
- DualFilterSegmentSelector(Cross-1)는 두 기법의 이중 필터링이 단독 적용보다 정밀한 세그먼트 선택을 가능하게 하며, 비연속 히트율 ≥30% 목표와 압축 정확도 ±1% 목표를 동시에 달성하는 경로를 제공한다.

## planner에게 전달할 요청사항
- **Activity C-1(TriAttentionCodec) 구현 시**: `src/cache/tri_attention_codec.py` 신규 생성. pre-RoPE 중심 벡터(`mu_q`, `mu_k`)는 캘리브레이션 배치(≥10 요청) 1회 실행 후 레이어별 파일로 저장. 삼각함수 시리즈 계수(`{a_m}`, m=1~8)도 캘리브레이션으로 학습. **accuracy-preserving 검증 필수**: `experiments/run_triattention_accuracy.py` 스크립트 작성, AIME25/LongBench 정확도 + WikiText-2 perplexity ±1% 검증. 압축률(compression_ratio) 파라미터로 0.1~0.5 범위 실험. vLLM 이식 후보로 WeianMao/triattention 오픈소스 참조.
- **Activity B-1(QueryCentricRecomputeCache) 구현 시**: `src/cache/query_centric_recompute.py` 신규 생성. 쿼리 임베딩은 쿼리 토큰의 평균 K 벡터(추가 모델 불필요). `recompute_budget_ratio` 파라미터(기본 0.20, 범위 0.10~0.30). Stage 1 관련성 스코어는 코사인 유사도(brute-force, N_segments ≤ 1K). Stage 2 레이어별 어텐션 메트릭은 누적 어텐션 스코어 재활용. 이전 사이클 SemanticSegmentCache/DHD와의 구조적 차별점(예산 배분 주체 차이) 명시적 주석 포함.
- **Activity B-2(InfoFlowChunkReorderCache) 구현 시**: `src/cache/infoflow_reorder.py` 신규 생성. 어텐션-정규 신호 계산은 `attn_score × norm(K_i)` 근사 사용. 재순서화 후 RoPE 재계산 비용 측정 필수(오버헤드 보고). 재순서화 전후 perplexity 비교 실험 포함.
- **Cross-1(DualFilterSegmentSelector) 구현 시**: `src/cache/dual_filter_selector.py` 신규 생성. B-1 관련성 스코어와 C-1 삼각함수 중요도 스코어를 독립 계산 후 결합. 1단계 40% 필터 + 2단계 20% 예산 제한 파이프라인 명시. 이중 필터 없는 단일 기법 대비 정확도 비교 실험 포함.
- **Activity C-3(AdaptiveKVCompressibilityCodec) 구현 시**: `src/cache/adaptive_kv_compressibility.py` 신규 생성. K/V 비대칭 압축 비율을 `configs/kv_compressibility_profile.yaml`로 외부화. 언어 감지는 langdetect 라이브러리 또는 경량 N-gram 기반. 온라인 perplexity 모니터링 훅 포함.
- **측정 지표 (이번 사이클 필수)**: (1) **pre-RoPE 중요도 추정 정확도**: TriAttentionCodec의 압축률별 AIME25/LongBench 정확도 + WikiText-2 perplexity 직접 측정, (2) **쿼리 중심 재계산 효율**: 재계산 비율(실제 재계산 토큰 / 전체 토큰) + 정확도 보존율 (96~101% 목표), (3) **비연속 히트율**: 쿼리 관련성 히트 vs 해시 히트 vs 재계산 분리 측정, (4) **청크 재순서화 효과**: 재순서화 전후 perplexity 변화 + 재순서화 오버헤드, (5) **처리량**: tokens/sec 베이스라인 대비 (목표 +20%).
- **테스트 필수 항목**: `tests/unit/test_tri_attention_accuracy.py` (pre-RoPE 압축 정확도 + perplexity ±1% 포함), `tests/unit/test_query_centric_recompute.py` (쿼리 관련성 스코어 + 재계산 예산 제한 검증), `tests/unit/test_infoflow_reorder.py` (재순서화 정확도 + RoPE 재계산 검증), `tests/integration/test_cross_bc_dual_filter.py` (이중 필터 통합 테스트)
