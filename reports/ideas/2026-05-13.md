## Activity A 아이디어

### 아이디어 A-1: PBKVAgentSegmentPreservationScheduler
- **가설**: 에이전틱 워크플로우에서 미래 에이전트 호출 시퀀스를 다단계 예측해 높은 재사용 잠재력을 가진 KV 세그먼트를 GPU 메모리에 선점 보존하면, 동적 워크플로우 환경에서도 비연속 KV 세그먼트 히트율과 처리량을 동시에 향상시킬 수 있다.
- **접근 방법**: PBKV(2605.06472) 방식의 예측기를 Activity B의 세그먼트-레벨로 재구성한다. (1) **세그먼트-레벨 PBKV 예측기**: 과거 에이전트 호출 이력과 현재 컨텍스트를 입력으로 받아 미래 N 스텝에서 어떤 비연속 KV 세그먼트가 재사용될 확률이 높은지 추정한다(경량 MLP, 입력: 세그먼트 임베딩 + 호출 이력 벡터). (2) **위계적 보존 정책**: 재사용 잠재력이 높은 세그먼트를 GPU Radix 트리(기존 `src/cache/radix.py`)에 보존하고, 낮은 세그먼트를 호스트 메모리(HiCache 스타일)로 퇴거한다. (3) **Lipschitz-연속 강건성 보장**: 예측기 오차가 크더라도 최악 성능이 예측 오차에 연속적으로 의존하도록 `preemption_margin` 파라미터로 보수적 프리패칭을 유지한다. (4) `src/scheduler/pbkv_agent_segment_scheduler.py`로 구현, `BaseScheduler` 상속.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드에서는 PBKV의 2계층 스토리지를 노드 간으로 확장해 활성 워크플로우 세그먼트는 로컬 GPU에, 비활성 세그먼트는 원격 HiCache에 배치한다. SGLang Q2 2026 에이전틱 분산 KVCache 로드맵과 인터페이스를 선행 정렬한다.
- **예상 효과**: 처리량 +25~40% (에이전틱 워크로드 레이턴시 1.39× 단축 기반), 비연속 세그먼트 히트율 +15~25%, Scheduling overhead TTFT +4% 이내
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 A-2: PBKVAdaptiveBitwidthAgentKVStorage
- **가설**: PBKV 예측기가 재사용 확률이 낮다고 판단한 에이전트 KV 세그먼트를 INT4로 압축해 GPU 메모리에 저비용으로 보존하고, 재사용 확률이 높은 세그먼트는 FP16으로 유지하면, 동일 GPU 메모리 예산에서 더 많은 세그먼트를 보존하여 캐시 히트율을 높일 수 있다.
- **접근 방법**: (1) PBKVAgentSegmentPreservationScheduler(A-1)의 재사용 확률 추정 결과를 비트폭 결정 입력으로 사용한다. (2) 재사용 확률 > `high_threshold`(0.7): FP16 보존, `medium_threshold`(0.3~0.7): INT8 압축, < `low_threshold`(0.3): INT4 압축 또는 호스트 메모리 오프로드. (3) 비트폭 전환은 `When Quantization Is Free(2605.05699)`의 SRFT Gaussianization 원칙을 적용해 양자화 오류를 최소화한다. (4) `src/scheduler/pbkv_adaptive_bitwidth_storage.py`로 구현.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 노드 간 전송 시 높은 재사용 확률 세그먼트만 FP16으로 전송하고, 낮은 확률 세그먼트는 INT4로 전송해 네트워크 대역폭을 절감한다.
- **예상 효과**: GPU 메모리 절감으로 보존 세그먼트 수 2.5~3×, 처리량 +15~20%, Scheduling overhead +3% 이내
- **구현 난이도**: medium
- **우선순위**: 2

## Activity B 아이디어

### 아이디어 B-1: KVFoldAccumulativeRadixCache
- **가설**: KV 캐시를 함수형 프로그래밍의 `foldl` 누산기로 취급하여 순차적으로 처리하는 청크들의 KV를 누산 상태로 유지하면, 기존 RadixAttention의 prefix-match 방식과 상보적으로 "순서 의존적 누산 재사용"을 지원하는 완전히 새로운 비연속 재사용 패러다임을 `src/cache/` 레이어에 훈련-프리로 통합할 수 있다.
- **접근 방법**: KV-Fold(2605.12471)의 foldl 프로토콜을 `src/cache/radix.py`의 확장으로 구현한다. (1) **청크 누산기**: 각 청크 처리 시 기존 누산된 KV 캐시를 접두사(prefix)로 참조하며 새 청크를 처리하고, 생성된 K·V를 누산 캐시에 추가한다. (2) **안정성 플래토 감지**: KV-Fold의 핵심 발견 — 스텝당 드리프트가 플래토에 수렴 — 을 `drift_threshold` 파라미터로 모니터링하여 수렴 후 추가 처리를 조기 종료한다. (3) **StreamingLLM 하이브리드**: 슬라이딩 윈도우 방식(StreamingLLM)이 메모리를 효율적으로 사용하지만 장거리 정보를 손실하는 반면, KVFoldAccumulativeRadixCache는 foldl 누산으로 장거리 정보를 완전 보존한다. 두 방식을 메모리 예산 기반으로 동적으로 선택한다. (4) `src/cache/kv_fold_accumulative.py`로 구현, `CacheStore` 인터페이스 상속.
- **예상 효과**: 비연속 히트율 +20~35% (순서 의존 청크 재사용 추가), TTFT 기존 RadixAttention 대비 −15~25% (prefix 불일치 요청에서 누산 재사용), 128K 컨텍스트 NIAH 정확도 100% (단일 40GB GPU 기준)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 B-2: KVMGatedSegmentCompressor
- **가설**: KVM(2605.09877)의 학습 가능한 게이팅 메커니즘을 비연속 KV 세그먼트 선택기로 적용하면, 어떤 세그먼트를 압축 상태에 통합할지를 학습 신호로 결정하는 "KVM-guided 비연속 세그먼트 선택적 통합기"가 가능하며, 기존 휴리스틱 기반 세그먼트 선택 대비 정확도를 보존하면서 더 많은 세그먼트를 압축 캐시에 유지할 수 있다.
- **접근 방법**: (1) 레이어당 학습 가능한 게이트 `G_l ∈ R^{d_k}`를 유지한다. (2) 새 비연속 세그먼트가 들어올 때 게이트와의 내적으로 통합 여부를 결정한다: `integrate_score = sigmoid(G_l · segment_embedding)`. (3) 통합 결정된 세그먼트만 공유 압축 KV 상태 `h_l`에 게이팅 업데이트: `h_l ← α·h_l + (1-α)·KV_segment`. (4) 고정-크기(bounded) 변형만 구현해 메모리가 세그먼트 수에 무관하게 고정되게 한다. (5) `src/cache/kvm_gated_segment.py`로 구현, 표준 연산만 사용(커스텀 CUDA 불필요). (6) 파인튜닝 가능 환경 전제 — 게이트를 소규모 캘리브레이션 셋에서 경량 파인튜닝(1K 스텝)으로 학습.
- **예상 효과**: 비연속 히트율 +10~20%, 메모리 −40~60% (고정-크기 KV 상태), accuracy delta ±0.7%
- **구현 난이도**: high
- **우선순위**: 3

## Activity C 아이디어

### 아이디어 C-1: ShadowMaskDistillationKVCompressor
- **가설**: KV 캐시 압축을 RL 후훈련(GRPO/PPO) 롤아웃 단계에 통합할 때 발생하는 오프-폴리시 편향을 Shadow Mask Distillation(2605.06850)의 이중 트랙 프레임워크로 구조적으로 제거하면, 기존 압축 기법(MixedDimKV, RSimVQ 등)을 RL 학습 루프에서 정확도 손실 없이 운용할 수 있으며 "학습 루프 안정성"이라는 새로운 accuracy-preserving 차원을 달성할 수 있다.
- **접근 방법**: (1) **Shadow mask 트랙**: 압축 KV로 롤아웃을 생성하되, 동일 패스에서 shadow mask로 전체 KV 연산도 병렬 실행해 정책 그래디언트를 완전 KV 기반 로그확률로 보정한다. (2) **Distillation 트랙**: 압축 모델이 완전 모델의 소프트 타겟을 모방하도록 KL divergence 손실을 추가한다. (3) **프레임워크-독립적 구현**: 고수준 텐서 슬라이싱의 Not-In-Place Allocation Spike를 우회하는 메모리 관리 패턴을 적용한다. (4) `src/cache/smd_kv_compressor.py`로 구현. 기존 압축 코덱(MixedDimPerTokenBudgetCodec, RSimVQCodec)을 inner compressor로 받아 SMD 래퍼를 씌우는 플러그인 구조. (5) vLLM GRPO/PPO 통합 인터페이스 추가: `vllm_integration/smd_rl_patch.py`.
- **예상 효과**: RL 후훈련 메모리 −30~50% (롤아웃 KV 압축), 정책 편향 제거 → RL 학습 안정성 유지, accuracy delta 추론 시 ±0.5% + RL 학습 곡선 Dense 베이스라인 동등 또는 초과
- **accuracy-preserving 근거**: SMD(2605.06850)가 shadow mask로 완전 KV 기반 그래디언트를 복원해 완벽한 그래디언트 정렬(perfect gradient alignment)을 달성한다. Distillation 트랙이 압축 모델 출력을 완전 모델에 근사시켜 추론 시 정확도를 보존한다. 1B~7B 모델에서 MATH500·AIME24·HotpotQA·TL;DR 전 설정에서 압축 Dense 베이스라인 동등 또는 초과가 실증됐다. HotpotQA·TL;DR에서는 Dense 베이스라인을 초과하는 정보 병목 효과가 관측된다.
- **구현 난이도**: high
- **우선순위**: 2

### 아이디어 C-2: SRFTFusedINT4KVKernel
- **가설**: SRFT(Subsampled Randomized Fourier Transform) Gaussianization과 INT4 nibble 패킹을 단일 퓨즈드 커널로 결합하면, KV 캐시의 아웃라이어 채널을 Gaussianize하면서 INT4 양자화를 적용해 메모리를 3× 절감하면서도 FP16 대비 추론 속도를 유지하거나 향상시킬 수 있다(CUDA 백엔드 일반화).
- **접근 방법**: `When Quantization Is Free(2605.05699)`의 SRFT + INT4 퓨즈드 커널 설계를 CUDA 백엔드(서버 GPU)에서 일반화한다. (1) **SRFT 변환**: 각 KV 채널에 부호 무작위화 + FFT를 적용해 아웃라이어 채널을 Gaussianize한다(오버헤드 ~25ns/vec 목표). (2) **채널별 스케일링(λ)**: 각 채널의 스케일 팩터를 FP16 사이드카로 저장한다. (3) **그룹별 abs-max 양자화**: 그룹 크기 G=128으로 INT4 양자화를 적용한다. (4) **INT4 nibble 패킹**: 2개 INT4 값을 1바이트에 패킹해 메모리를 4×(이론) / 3×(실효) 절감한다. (5) `src/cache/srft_int4_kv_kernel.py`로 구현, `HuggingFace Cache` 서브클래스로 노출. RateQuantReverseWaterfillingCodec(이전 사이클 기구현)의 헤드별 최적 비트폭을 입력으로 받아 동적 비트폭 SRFT 커널로 확장한다.
- **예상 효과**: 메모리 −60~67% (3~4× 압축), 추론 속도 FP16 대비 0~+10% (SRFT 오버헤드가 대역폭 절감으로 상쇄), accuracy delta ±0.5% 이내
- **accuracy-preserving 근거**: SRFT의 Gaussianization이 아웃라이어 채널을 정규 분포로 변환해 INT4 양자화 오류를 최소화한다. 원논문(2605.05699)에서 Gemma-3 1B 기준 256~4096 토큰 전 구간에서 INT4가 FP16보다 빠르면서 3× 메모리 압축 + 품질 유지가 실증됐다. SRFT와 SRHT가 통계적으로 동등한 Gaussianization 효과를 제공해 CUDA 환경에서도 동일한 양자화 오류 특성이 기대된다.
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 C-3: MELTGatedSharedKVCompressor
- **가설**: MELT(2605.07721)의 루프드 트랜스포머 KV 메모리 고정 원칙 — 레이어당 학습 가능한 게이팅으로 단일 공유 KV 캐시를 시간적으로 갱신 — 을 표준 트랜스포머의 장문 추론 체인(chain-of-thought)에 적용하면, 추론 깊이(루프 수)가 증가해도 KV 메모리가 고정되는 "계산-메모리 분리" 압축을 달성할 수 있다.
- **접근 방법**: (1) 각 레이어 `l`에 학습 가능한 게이트 `G_l ∈ R^{d_model}`을 추가한다. (2) 루프 반복마다 게이트로 현재 KV와 공유 KV 상태를 혼합: `KV_shared_l ← gate(G_l) * KV_current + (1 - gate(G_l)) * KV_shared_l`. (3) 공유 KV 상태만 메모리에 유지하고 개별 루프의 KV는 즉시 해제한다 — 루프 수에 무관하게 레이어당 O(L×d)로 메모리를 고정한다. (4) 파인튜닝: 소규모 캘리브레이션 데이터에서 게이트만 학습(1K~2K 스텝). (5) `src/cache/melt_gated_shared_kv.py`로 구현. Chain-of-Thought 추론 파이프라인(`src/engine/runner.py`)과 통합.
- **예상 효과**: 장문 CoT 추론 시 KV 메모리 −70~80% (루프 누적 대비), 추론 벤치마크 성능 표준 트랜스포머 동등 이상, accuracy delta ±0.8% 이내
- **accuracy-preserving 근거**: MELT(2605.07721)가 루프드 트랜스포머 설정에서 비슷한 크기의 표준 트랜스포머를 초과하는 추론 벤치마크 성능을 달성했다. 학습 가능한 게이팅이 중요 정보를 공유 KV 상태에 선택적으로 통합하는 역할을 하며, 분리된 계산 깊이가 표현력을 유지한다. 게이트만 파인튜닝하므로 모델 본체 파라미터는 고정되어 기반 정확도가 보존된다.
- **구현 난이도**: high
- **우선순위**: 3

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: PBKVKVFoldAgentChunkPreCaching (A+B)
- **조합 근거**: PBKVAgentSegmentPreservationScheduler(A-1)가 미래 에이전트 호출에서 필요할 청크를 예측하고, KVFoldAccumulativeRadixCache(B-1)가 그 청크들을 foldl 방식으로 사전에 누산 상태에 통합해 둔다. 에이전트가 실제로 해당 청크를 요청하면 이미 누산된 KV 상태에서 즉시 서빙할 수 있어 TTFT를 대폭 단축한다. "예측으로 무엇을 보존할지 결정 + foldl 누산으로 어떤 형태로 보존할지 결정"하는 이중 계층 에이전틱 KV 관리다.
- **구체 설계**:
  1. `AgenticChunkPreCachingPipeline` 모듈 (`src/cache/agentic_chunk_precaching.py`)
  2. PBKV 예측기가 미래 N 스텝 내 재사용 확률 높은 청크 세트 S를 결정
  3. KVFoldAccumulativeRadixCache가 S의 청크들을 foldl 순서로 사전 누산 처리
  4. 누산된 KV 상태를 GPU Radix 트리 리프 노드에 "pre-folded prefix"로 저장
  5. 에이전트 요청 시 pre-folded prefix에서 추가 청크만 incremental foldl 처리
  6. 예측 미스 시 fallback: 일반 RadixAttention으로 전환
- **예상 복합 효과**: 처리량 +35~50%, TTFT −50~65% (pre-cached foldl 상태에서 서빙), 비연속 히트율 +25~40%
- **구현 난이도**: high
- **우선순위**: 1

### 아이디어 Cross-2: KVFoldSRFTINT4CompressedAccumulator (B+C)
- **조합 근거**: KVFoldAccumulativeRadixCache(B-1)의 누산 KV 상태는 점점 커지는 경향이 있다. SRFTFusedINT4KVKernel(C-2)로 누산 상태를 INT4로 압축 유지하면, foldl 누산의 정보 보존 특성을 유지하면서 메모리 사용량을 3× 절감할 수 있다. 특히 누산 상태의 오래된 청크는 SRFT + INT4로 압축하고, 최신 청크는 FP16으로 유지하는 시간적 혼합 정밀도 전략이 가능하다.
- **구체 설계**:
  1. `CompressedFoldAccumulator` 모듈 (`src/cache/compressed_fold_accumulator.py`)
  2. 누산 상태를 시간 기준으로 구분: 최신 `window` 청크(FP16 보존), 오래된 청크(SRFT+INT4 압축)
  3. foldl 추가 시 신규 청크를 FP16으로 누산 → `window`를 초과하면 SRFT+INT4로 압축 강등
  4. 복원 시 INT4 디코딩 + SRFT 역변환으로 FP16 복원 후 어텐션 계산
  5. 압축 오버헤드가 25ns/vec 이내로 foldl 처리 속도에 영향 최소화
- **예상 복합 효과**: 처리량 +25~40%, 메모리 −50~65% (foldl 누산 + INT4 압축), 비연속 히트율 +20~35%, accuracy delta ±0.7% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 Cross-3: SMDRLTrainingWithNonContiguousKVReuse (B+C)
- **조합 근거**: ShadowMaskDistillationKVCompressor(C-1)를 KVFoldAccumulativeRadixCache(B-1) 또는 기존 비연속 KV 재사용 기법과 함께 RL 파인튜닝에 적용하면, 비연속 세그먼트 압축을 포함하는 RL 정렬 루프에서 오프-폴리시 편향 없이 학습하는 "비연속 KV 재사용 인식 RL 파인튜닝"이 가능하다. 비연속 세그먼트 재사용이 도입하는 추가 KV 근사 오차를 SMD의 shadow mask가 보정해 RL 학습을 안정화한다.
- **구체 설계**:
  1. `NonContiguousRLFinetuner` 모듈 (`src/engine/nc_rl_finetuner.py`)
  2. 롤아웃 단계: KVFoldAccumulativeRadixCache로 비연속 세그먼트를 재사용하며 압축 KV로 추론 실행
  3. Shadow mask: 동일 패스에서 완전 KV 연산도 병렬 실행 → 정책 그래디언트를 완전 KV 로그확률로 보정
  4. Distillation 손실: 비연속 재사용 모델 출력 vs 완전 KV 모델 출력의 KL divergence 최소화
  5. 비연속 세그먼트 압축(SRFTFusedINT4)과 결합해 롤아웃 메모리를 추가 절감
- **예상 복합 효과**: RL 훈련 시 메모리 −40~60%, 비연속 KV 재사용 정확도 RL 학습 후 +5~10%p 향상, 정책 편향 제거로 RL 학습 안정성 유지, accuracy delta ±0.5% 이내
- **구현 난이도**: high
- **우선순위**: 3

### 아이디어 Cross-4: PBKVKVFoldSMDFullPipeline (A+B+C)
- **조합 근거**: 에이전틱 장문 추론과 RL 정렬을 통합하는 완전 파이프라인: PBKV 예측 기반 에이전트 세그먼트 보존(A-1) → KV-Fold foldl 청크 누산(B-1) → SRFT+INT4 압축(C-2) → SMD RL 학습 오프-폴리시 편향 제거(C-1) → MELT 루프 KV 메모리 고정(C-3). PBKV가 어떤 청크를 보존할지 결정하고, KV-Fold가 청크를 누산하며, SRFT+INT4가 누산 상태를 압축하고, SMD가 RL 파인튜닝 중 편향을 제거하며, MELT가 장문 CoT 루프에서 메모리를 고정한다.
- **예상 복합 효과**: 처리량 +40~60%, 메모리 −60~75%, 비연속 히트율 +30~45%, accuracy delta ±0.8% 이내, RL 학습 안정성 Dense 베이스라인 동등
- **구현 난이도**: high
- **우선순위**: 4

## 이번 사이클 최우선 구현 타겟
**Cross-1 (PBKVKVFoldAgentChunkPreCaching, A+B)** 과 **C-2 (SRFTFusedINT4KVKernel)** 를 최우선 구현 타겟으로 권장한다.

이유:
1. B-1(KVFoldAccumulativeRadixCache)은 훈련-프리로 즉시 `src/cache/`에 통합 가능하고, 기존 `radix.py`를 확장하는 방식이라 구현 재사용이 높다. 128K 컨텍스트 NIAH 100% 달성이 비연속 히트율 목표(30% 이상)에 직접 기여한다.
2. A-1(PBKVAgentSegmentPreservationScheduler)은 구현 난이도 medium으로, 기존 스케줄러 인터페이스를 상속해 에이전틱 워크로드에서 레이턴시를 1.39× 단축하는 가장 실용적인 경로다.
3. C-2(SRFTFusedINT4KVKernel)는 메모리 −60~67%와 속도 유지를 동시에 달성하는 가장 즉각적인 Activity C 기여다. RateQuantReverseWaterfillingCodec(기구현)과 플러그인 방식으로 결합 가능해 구현 비용이 낮다.
4. Cross-1은 A-1과 B-1의 조합으로 구현 재사용률이 높으며, 처리량 +35~50%와 비연속 히트율 +25~40%를 동시에 달성한다.
5. C-1(SMD)과 C-3(MELT)은 파인튜닝 또는 아키텍처 변경이 필요하므로 이후 사이클에서 순차 구현 권장.

## planner에게 전달할 요청사항
- **B-1 구현 시**: `src/cache/kv_fold_accumulative.py` 신규 생성. foldl 누산기 인터페이스: `fold_chunk(chunk_tokens, accumulated_kv)` → 새 누산 KV 반환. 드리프트 플래토 감지: `drift_threshold=1e-3` 파라미터 YAML 외부화. 기존 `src/cache/radix.py`와 통합 경로 명시: RadixAttention의 prefix 매칭 실패 시 KV-Fold 누산 경로로 fallback. **accuracy-preserving 검증 필수**: NIAH 16K/64K/128K 컨텍스트에서 검색 정확도 + WikiText-2 perplexity ±1% 이내.
- **A-1 구현 시**: `src/scheduler/pbkv_agent_segment_scheduler.py` 신규 생성. 예측기 MLP: 입력 = 세그먼트 임베딩(d=256) + 최근 호출 이력(10 스텝), 출력 = 세그먼트별 재사용 확률. `preemption_margin=0.3` 파라미터로 보수적 프리패칭 유지. GPU Radix 트리 보존 용량 임계값(`gpu_preserve_threshold`) YAML 외부화. **측정 필수**: 에이전틱 워크로드 시뮬레이션(5~10 스텝 동적 워크플로우)에서 TTFT p50/p99 측정.
- **C-2 구현 시**: `src/cache/srft_int4_kv_kernel.py` 신규 생성. SRFT: `torch.fft.fft` 기반 Randomized Hadamard/Fourier Transform + 부호 무작위화. 채널별 스케일링: FP16 사이드카 저장. 그룹별 abs-max INT4 양자화: `group_size=128`. `HuggingFace Cache` 서브클래스 인터페이스. RateQuantReverseWaterfillingCodec(기구현)의 헤드별 비트폭 출력을 SRFT 커널 입력으로 연결하는 어댑터 구현. **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 압축률 vs 정확도 곡선(그룹 크기 G=64/128/256 별).
- **Cross-1 구현 시**: `src/cache/agentic_chunk_precaching.py` 신규 생성. PBKV 예측기와 KV-Fold 누산기의 인터페이스 계약 명시: 예측기는 청크 ID 세트를 반환하고, 누산기는 해당 ID 순서로 foldl 처리 후 "pre-folded prefix" 노드를 RadixCache에 등록. 예측 미스 fallback: `fallback_to_radix_attention()` 인터페이스. **처리량 vs 기준선 비교 측정 필수**: tokens/sec + 비연속 히트율.
- **목표 지표 중 이번 사이클에서 측정할 지표**: 
  - Inference Throughput (목표 +20% 이상)
  - Non-Contiguous Cache Hit Rate (목표 30% 이상)
  - KV Cache Memory Reduction (목표 −30% 이상, C-2 구현 시 −60% 이상)
  - Compression Accuracy Delta (필수: ±1% 이내)
  - TTFT p50 (목표 Scheduling overhead +5% 이내)
- **Activity C 구현 시 공통 필수 사항**: 모든 C 아이디어는 `src/metrics/` 내 perplexity 측정 코드를 포함하고, 압축 전후 perplexity를 `results/<exp-name>/metrics.json`에 함께 기록해야 한다. C-1(SMD)은 추가로 RL 학습 안정성 측정(학습 곡선 Dense 베이스라인 비교)을 포함해야 한다.
