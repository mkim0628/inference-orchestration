## 변화 요약
1. **vLLM v0.21.0 멀티-커넥터 HMA 플러그인 아키텍처 공식화**: 이전 16개 사이클 전체는 스케줄러 내부 로직(오프로딩 정책, 퇴거 결정, 라우팅)을 다뤘으나, v0.21의 멀티-커넥터 HMA는 "압축/오프로딩 정책을 독립 HMA 커넥터로 플러그인하는 공식 API"를 제공한다. 이전 사이클에서 제안된 어떤 A 기법도 이 플러그인 아키텍처(HMA Connector Interface)를 통한 스케줄러-압축 정책 통합을 설계하지 않았다.
2. **DCP/PCP 퍼-잡 완료 추적 기반 요청별 오프로딩 SLA 스케줄링**: v0.21의 DCP(per-job store completion)/PCP(per-job prefetch completion) 오프로딩 커넥터 시맨틱이 "요청별 오프로딩 완료 이벤트를 스케줄러가 직접 구독해 SLA 기반 배치 결정을 내리는" 새로운 설계 차원을 열었다. 이전 A 기법들(PreemptiveKVOffloadScheduler, NAtHDDROffloadingScheduler 등)은 오프로딩 완료를 비동기 스트림으로 처리했으나, DCP/PCP 이벤트 기반 스케줄러 피드백 루프는 완전히 새로운 메커니즘이다.
3. **vLLM Q2 2026 온라인 양자화 리팩터 — production/RL 워크로드용 재설계 방향**: INT8 동적 퍼-토큰 KV 양자화 온라인 리팩터가 "생산/RL 워크로드용으로 더 유연·메모리효율적"으로 재설계되는 중이라는 공식 로드맵 발표는, Activity C의 "RL 추론 워크로드에서 온라인 정밀도 적응 양자화"라는 새로운 실용 방향을 제시한다. 이전 16개 사이클의 C 기법들(RateQuant, FibQuantVQ, GlobalRetentionGate 등)은 모두 정적 캘리브레이션 기반이거나 오프라인 파인튜닝을 전제했다.

## Activity A 아이디어

### 아이디어 A-1: HMAMultiConnectorCompressionPluginScheduler (HMA 멀티-커넥터 압축 정책 플러그인 스케줄러)
- **가설**: vLLM v0.21.0의 멀티-커넥터 HMA 아키텍처를 활용하면, Activity C의 각 압축 기법(GlobalRetentionGateEvictionCodec, SPKVWriteTimeCodec, RateQuantReverseWaterfillingCodec 등)을 독립 HMA 커넥터로 등록하고 스케줄러가 요청 특성(컨텍스트 길이, SLA 티어, 모달리티)에 따라 런타임에 적절한 압축 커넥터를 선택·조합하는 "압축 정책 동적 디스패치 스케줄러"가 가능하다. 이는 단일 스케줄러가 여러 압축 전략을 요청별로 선택하는 최초의 Activity A 설계로, 이전 모든 A 기법이 고정된 단일 오프로딩/퇴거 정책을 적용한 것과 근본적으로 다르다.
- **접근 방법**:
  1. `HMAMultiConnectorScheduler` 클래스 (`src/scheduler/hma_multi_connector_scheduler.py`): `BaseScheduler` 상속
  2. **HMA 커넥터 레지스트리**: 각 Activity C 압축 코덱을 `HMAConnectorInterface`로 래핑해 레지스트리에 등록. 레지스트리 구조: `Dict[str, HMAConnectorInterface]` — 키: 압축 기법 이름, 값: 코덱 인스턴스. 등록 API: `register_connector(name: str, codec: CompressionCodec)`.
  3. **요청 특성 기반 커넥터 선택 정책**: 스케줄러가 각 요청의 프로파일을 평가해 최적 커넥터 선택.
     - 컨텍스트 길이 > `long_ctx_threshold`(기본 4096): `GlobalRetentionGateEvictionCodec` 선택 (긴 컨텍스트에서 전역 경쟁 퇴거 우위).
     - RL/추론 워크로드 표식(`is_reasoning=True`): `NAtHDDROffloadingScheduler` 정책 + 퇴거 억제.
     - 짧은 컨텍스트 고처리량 요청: `RateQuantReverseWaterfillingCodec` 선택 (빠른 정보이론 최적 압축).
     - 멀티모달 요청: `SPKVWriteTimeCodec` (write-time 희소화로 모달리티별 유틸리티 분리).
     - 선택 로직: `select_connector(request: Request) → str` 메서드, YAML `connector_dispatch_policy` 외부화.
  4. **DCP/PCP 이벤트 연동 (vLLM v0.21.0 HMA 커넥터 시맨틱 활용)**: 각 커넥터의 저장 완료(DCP: per-job store completion) 및 프리페치 완료(PCP: per-job prefetch completion) 이벤트를 스케줄러 피드백 루프에 연결. 오프로딩이 완료되지 않은 요청은 배치 진행 보류(back-pressure) → HBM 압박 없이 안전한 배치 진행.
  5. **커넥터 조합(파이프라인 체이닝)**: `pipeline_mode=True` 시 복수 커넥터를 순차 적용. 예: SPKVWriteTimeCodec (write-time) → GlobalRetentionGateEvictionCodec (post-write). 각 커넥터 출력이 다음 커넥터 입력으로 전달되는 파이프라인 컨트랙트.
  6. **이전 A 기법과의 차별점**: 이전 16개 사이클의 모든 A 기법 — PBKVAgentSegmentPreservationSchedulerMixin, NAtHDDROffloadingScheduler, RadixFeatherBatchScheduler, PreemptiveKVOffloadScheduler 등 — 은 각각 단일 스케줄링 정책을 구현했다. HMAMultiConnectorScheduler는 "여러 압축 정책을 요청별로 동적 선택·조합하는 메타-스케줄러"라는 새로운 설계 계층을 도입한다. 특히 vLLM v0.21.0의 공식 멀티-커넥터 HMA API를 스케줄러 레이어에서 활용하는 최초의 기법이다.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드에서 HMA 커넥터 레지스트리를 노드 간 공유 컨피그로 동기화. 각 노드의 로컬 HMA 인스턴스가 동일 커넥터 집합을 등록하되, 커넥터 선택 결정은 마스터 스케줄러에서 중앙화. DCP/PCP 이벤트를 RDMA 메시지로 노드 간 전파해 클러스터-와이드 back-pressure 실현.
- **예상 효과**: 처리량 +25~40% (요청 특성에 맞는 압축 정책 선택으로 각 요청의 최적 메모리-정확도 트레이드오프 달성), Scheduling overhead TTFT +4% 이내 (커넥터 선택 오버헤드: O(1) 딕셔너리 룩업 + 요청 프로파일 평가 < 0.1ms)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 A-2: DCPPCPEventDrivenOffloadingFeedbackScheduler (DCP/PCP 이벤트 기반 오프로딩 피드백 스케줄러)
- **가설**: vLLM v0.21.0의 DCP(per-job store completion) 및 PCP(per-job prefetch completion) 오프로딩 커넥터 시맨틱을 스케줄러 피드백 루프에 통합하면, 각 요청의 KV 오프로딩 완료 상태를 정확히 추적해 "오프로딩이 완료된 요청만 배치에 포함"하는 데드라인-인식 배치 전략을 구현할 수 있으며, 오프로딩 완료 대기 시간으로 인한 HBM 오염(stale HBM occupation)을 제거해 실효 HBM 용량을 증가시킬 수 있다. 이전 모든 A 기법은 오프로딩 완료 이벤트 구독 없이 비동기 스트림이나 폴링 방식으로 오프로딩 상태를 추적했다.
- **접근 방법**:
  1. `DCPPCPFeedbackScheduler` 클래스 (`src/scheduler/dcppcp_feedback_scheduler.py`): `BaseScheduler` 상속
  2. **DCP/PCP 이벤트 구독 인터페이스**: `OffloadingEventBus` — 각 요청 ID별 DCP/PCP 완료 이벤트를 등록/구독하는 경량 이벤트 버스. 구독: `subscribe_dcp(job_id: str, callback: Callable)`, `subscribe_pcp(job_id: str, callback: Callable)`. vLLM v0.21.0 HMA OffloadingConnector의 DCP/PCP 인터페이스를 `OffloadingConnectorAdapter`로 래핑해 이벤트 버스에 연결.
  3. **배치 편성 전략**:
     - 새 배치 편성 시 각 후보 요청의 오프로딩 상태 확인: DCP 완료(`store_done=True`)인 요청 우선 배치.
     - PCP 완료(`prefetch_done=True`)인 요청은 DDR→HBM 프리페치 완료 상태이므로 즉시 HBM 접근 가능 → 높은 우선순위.
     - DCP/PCP 미완료 요청은 대기 큐에 유지, 완료 이벤트 수신 시 즉시 우선순위 큐로 이동.
     - 공정성 보장: `max_offload_wait_ms`(기본 50ms) 초과 시 미완료 요청도 강제 배치(starvation 방지).
  4. **실효 HBM 용량 계산**: 각 타임스텝에서 "DCP 완료 완료 요청의 KV 합계"를 실효 HBM 절감량으로 계산해 스케줄러 배치 크기 결정에 반영. `effective_hbm_free = sum(kv_size(req) for req in dcp_done_requests)`.
  5. **DCP 기반 세그먼트 재사용 우선순위**: DCP 완료된 요청의 KV 세그먼트가 DDR에 안전하게 저장됐음이 보장되므로, 해당 세그먼트를 비연속 재사용 후보로 즉시 인덱싱. `src/cache/segmented.py`의 `SegmentedHashCache`에 `mark_offload_complete(segment_id)` 인터페이스 추가.
  6. **이전 A 기법과의 차별점**: NAtHDDROffloadingScheduler(05-16)가 누적 어텐션 점수로 오프로딩 대상을 결정하고 torch.cuda.Stream 비동기 전송을 사용한 반면, DCPPCPFeedbackScheduler는 "오프로딩 완료 이벤트"를 배치 편성의 입력으로 사용하는 피드백 루프 구조를 도입한다. 전자가 "어떤 KV를 오프로딩할지 결정"하는 정책이라면, 후자는 "오프로딩 완료 여부가 배치 편성 결정에 실시간 영향"을 주는 제어 흐름 구조다. DCP/PCP vLLM v0.21.0 공식 시맨틱을 스케줄러 입력으로 사용하는 최초의 기법이다.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드에서 DCP/PCP 이벤트를 노드 간 경량 gRPC 메시지로 전파. 원격 노드에서 오프로딩 완료된 KV의 DCP 이벤트를 마스터 스케줄러가 수신해 클러스터-와이드 실효 HBM 용량 계산. PCP(원격 DDR → 로컬 HBM 프리페치 완료)는 RDMA 전송 완료 이벤트와 매핑.
- **예상 효과**: 처리량 +15~25% (오프로딩 완료 확인 후 배치로 HBM 오염 제거 + 실효 HBM 용량 증가), HBM 활용률 +20~30% (stale HBM occupation 제거), Scheduling overhead TTFT +3% 이내 (이벤트 버스 오버헤드 < 0.05ms/요청)
- **구현 난이도**: medium
- **우선순위**: 2

## Activity B 아이디어

### 아이디어 B-1: HMAConnectorNonContiguousSegmentMigrationCache (HMA 커넥터 기반 비연속 세그먼트 계층 마이그레이션 캐시)
- **가설**: 비연속 세그먼트 캐시에서 자주 재사용되는 세그먼트는 HBM에, 덜 사용되는 세그먼트는 CPU DDR에, 드물게 사용되는 세그먼트는 디스크에 자동 마이그레이션하되, vLLM v0.21.0 HMA의 멀티-커넥터 아키텍처를 활용해 각 계층 간 마이그레이션을 HMA 커넥터 인터페이스로 통일하면, 비연속 세그먼트의 계층 간 이동이 표준화된 HMA 커넥터 API를 통해 이루어지므로 커스텀 전송 로직 없이 vLLM 프레임워크 내에서 비연속 세그먼트 멀티-티어 캐시를 구현할 수 있다. 이전 B 기법들(RoPEReencodingNonContiguousCache, KVFoldAccumulativeRadixCache 등)은 단일 메모리 계층(GPU HBM)에서만 세그먼트를 관리했다.
- **접근 방법**:
  1. `HMASegmentTierCache` 클래스 (`src/cache/hma_segment_tier_cache.py`): `CacheStore` 인터페이스 상속
  2. **3-티어 세그먼트 레지스트리**: 각 세그먼트 ID에 대해 현재 계층 위치를 추적하는 레지스트리 `Dict[segment_id → Tier]`. Tier = {HBM, DDR, DISK}. 기존 `src/cache/segmented.py`의 LRU 퇴거 정책을 확장해 퇴거 시 DISK 삭제 대신 DDR 마이그레이션.
  3. **HMA 커넥터 기반 계층 이동**: HBM → DDR: `HMAConnector.store(segment_id, kv_tensor)` 호출 (DCP 이벤트 대기). DDR → HBM: `HMAConnector.prefetch(segment_id)` 호출 (PCP 이벤트 대기 후 반환). DDR → DISK: 별도 DISK 커넥터 호출. HMA 커넥터 API를 `SegmentStorageConnector` 인터페이스로 추상화해 vLLM v0.21.0 HMA와 독립 구현 양쪽을 지원.
  4. **재사용 빈도 기반 계층 결정 정책**: 각 세그먼트의 `access_count`와 `last_access_time` 추적. `access_count ≥ hot_threshold`(기본 3): HBM 유지. `1 ≤ access_count < hot_threshold` and `last_access_time` 최근 `warm_window`(기본 5분) 내: DDR 유지. 그 외: DISK 마이그레이션 또는 퇴거. 세그먼트 재사용 시 자동 HBM 프로모션(DDR→HBM 프리페치).
  5. **비연속 세그먼트 히트 시 계층 인식 조합**: `get_segments(segment_ids)` 호출 시 각 세그먼트의 현재 계층 확인 → HBM 세그먼트는 즉시 반환, DDR 세그먼트는 비동기 프리페치 후 반환, DISK 세그먼트는 ondemand 복구. 혼합 계층 세그먼트 집합도 비연속 재사용 가능.
  6. **이전 B 기법과의 차별점**: 이전 모든 B 기법이 GPU HBM 내에서의 세그먼트 저장/조회만 다룬 반면, HMASegmentTierCache는 vLLM v0.21.0 HMA를 통해 세그먼트 캐시 자체를 HBM/DDR/DISK 3-티어로 확장한다. "비연속 세그먼트의 메모리 계층 마이그레이션"이라는 새로운 B 설계 차원으로, KVRMStaticGraphCoalescingSegmentScheduler(05-16 A-1)의 물리 전송 최적화와 직교하고 NAtHDDROffloadingScheduler(05-16 A-2)의 KV 오프로딩과도 다른 "세그먼트 캐시 자체의 멀티-티어 관리"다.
- **예상 효과**: 비연속 히트율 +20~35%p (DDR/DISK 세그먼트까지 히트 가능), 유효 세그먼트 캐시 용량 10~50× 증가 (HBM 용량 한계를 DDR/DISK로 확장), TTFT 증가 +5~8% (DDR 프리페치 지연 포함 — hot segment HBM 히트는 동일)
- **구현 난이도**: medium
- **우선순위**: 2

## Activity C 아이디어

### 아이디어 C-1: OnlineAdaptivePrecisionQuantizerForRLWorkloads (RL 워크로드용 온라인 적응 정밀도 KV 양자화기)
- **가설**: vLLM Q2 2026 로드맵이 "생산/RL 워크로드용으로 더 유연·메모리효율적인 온라인 양자화 리팩터"를 공식화함에 따라, RL 추론 특유의 반복적 생성-평가 사이클에서 각 디코딩 스텝의 누적 리워드 신호를 양자화 정밀도 결정에 피드백으로 사용하는 "리워드-인식 온라인 적응 KV 양자화기"가 가능하다. 이전 16개 사이클의 모든 C 기법은 정적 캘리브레이션(offline) 또는 고정 정밀도 정책을 사용했으며, RL 추론의 동적 리워드 신호를 양자화 정밀도에 실시간 반영한 기법은 전혀 없었다.
- **접근 방법**:
  1. `RLAdaptivePrecisionQuantizer` 클래스 (`src/cache/rl_adaptive_precision_quantizer.py`): `CompressionCodec` 인터페이스 구현
  2. **RL 워크로드 감지**: RL 추론 워크로드를 감지하는 휴리스틱 — 동일 프롬프트에 대한 반복 생성(num_completions > 1), CoT 길이(> `cot_length_threshold`, 기본 512 토큰), 또는 외부 `is_rl_mode=True` 플래그.
  3. **온라인 적응 정밀도 정책**:
     - RL 생성 사이클 초기(첫 `warmup_steps`, 기본 10 디코딩 스텝): FP16 전체 정밀도 유지 (탐색 단계, 정밀 정보 유지 필요).
     - 생성 진행 중 누적 어텐션 엔트로피 `H_t = -Σ_i p_i log p_i` 계산. 어텐션 엔트로피가 낮은 토큰(집중적 어텐션, 고중요도): FP16 유지. 어텐션 엔트로피가 높은 토큰(분산된 어텐션, 저중요도): INT8 또는 INT4로 동적 다운스케일.
     - 리워드 신호 피드백(RL 평가 이후): 직전 생성의 리워드 스코어가 `high_reward_threshold`(기본 0.8) 이상이면 다음 생성에서 더 공격적인 압축(낮은 정밀도) 허용. 리워드 스코어가 낮으면 정밀도를 자동으로 높이는 피드백 루프.
     - 정밀도 레벨: {FP16, INT8, INT4} 중 선택. 전환 기준을 `precision_schedule` YAML 외부화.
  4. **어텐션 엔트로피 기반 채널별 정밀도 할당**:
     - `attention_entropy_per_token(token_i) = -Σ_h Σ_t attn_{h,t,i} log(attn_{h,t,i} + ε)` 계산 (모든 헤드 평균).
     - 상위 20% 저엔트로피(고중요도) 토큰: FP16.
     - 중간 60%: INT8.
     - 하위 20% 고엔트로피(저중요도) 토큰: INT4.
     - 비율은 `precision_ratio_fp16/int8/int4` YAML 외부화.
  5. **vLLM Q2 2026 온라인 양자화 리팩터 통합 경로**: vLLM의 INT8 동적 퍼-토큰 양자화 프레임워크(Q2 2026 완성 예정)에 `RLAdaptivePrecisionQuantizer`를 외부 플러그인으로 등록하는 경로 설계. 플러그인 인터페이스: `apply_online_quantization(kv_tensor, step_id, reward_signal) → quantized_kv`.
  6. **이전 C 기법과의 차별점**: 기구현된 RateQuantReverseWaterfillingCodec(05-11)은 헤드 분산 기반 오프라인 최적 비트 할당, GlobalRetentionGateEvictionCodec(05-16)은 전역 리텐션 스코어 기반 퇴거, LookaheadKVEvictionCodec(05-15)은 미래 패턴 예측 기반 퇴거다. RLAdaptivePrecisionQuantizer는 "RL 추론 사이클의 리워드 신호 + 온라인 어텐션 엔트로피를 실시간으로 양자화 정밀도 결정에 반영"하는 완전히 새로운 C 메커니즘으로, 이전 16개 사이클 전체에 없던 "RL-aware online adaptive quantization" 패러다임이다.
- **예상 효과**: 메모리 −40~70% (RL 생성 중 어텐션 엔트로피 기반 동적 압축), accuracy delta ±0.5% 이내 (고중요도 토큰 FP16 보존 + 리워드 피드백 루프), RL 추론 처리량 +20~35% (INT8/INT4 압축 토큰 증가로 HBM 공간 확보)
- **accuracy-preserving 근거**: (1) 어텐션 엔트로피가 낮은(집중도 높은) 상위 20% 토큰은 항상 FP16 전체 정밀도로 유지하므로 모델 출력 품질의 핵심 정보가 보존된다. (2) 리워드 피드백 루프가 압축으로 인한 정확도 저하를 직접 측정·보정하는 자동 안전장치 역할을 한다. (3) warmup_steps 기간 동안 FP16 완전 정밀도를 유지해 RL 탐색 초기의 중요 패턴이 손실되지 않는다. 검증: WikiText-2 perplexity ±1% + RL 추론 태스크(GSM8K, MATH-500) 리워드 스코어 변화 + 정밀도 레벨별(FP16/INT8/INT4) 정확도 곡선 + 리워드 피드백 루프 수렴 스텝 측정.
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 C-2: NVFPFourDynamicKVQuantizationBudgetAdapter (NVFP4 동적 KV 양자화 예산 어댑터)
- **가설**: vLLM Q2 2026 로드맵이 INT8 → NVFP4 순차 지원을 공식화함에 따라, 동일 연산자 내에서 INT8과 NVFP4를 요청 및 레이어 특성에 따라 동적으로 선택하는 "혼합 정밀도 동적 KV 양자화 예산 어댑터"가 필요하다. 기존 FibQuantVQCodec(05-14)이 구면 좌표 기반 VQ로 고정 압축 티어를 제공했다면, NVFP4 형식이 공식 지원되는 시점에 맞춰 "INT8(레이어별 안정 보장) + NVFP4(레이어별 고압축 허용)"의 이중 정밀도 전환 정책을 구현하면 기존 모든 양자화 기법 대비 더 넓은 메모리-정확도 트레이드오프 범위를 지원할 수 있다.
- **접근 방법**:
  1. `NVFPFourBudgetAdapter` 클래스 (`src/cache/nvfp4_budget_adapter.py`): `CompressionCodec` 인터페이스 구현
  2. **레이어별 정밀도 결정 프로파일**: 오프라인 캘리브레이션(100~200 시퀀스)에서 각 레이어의 "KV 텐서 동적 범위 비율" = `max(|kv|) / mean(|kv|)` 계산. 동적 범위 비율이 높은 레이어(아웃라이어 큰 레이어): INT8 유지 (NVFP4 아웃라이어 표현 제한 회피). 동적 범위 비율이 낮은 레이어(균등 분포): NVFP4 적용. 결과를 `configs/nvfp4_layer_precision_profile.yaml`에 저장.
  3. **동적 예산 제어**: `total_memory_budget_ratio`(기본 0.5) 파라미터로 전체 KV 메모리 예산 설정. NVFP4 레이어 수를 예산 내에서 최대화하되, 각 레이어의 정확도 기여도(`layer_importance_score`)를 고려해 중요도 낮은 레이어부터 NVFP4 적용. Lagrange 최적화: `max Σ NVFP4 layers, s.t. weighted accuracy error ≤ threshold`.
  4. **INT8/NVFP4 통합 양자화 커널 설계**: 두 정밀도 포맷을 단일 `quantize(kv, precision: Literal["int8", "nvfp4"]) → quantized_kv` 인터페이스로 처리. INT8: 채널별 min-max 스케일링. NVFP4: 4-bit float (vLLM NVFP4 백엔드와 인터페이스 일치 설계 — Q2 2026 리팩터 완성 시 직접 연결). 현재 구현: INT8 실구현 + NVFP4 시뮬레이션(FP16에서 4-bit float 시뮬레이션).
  5. **vLLM Q2 2026 로드맵 정렬**: INT8 동적 퍼-토큰 양자화 리팩터 완성 후 `NVFPFourBudgetAdapter`가 vLLM의 공식 온라인 양자화 프레임워크에 플러그인되는 경로를 설계. `apply_nvfp4_patch()` API로 vLLM 어텐션 백엔드에 주입.
  6. **이전 C 기법과의 차별점**: RateQuantReverseWaterfillingCodec(05-11)이 정보이론 최적 비트 할당(avg_bits=4.0 고정)을 사용하고, FibQuantVQCodec(05-14)이 구면 좌표 VQ 고정 티어를 사용하는 반면, NVFPFourBudgetAdapter는 "INT8과 NVFP4 포맷 간 레이어별 동적 전환 + 총 예산 기반 Lagrange 최적화"라는 새로운 접근을 도입한다. 특히 NVFP4 형식(Blackwell GPU 네이티브 지원)을 대상으로 하는 최초의 C 기법이며, vLLM Q2 2026 공식 로드맵과 직접 정렬된다.
- **예상 효과**: 메모리 −60~75% (NVFP4 적용 레이어 4-bit, INT8 적용 레이어 8-bit 혼합), accuracy delta ±0.8% 이내 (동적 범위 낮은 레이어만 NVFP4 적용), Effective Context Length 4× 이상 (NVFP4 레이어 용량 4× + INT8 레이어 2× 혼합)
- **accuracy-preserving 근거**: (1) 캘리브레이션 기반 동적 범위 비율로 아웃라이어가 큰 레이어는 INT8로 보호 — NVFP4 포화(saturation) 오류 방지. (2) 레이어 중요도 가중 Lagrange 최적화가 정확도 영향이 큰 레이어를 INT8로 자동 보호. (3) 기존 RateQuantReverseWaterfillingCodec과 동일한 역 물채우기 원칙 적용으로 정보이론적 최적성 유지. 검증: WikiText-2 perplexity ±1% + NVFP4 시뮬레이션 레이어별 오차 분포 측정 + 동적 범위 임계값(2.0/3.0/5.0) 별 정확도 곡선 + 기구현 RateQuantReverseWaterfillingCodec과 동일 설정 비교.
- **구현 난이도**: medium
- **우선순위**: 2

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: HMAPluginChainedACPipeline (A+C) — HMA 멀티-커넥터 플러그인 체이닝 A+C 통합 파이프라인
- **조합 근거**: HMAMultiConnectorScheduler(A-1)가 요청 특성에 따라 적절한 압축 커넥터를 선택하고, RLAdaptivePrecisionQuantizer(C-1)와 NVFPFourBudgetAdapter(C-2)가 각각 독립 HMA 커넥터로 등록되면, 스케줄러는 RL 워크로드에 C-1(어텐션 엔트로피 기반 적응 정밀도)을, 일반 긴 컨텍스트 요청에 C-2(NVFP4/INT8 혼합)를, 비연속 재사용 요청에 GlobalRetentionGateEvictionCodec을 동적 선택·조합하는 완전한 A+C 파이프라인이 된다. DCPPCPFeedbackScheduler(A-2)가 각 커넥터의 완료 이벤트를 추적해 배치 편성의 back-pressure를 제공함으로써 압축-스케줄링 피드백 루프가 완성된다. 이전 05-16의 Cross-1(NAtHRetentionTierDecider, A+C)이 이중 신호(누적 어텐션 + 전역 리텐션)의 티어 결정 통합이었다면, 이번 Cross-1은 "멀티-커넥터 플러그인 API를 통한 압축 정책의 동적 디스패치 + DCP/PCP 이벤트 기반 피드백"이라는 완전히 다른 A+C 통합 축이다.
- **구체 설계**:
  1. `HMAChainedACPipeline` 모듈 (`src/engine/hma_chained_ac_pipeline.py`)
  2. **커넥터 레지스트리 초기화**: 시작 시 3개 C 커넥터 등록 — `"rl_adaptive"`: RLAdaptivePrecisionQuantizer(C-1), `"nvfp4_budget"`: NVFPFourBudgetAdapter(C-2), `"global_retention"`: GlobalRetentionGateEvictionCodec(05-16 기구현).
  3. **요청 프로파일 기반 커넥터 선택 규칙** (YAML 외부화):
     - `is_rl_mode=True` 또는 `num_completions > 1`: `"rl_adaptive"` 선택
     - `context_length > 4096` and `is_rl_mode=False`: `"global_retention"` 선택
     - `context_length <= 4096` and `memory_pressure > threshold`: `"nvfp4_budget"` 선택
     - 기본값: `"global_retention"`
  4. **DCP/PCP 피드백 통합**: DCPPCPFeedbackScheduler(A-2)가 각 커넥터의 완료 이벤트를 추적. 커넥터 체인 실행 중 중간 커넥터의 DCP 이벤트가 발생하면 다음 커넥터 실행 자동 트리거.
  5. **파이프라인 체이닝 모드**: `chain_mode=True` 시 선택된 커넥터 + `GlobalRetentionGateEvictionCodec`을 순차 적용 (write-time 압축 → post-write 퇴거 이중 필터).
  6. **B 연동**: DCPPCPFeedbackScheduler의 DCP 완료 이벤트가 HMASegmentTierCache(B-1)의 `mark_offload_complete(segment_id)` 호출로 연결 → 오프로딩 완료 세그먼트가 즉시 비연속 재사용 후보로 인덱싱.
- **예상 복합 효과**: 처리량 +30~50% (요청별 최적 압축 정책 + DCP/PCP 피드백으로 HBM 효율 극대화), 메모리 −50~75% (커넥터별 최적 압축 + 파이프라인 체이닝), accuracy delta ±0.8% 이내 (커넥터별 accuracy-preserving 보장)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 Cross-2: DCPPCPDrivenNonContiguousSegmentIndexer (A+B) — DCP/PCP 이벤트 기반 비연속 세그먼트 즉시 인덱싱 파이프라인
- **조합 근거**: DCPPCPFeedbackScheduler(A-2)의 DCP(오프로딩 저장 완료) 이벤트와 HMASegmentTierCache(B-1)의 DDR 세그먼트 레지스트리를 연결하면, "오프로딩 완료 즉시 해당 KV 세그먼트를 비연속 재사용 인덱스에 자동 등록"하는 A+B 파이프라인이 가능하다. 이전 사이클에서 A+B 결합(HitAwarePPDRouter 05-09, KVRMStaticGraphCoalescingSegmentScheduler+SPKVSparsityProfiledSegmentReusePolicy 05-16)은 "라우팅 + 재사용 정책" 또는 "전송 최적화 + 헤드 선택"에 집중했으나, "오프로딩 완료 이벤트 → 비연속 세그먼트 인덱싱 자동화"라는 이벤트 기반 A+B 통합은 새로운 조합이다.
- **구체 설계**:
  1. `DCPDrivenSegmentIndexer` 모듈 (`src/cache/dcp_driven_segment_indexer.py`)
  2. DCP 이벤트 수신 시 해당 job_id의 KV 세그먼트를 청크 분할 → SHA-256 위치-독립 해시 계산 → `HMASegmentTierCache`의 DDR 레지스트리에 자동 등록.
  3. PCP 이벤트(DDR → HBM 프리페치 완료) 수신 시 해당 세그먼트를 HBM 레지스트리로 승격.
  4. 기존 `SegmentedHashCache`와의 통합: `on_dcp_complete(job_id, kv_segments)` 콜백이 `SegmentedHashCache.put_batch(segment_ids, kv_tensors)` 호출.
  5. 이벤트 기반 인덱싱으로 오프로딩 완료와 재사용 가능성 사이의 지연(indexing lag) 제거.
- **예상 복합 효과**: 처리량 +20~35% (DCP 완료 즉시 재사용 가능으로 세그먼트 재사용 기회 확대), 비연속 히트율 +15~25%p (DDR 세그먼트까지 히트 풀 확장), Scheduling overhead TTFT +2% 이내 (이벤트 콜백 오버헤드 < 0.02ms)
- **구현 난이도**: low
- **우선순위**: 3

## 이번 사이클 최우선 구현 타겟
**A-1 (HMAMultiConnectorCompressionPluginScheduler)** 과 **C-1 (RLAdaptivePrecisionQuantizer)** 및 **Cross-1 (HMAChainedACPipeline, A+C)** 를 최우선 구현 타겟으로 권장한다.

이유:
1. C-1(RLAdaptivePrecisionQuantizer)은 온라인 어텐션 엔트로피 기반 적응 정밀도라는 완전히 새로운 C 패러다임으로, 구현 난이도 medium이고 vLLM Q2 2026 로드맵 방향과 직접 정렬된다. 기구현 GlobalRetentionGateEvictionCodec과 동일 설정에서 RL 워크로드 시나리오로 비교하면 "리워드 피드백 루프 있음 vs. 없음"의 효과를 정량화할 수 있다.
2. A-1(HMAMultiConnectorScheduler)은 vLLM v0.21.0 HMA 공식 API를 활용하는 새로운 스케줄러 설계로, 기구현된 C 코덱들(GlobalRetentionGateEvictionCodec, LookaheadKVEvictionCodec, RateQuantReverseWaterfillingCodec)을 즉시 커넥터로 재활용할 수 있어 구현 효율이 높다.
3. Cross-1(HMAChainedACPipeline)은 A-1과 C-1의 자연스러운 A+C 통합으로, 이전 사이클 Cross-1(NAtHRetentionTierDecider)과 차별화된 "플러그인 아키텍처 기반 A+C" 접근이다.
4. A-2(DCPPCPFeedbackScheduler)는 구현 난이도 medium으로 A-1 이후 다음 우선순위. B-1(HMASegmentTierCache)은 B의 새로운 멀티-티어 방향으로 A-2와 결합하면 Cross-2로 확장 가능.
5. C-2(NVFPFourBudgetAdapter)는 NVFP4 공식 지원 완성(Q2 2026) 후 이식이 더 효과적이므로 이번 사이클에서는 INT8 실구현 + NVFP4 시뮬레이션으로 프로토타이핑 권장.

## planner에게 전달할 요청사항
- **A-1(HMAMultiConnectorScheduler) 구현 시**: `src/scheduler/hma_multi_connector_scheduler.py` 신규 생성. `HMAConnectorInterface` 추상 베이스 클래스 정의 (`compress(kv: Tensor, request_profile: dict) → Tensor` 메서드). `register_connector(name: str, codec: CompressionCodec)` 레지스트리 API. `select_connector(request: Request) → str` 선택 메서드. `connector_dispatch_policy` YAML 외부화. 기구현 코덱 3종(GlobalRetentionGateEvictionCodec, LookaheadKVEvictionCodec, RateQuantReverseWaterfillingCodec)을 HMAConnectorInterface로 래핑하는 어댑터 클래스 작성. `pipeline_mode=True` 파이프라인 체이닝 지원. **스케줄링 오버헤드 검증 필수**: 커넥터 선택 오버헤드 < 0.1ms/요청, 기구현 NAtHDDROffloadingScheduler와 동일 설정 TTFT 비교.
- **C-1(RLAdaptivePrecisionQuantizer) 구현 시**: `src/cache/rl_adaptive_precision_quantizer.py` 신규 생성. `CompressionCodec` 인터페이스 구현. 어텐션 엔트로피 계산: `torch.log(attn_weights + 1e-8)` 기반 Shannon 엔트로피. RL 감지 휴리스틱: `num_completions > 1` or `is_rl_mode=True` 플래그. 정밀도 레벨 `{FP16, INT8, INT4}` 전환 + `precision_ratio_fp16/int8/int4 = [0.2, 0.6, 0.2]` YAML 외부화. `warmup_steps=10` YAML 외부화. 리워드 피드백: `update_reward_signal(reward: float)` 인터페이스 (외부 RL 환경에서 주입). **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + GSM8K/MATH-500 정확도 + RL 워크로드 시뮬레이션(동일 프롬프트 10회 반복 생성) + 정밀도 레벨별 attention error 곡선 + 기구현 GlobalRetentionGateEvictionCodec과 동일 설정 비교.
- **A-2(DCPPCPFeedbackScheduler) 구현 시**: `src/scheduler/dcppcp_feedback_scheduler.py` 신규 생성. `OffloadingEventBus` 경량 이벤트 버스 구현 (`threading.Event` 또는 `asyncio.Event` 기반). `subscribe_dcp/subscribe_pcp(job_id, callback)` API. DCP/PCP vLLM v0.21.0 HMA 인터페이스 래핑: `OffloadingConnectorAdapter` 클래스. `max_offload_wait_ms=50` YAML 외부화. `effective_hbm_free` 실시간 계산 및 `results/<exp-name>/metrics.json` 기록. **스케줄링 오버헤드 검증 필수**: 이벤트 버스 오버헤드 < 0.05ms/요청, 기구현 NAtHDDROffloadingScheduler 대비 HBM 활용률 향상 측정.
- **B-1(HMASegmentTierCache) 구현 시**: `src/cache/hma_segment_tier_cache.py` 신규 생성. `CacheStore` 인터페이스 상속. `SegmentStorageConnector` 추상 인터페이스 (HMA 커넥터와 독립 구현 양쪽 지원). `access_count` + `last_access_time` 추적 딕셔너리. `hot_threshold=3`, `warm_window=300` (초) YAML 외부화. `get_segments(segment_ids)` 계층 인식 혼합 반환. `mark_offload_complete(segment_id)` 인터페이스. **비연속 히트율 검증 필수**: DDR 세그먼트 포함 히트율 vs. HBM 전용 히트율 비교 + 계층별 세그먼트 분포 통계.
- **Cross-1(HMAChainedACPipeline) 구현 시**: `src/engine/hma_chained_ac_pipeline.py` 신규 생성. A-1 커넥터 레지스트리와 C-1/C-2/기구현 코덱 3종 통합. `chain_mode` YAML 외부화. DCP 이벤트 → B-1 `mark_offload_complete()` 연결. **복합 효과 측정 필수**: 처리량/메모리/정확도 3방향 비교 — 단독 A-1 + 단독 C-1 + 결합 Cross-1 + 기구현 Cross-1(NAtHRetentionTierDecider).
- **목표 지표 중 이번 사이클에서 측정할 지표**:
  - Inference Throughput (A-1 커넥터 선택 오버헤드 포함 전체 처리량 +20% 이상 목표)
  - KV Cache Memory Reduction (C-1 RL 적응 정밀도 −40~70%, C-2 NVFP4 시뮬레이션 −60~75% 목표)
  - Compression Accuracy Delta (필수 ±1% 이내 — RL 워크로드 리워드 피드백 수렴 후 측정)
  - Scheduling Overhead (A-1 커넥터 선택 + A-2 DCP/PCP 이벤트 총 오버헤드 TTFT +5% 이내)
- **Activity C 구현 시 공통 필수 사항**: 모든 C 기법은 `src/metrics/` 내 perplexity 측정 코드 포함 + 압축 전후 perplexity를 `results/<exp-name>/metrics.json`에 기록. C-1은 RL 워크로드 시뮬레이션(동일 프롬프트 10회 반복)에서의 리워드 피드백 수렴 곡선을 추가 측정 필수.
- **vLLM 0.21.0 이식 경로**: A-1 HMAMultiConnectorScheduler → `vllm_integration/scheduler_patch.py`에 `HMAMultiConnectorSchedulerMixin` 추가. `make_hma_multi_connector_scheduler_class()` 팩토리 API. C-1 RLAdaptivePrecisionQuantizer → `vllm_integration/attention_backend_patch.py`에 `RLAdaptivePrecisionAttentionHook` 추가. vLLM v0.21.0의 멀티-커넥터 HMA 인터페이스와 직접 연동되는 `HMAConnectorAdapter` 구현 포함.
