## Activity A 아이디어

### 아이디어 A-1: KVDriveAttentionAwarePipelineScheduler (KVDrive 어텐션-인식 3계층 파이프라인 스케줄러)
- **가설**: KVDrive(2605.18071)의 핵심 세 원칙 — (1) 어텐션 점수 기반 슬라이딩 윈도우로 HBM 잔류 항목 결정, (2) I/O-컴퓨트 오버랩 파이프라인 재구성으로 계층 간 스톨 제거, (3) GPU HBM·호스트 DRAM·SSD 3계층 조율 — 을 `src/scheduler/` 레이어에서 구현하면, 스파스성 임계값에 도달한 이후에도 KV 전송량을 지속적으로 줄일 수 있어 기존 NAtHDDROffloadingScheduler(05-16, DDR까지 2계층, 어텐션 EMA 기반)와 HMAMultiConnectorScheduler(05-17, 커넥터 선택 메타-스케줄러)가 해결하지 못한 "SSD까지 포함한 3계층 전체의 I/O-컴퓨트 오버랩 스케줄링"을 달성할 수 있다. 특히 배치 크기와 컨텍스트 길이가 증가할수록 KV 전송량이 급증하는 구조적 한계를 파이프라인 재구성으로 제거하는 것이 핵심이다.
- **접근 방법**:
  1. `KVDriveAttentionPipelineScheduler` 클래스 (`src/scheduler/kvdrive_attention_pipeline_scheduler.py`): `BaseScheduler` 상속
  2. **어텐션 행동 기반 계층 배치 정책 (Attention-Based Placement)**:
     - 각 KV 토큰에 대해 누적 어텐션 점수 `cumul_attn(token_i) = Σ_t attn(q_t, k_i)` 계산 (NAtH EMA 방식과 유사하나 3-tier 목적지 결정에 특화).
     - 슬라이딩 윈도우 `W_hbm` (기본 최근 `local_window_size=512` 토큰은 항상 HBM 유지).
     - 슬라이딩 윈도우 밖 토큰의 계층 결정:
       - `cumul_attn ≥ attn_hbm_threshold` (기본 상위 20%): **HBM 잔류** — 핵심 장기 컨텍스트 토큰
       - `attn_dram_threshold ≤ cumul_attn < attn_hbm_threshold` (기본 20~70%): **DRAM 오프로딩** — 중간 중요도
       - `cumul_attn < attn_dram_threshold` (기본 하위 30%): **SSD 오프로딩** — 장기 보관, 드문 접근
     - 모든 임계값 `attn_hbm_threshold`, `attn_dram_threshold`, `local_window_size` YAML 외부화 (`configs/kvdrive_placement_policy.yaml`).
     - **NAtHDDROffloadingScheduler(05-16)와의 차별점**: NAtH가 4-티어 분류 + 영구 퇴거 비율 3% 제약에 집중했다면, 이 기법은 SSD를 세 번째 계층으로 포함하고 퇴거(eviction) 없이 SSD 보관으로 대체하며, I/O-컴퓨트 오버랩 파이프라인 재구성을 핵심 설계 요소로 추가한다.
  3. **파이프라인 재구성 (I/O-Compute Overlap)**:
     - KVDrive의 핵심 발견: I/O 바운드 단계와 컴퓨트 바운드 단계를 명시적으로 오버랩하는 "파이프라인 재구성"으로 이종 자원 간 스톨을 제거.
     - 구현: 디코딩 스텝 내 두 개의 CUDA Stream 운영:
       - **Stream A (KV I/O 스트림)**: DRAM/SSD에서 다음 디코딩 스텝에 필요한 KV를 HBM으로 비동기 프리페치.
       - **Stream B (컴퓨트 스트림)**: 현재 디코딩 스텝의 어텐션 계산 + 결과 처리.
     - Stream A의 프리페치 완료를 `torch.cuda.Event.wait()`으로 Stream B 어텐션 계산 시작 전에 동기화. 단, 프리페치가 늦을 경우 HBM 잔류 항목만으로 즉시 계산 시작 (대기 없는 graceful fallback).
     - **3계층 I/O 조율**: HBM→DRAM 전송: `torch.cuda.Stream` 비동기. DRAM→HBM 프리페치: 동일. SSD→DRAM 전송: `asyncio.run_in_executor` 비동기 파일 I/O. SSD 접근 레이턴시 보정: 선점적으로 2~3 디코딩 스텝 전에 SSD 프리페치 요청 발행.
  4. **계층별 KV 레지스트리**:
     - `KVTierRegistry` 클래스 (`src/scheduler/kv_tier_registry.py`): 각 토큰 ID → {tier: HBM/DRAM/SSD, physical_location, approx_size} 매핑.
     - 계층 결정 갱신 주기: 매 `tier_update_interval`(기본 32) 디코딩 스텝마다 누적 어텐션 점수 재계산 + 계층 재배정.
     - `KVTierRegistry.get_tier(token_id)` → 현재 계층 위치 O(1) 조회.
  5. **기존 A 기법과의 차별점 요약**:
     - NAtHDDROffloadingScheduler(05-16): DDR까지 2계층, 퇴거 비율 3% 제약, EMA 어텐션 기반 티어 결정 → 이 기법: SSD까지 3계층, 영구 퇴거 없음(SSD 보관), 슬라이딩 윈도우 + 누적 어텐션 기반 HBM 잔류 결정 + I/O-컴퓨트 오버랩 파이프라인.
     - HMAMultiConnectorScheduler(05-17): HMA 커넥터 API 기반 압축 정책 디스패치 메타-스케줄러 → 이 기법: KV 데이터 자체의 3계층 물리적 이동 조율 + 파이프라인 재구성.
     - AMPDLazySegmentFetchScheduler(05-18): 세그먼트 집합 확정 전 전송 보류(pull-on-demand) → 이 기법: 어텐션 행동 기반 계층 배치 + I/O-컴퓨트 오버랩 (타이밍 제어 방향이 다름: 확정 후 pull vs. 예측 기반 push 조율).
  6. **vLLM v0.21.0 HMA 통합 경로**: `vllm_integration/scheduler_patch.py`에 `KVDriveAttentionPipelineMixin` 추가. vLLM v0.21.0 HMA 멀티-커넥터 아키텍처: HBM 커넥터(기본), DRAM 커넥터(DCP/PCP 이벤트 기반), SSD 커넥터(asyncio I/O). `KVTierRegistry`를 HMA 커넥터 레지스트리로 연결.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드에서 각 노드가 독립적으로 3계층 스케줄러를 실행. 노드 간 KV 마이그레이션 시: 소스 노드 DRAM/SSD의 KV를 대상 노드 DRAM으로 RDMA 전송 후 HBM 프리페치. SSD 계층의 KV는 노드 간 전송이 느리므로, DRAM 계층 KV만 노드 간 공유 (SSD는 각 노드 로컬 보관). InfiniBand 환경: DRAM→DRAM RDMA (200Gbps) + SSD 로컬 보관 정책.
- **예상 효과**: 처리량 +35~60% (KVDrive 원논문 1.74× 기반, 파이프라인 오버랩으로 디코딩 지연 제거), KV Cache Memory Reduction −70~85% (SSD까지 3계층 오프로딩으로 HBM 점유 최소화), Cache Eviction Rate ≈ 0% (영구 퇴거 없이 SSD 보관으로 대체), Scheduling overhead TTFT +4% 이내 (계층 결정 오버헤드: `KVTierRegistry.get_tier()` O(1) + 어텐션 점수 EMA 갱신 < 0.05ms/스텝)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 A-2: ThunderAgentLLMProgramKVThrashingPreventionScheduler (ThunderAgent LLM Program 기반 KV 쓰래싱 방지 스케줄러)
- **가설**: 에이전틱 서빙에서 기존 요청 단위 스케줄링이 KV 캐시 쓰래싱(thrashing) — 활성 KV가 메모리를 가득 채워 지속적 퇴거·재로드 루프 발생 — 을 방지하지 못하는 한계를, ThunderAgent(2602.13692)의 LLM Program 추상화 — 에이전틱 워크플로우를 정적으로 파싱해 각 스텝의 KV 수요와 재사용 가능성을 사전에 파악 + 과부하 노드의 프로그램을 주기적으로 일시 중지해 KV 캐시 공간 확보 + 비동기 환경 사전 준비로 실행-준비 대기 오버랩 — 로 해결하면, 에이전틱 워크플로우 전체를 1급 스케줄링 단위로 관리해 KV 쓰래싱을 구조적으로 방지하고 처리량을 극대화할 수 있다. PBKV(05-13)가 확률적 미래 호출 예측으로 KV를 보존하는 접근이라면, 이 기법은 정적 Program 구조 분석으로 KV 수요를 결정론적으로 파악하고 쓰래싱 방지를 위한 프로그램-레벨 선점을 수행하는 근본적으로 다른 설계다.
- **접근 방법**:
  1. `ThunderAgentProgramScheduler` 클래스 (`src/scheduler/thunder_agent_program_scheduler.py`): `BaseScheduler` 상속
  2. **LLM Program 파서 (Program Parser)**:
     - 에이전틱 워크플로우를 입력으로 받아 `LLMProgram` 추상화로 파싱. `LLMProgram = {steps: [ProgramStep], kv_dependency_graph: DAG}`.
     - `ProgramStep = {step_id, input_context_hash, estimated_kv_size, can_reuse_from: List[step_id], tool_requirements: List[str]}`.
     - 파서 구현: 워크플로우 명세(JSON/YAML)를 입력으로 받는 `LLMProgramParser` 클래스. 각 스텝의 입력 컨텍스트 해시(`SimHash 64-bit`)와 이전 스텝과의 컨텍스트 공유 비율을 분석해 `can_reuse_from` 의존성 DAG를 구성.
     - 파싱 오버헤드: 워크플로우당 1회 파싱 (스텝 수 N에 대해 O(N²) 의존성 분석, N ≤ 100 에이전틱 스텝 기준 < 10ms).
  3. **KV 수요 사전 예측 및 할당**:
     - `LLMProgram`의 KV 의존성 DAG를 기반으로 각 스텝의 KV 수요를 사전 계산: `kv_demand(step) = estimated_kv_size(step) - reusable_kv_size(step)`.
     - `reusable_kv_size(step)` = `can_reuse_from` 스텝들의 KV 중 현재 캐시에 있는 비율 × 해당 KV 크기 합.
     - GPU 메모리 예산 `memory_budget` 대비 전체 Program의 총 KV 수요 `total_kv_demand`를 사전 계산: `memory_pressure = total_kv_demand / memory_budget`.
  4. **주기적 모니터 + 과부하 노드 프로그램 일시 중지 (Periodic Monitor with Thrashing Prevention)**:
     - ThunderAgent의 핵심 메커니즘: 주기적 모니터(`monitor_interval_ms`, 기본 50ms)가 각 노드의 활성 Program 수와 KV 캐시 사용률을 확인.
     - **쓰래싱 감지 조건**: `kv_cache_utilization > thrash_threshold` (기본 0.90) AND `eviction_rate > evict_rate_threshold` (기본 0.15 = 15%/s).
     - **쓰래싱 방지 조작**: 쓰래싱 감지 시 우선순위가 낮은 Program을 **일시 중지(pause)** — 해당 Program의 현재 KV를 `KVDriveAttentionPipelineScheduler(A-1)의 DRAM/SSD` 계층으로 이동 후 전역 대기 큐로 이동. 메모리 확보 후 재개.
     - **재개(resume) 조건**: 메모리 여유(`kv_cache_utilization < resume_threshold`, 기본 0.70) 회복 시 대기 큐에서 Program 재개.
     - 모니터 오버헤드: 50ms 주기 + `kv_cache_utilization` 조회 O(1) → 무시 가능.
  5. **도구 자원 비동기 사전 준비 (Async Tool Environment Prefetch)**:
     - ThunderAgent의 두 번째 핵심: 다음 스텝에서 호출할 외부 도구 환경을 현재 스텝 실행 중에 비동기로 준비.
     - 구현: `ToolResourcePrefetcher` (`src/scheduler/tool_resource_prefetcher.py`): `asyncio.create_task(prefetch_tool_env(next_step.tool_requirements))`로 도구 환경 준비 시작. 다음 스텝 시작 전 `await prefetch_task`.
     - KV 관련성: 도구 실행 대기 시간에 KV 프리페치(A-1의 DRAM→HBM 프리페치)를 오버랩해 도구-실행 대기 = KV I/O 스톨을 동시에 해소.
  6. **PBKV(05-13)와의 차별점**:
     - PBKVAgentSegmentPreservationScheduler(05-13): 미래 에이전트 호출을 확률적 MLP 예측기로 추정 → KV 세그먼트를 GPU/호스트에 선점 보존하는 "확률적 예측 기반 보존".
     - ThunderAgentProgramScheduler: 워크플로우를 LLM Program으로 정적 파싱 → KV 수요를 결정론적으로 분석 → 쓰래싱 방지를 위한 프로그램-레벨 일시 중지 + 도구 환경 비동기 준비라는 "정적 분석 기반 쓰래싱 방지".
     - 핵심 차이: PBKV는 "어떤 KV를 보존할지"의 보존 결정이고, ThunderAgent는 "언제 프로그램을 멈출지"의 선점 결정 + 쓰래싱 방지 제어 구조.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드에서 `PeriodicMonitor`가 클러스터 전체 노드의 KV 캐시 사용률을 집계. 과부하 노드의 Program을 언더-로드 노드로 마이그레이션 (Program-레벨 마이그레이션 = KV 세그먼트 + 프로그램 상태 전체를 RDMA로 이동). 마이그레이션 비용 vs. 쓰래싱 손실 비교 결정: `migration_cost_ms < thrashing_loss_ms` 조건으로 마이그레이션 결정. InfiniBand 200Gbps 기준: 1GB KV = 40ms 전송, 쓰래싱 손실 > 40ms이면 마이그레이션 실행.
- **예상 효과**: 처리량 +50~260% (ThunderAgent 원논문 vLLM 대비 1.5~3.6× 기반, RL 롤아웃 환경 1.8~3.9×), KV 쓰래싱 발생률 −80~90% (프로그램-레벨 일시 중지로 쓰래싱 구조적 방지), TTFT p50 +4% 이내 (파서 1회 오버헤드 < 10ms + 모니터 50ms 주기)
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 B-1: ThunderAgentProgramStaticNonContiguousSegmentReservationCache (ThunderAgent Program 정적 분석 기반 비연속 세그먼트 사전 예약 캐시)
- **가설**: ThunderAgent(2602.13692)의 LLM Program 정적 분석이 에이전틱 워크플로우의 어느 스텝 간에 KV 재사용이 가능한지를 결정론적으로 파악한다는 점을, Activity B의 비연속 세그먼트 재사용에 적용하면, 실행 시점(동적) 재사용 결정을 내리기 전에 파싱 시점(정적)에 재사용 가능한 비연속 세그먼트 집합을 사전 예약(reserve)하고 해당 세그먼트를 적절한 메모리 계층에 미리 배치할 수 있다. 기존 B 기법들(RelayUShapeLayerSelectiveSegmentCache, KVFoldAccumulativeRadixCache, AMPDAdapShotLazyLoadPipeline 등)이 모두 "세그먼트 재사용 결정을 실행 시 동적으로 내리는" 접근인 반면, Program 정적 분석으로 실행 전에 재사용 계획을 수립하는 "정적-예약(static reservation)" 패러다임은 이전 19개 사이클 전체에 없었다.
- **접근 방법**:
  1. `ThunderAgentStaticSegmentReservationCache` 클래스 (`src/cache/thunder_agent_static_segment_reservation.py`): `CacheStore` 인터페이스 상속
  2. **LLM Program 파싱 → 재사용 가능 세그먼트 맵 구성**:
     - `ThunderAgentProgramScheduler(A-2)`의 `LLMProgram.kv_dependency_graph`에서 각 스텝 쌍 `(step_i, step_j)` 간의 KV 재사용 가능성을 추출.
     - 재사용 가능 조건: `step_j.can_reuse_from` ∋ `step_i` AND `SimHash(step_i.input_context) ∩ SimHash(step_j.input_context)` > `reuse_threshold` (기본 0.6).
     - 결과: `SegmentReservationMap = Dict[step_id → List[ReusableSegment]]`. `ReusableSegment = {segment_id, source_step_id, estimated_position_range, reuse_probability}`.
  3. **정적 예약 실행**:
     - Program 파싱 완료 즉시 `SegmentReservationMap`의 높은 재사용 확률 세그먼트(기본 `reuse_probability ≥ 0.5`)를 캐시에 **사전 예약(reserve)**: 해당 세그먼트 ID를 메모리에서 퇴거 불가(pinned) 상태로 마킹.
     - 예약된 세그먼트의 물리적 위치: `KVDriveAttentionPipelineScheduler(A-1)`의 계층 결정 결과에 따라 HBM/DRAM/SSD 중 적합한 계층에 배치. 높은 재사용 확률(`reuse_probability ≥ 0.8`) 세그먼트는 HBM 강제 배치(어텐션 점수 임계값 무시).
     - `reserve_segment(segment_id, min_tier=HBM, until_step_id)` 인터페이스: `until_step_id`에 해당하는 Program 스텝이 완료될 때까지 예약 유지.
  4. **동적 재사용과의 하이브리드 설계**:
     - 정적 예약이 없는 세그먼트(Program 파서가 파악하지 못한 재사용 기회): 기존 동적 결정(SegmentedHashCache, AdapShotLazyLoadPipeline 등)으로 fallback.
     - RelayCaching(05-15의 B 기법)의 동적 레이어 범위 프로파일러: 정적 예약 세그먼트 중 "어떤 레이어를 재사용할지"를 실행 시 결정하는 "정적 예약 + 동적 레이어 선택" 이중 결정 구조.
     - `SegmentReservationMap` 갱신: 실행 중 실제 재사용 결과 피드백 → 다음 Program 실행 시 `reuse_probability` 업데이트 (온라인 학습 없는 단순 EMA 갱신).
  5. **메모리 예산 관리**:
     - 예약된 세그먼트 총 크기가 `max_reservation_budget` (기본 GPU HBM의 20%) 초과 시 낮은 `reuse_probability` 세그먼트부터 예약 해제(DRAM 강등).
     - `ThunderAgentProgramScheduler(A-2)`의 쓰래싱 감지 이벤트 수신 시 일시적으로 모든 예약을 DRAM으로 강등 (메모리 확보 우선).
  6. **이전 B 기법과의 차별점**:
     - AMPDAdapShotLazyLoadPipeline(05-18 B-1): 세그먼트 집합 확정 후 지연 로드 + 재인코딩 파이프라인 (실행 시 결정).
     - SPKVSparsityProfiledSegmentReusePolicy(05-16 B-1): SP-KV 구조적 희소성 오프라인 프로파일 기반 헤드 선택 (모델 고정 프로파일).

- **예상 효과**: 비연속 히트율 +25~40% (정적 예약으로 필요한 세그먼트를 실행 전에 HBM에 배치 → 실행 시 즉시 히트), TTFT −35~55% (정적 예약 세그먼트 즉시 히트로 로드 지연 제거), 세그먼트 로드 지연 −60~80% (예약된 세그먼트는 HBM에서 즉시 반환)

## Activity C 아이디어

### 아이디어 C-1: KVDriveTierDifferentiatedCompressionScheduler (KVDrive 3계층 자동 차등 압축 스케줄러)
- **가설**: KVDrive(2605.18071)의 HBM·DRAM·SSD 3계층 KV 배치 위에, 각 계층의 접근 패턴과 대역폭 특성에 맞는 차등 압축 정책을 자동으로 적용하면 — HBM: FP8(속도 우선, 낮은 압축), DRAM: FibQuant VQ(공간 효율, 중간 압축), SSD: 고압축 양자화+엔트로피 코딩(최대 절감) — 각 계층의 접근 빈도와 전송 대역폭에 최적화된 압축이 달성되어 기존 단일 압축 정책(모든 계층에 동일 기법 적용) 대비 전체 메모리 절감과 정확도 보존을 동시에 향상시킬 수 있다. 이전 19개 사이클의 모든 C 기법은 단일 압축 정책을 전체 KV에 적용했으며, 계층별 접근 패턴 차이를 압축 정책 결정에 반영한 기법은 없었다. HMAMultiConnectorScheduler(05-17)가 "요청 특성별 압축 커넥터 선택"이라면, 이 기법은 "KV가 저장된 계층별 압축 강도 자동 결정"이라는 완전히 다른 선택 차원이다.
- **접근 방법**:
  1. `KVDriveTierCompressionCodec` 클래스 (`src/cache/kvdrive_tier_compression_codec.py`): `CompressionCodec` 인터페이스 구현
  2. **계층별 압축 정책 매핑**:
     - **HBM 계층** (최빈 접근, 낮은 레이턴시 요구): `FP8` 양자화 — 1.5~2× 압축, 정확도 손실 최소. 기존 기구현 INT8 양자화 기반에 FP8 확장. HBM ↔ DRAM 전송 시 FP8→FP16 즉석 복원(in-flight dequantization).
     - **DRAM 계층** (중간 빈도 접근, 중간 레이턴시): `FibQuantVQCodec`(05-14 기구현) — 4~10× VQ 압축, 랜덤-접근 친화 codebook. DRAM 대역폭(~50GB/s) 활용 극대화. DRAM → HBM 전송 시 VQ 복원(codebook lookup) 오버헤드 < 1ms.
     - **SSD 계층** (드문 접근, 고 레이턴시 허용): `FibQuantVQCodec` 최고 압축 모드 (10×+) + `SPKVWriteTimeCodec`(05-16 기구현)의 write-time 희소화 결합 → 실효 압축 30~50×. SSD 대역폭(~3GB/s)에서 압축이 전송 시간을 단축하는 "압축 오버헤드 < 전송 비용 절감" 구간 최대화.
  3. **계층 이동 시 자동 재압축**:
     - KV가 계층 간 이동할 때 자동으로 목적지 계층의 압축 정책 적용:
       - HBM → DRAM: FP8 압축된 KV를 VQ로 재압축 (`fp8_to_vq_transcode()` 커널 — FP8 복원 → VQ 인코딩 퓨즈드 연산).
       - DRAM → SSD: VQ 압축된 KV에 추가 write-time 희소화 + 엔트로피 코딩 적용 (`vq_to_ssd_compress()` — codebook 기반 추가 스파스화).
       - SSD → DRAM → HBM 복원: 역순 재압축 — SSD에서 읽은 KV를 DRAM 레벨 VQ로 복원 후 HBM FP8로 최종 복원.
     - 재압축 오버헤드: HBM→DRAM 재압축 ≈ 0.5ms/MB, DRAM→SSD ≈ 2ms/MB (SSD 쓰기 지연이 지배적이므로 재압축 오버헤드 무시).
  4. **정확도 보존 메커니즘**:
     - 계층별 압축 강도가 다르므로 가장 중요한 KV(HBM, 고빈도 접근)에 가장 낮은 압축(FP8 ≈ 2×, 정확도 손실 최소)을 적용해 핵심 어텐션 패턴 보존.
     - SSD 계층의 고압축 KV는 어텐션 점수가 낮은 토큰(누적 어텐션 하위 30%)에만 적용되므로 모델 출력에 미치는 영향이 제한적.
     - `NAtHDDROffloadingScheduler(05-16)`의 이론: "정확도는 영구 퇴거 비율에만 의존" → 이 기법에서는 영구 퇴거 없이 SSD 보관이므로 정확도 영향 = SSD 계층 고압축 오차만 (퇴거 비율 0%).
     - 압축 오차 누적 방지: SSD 계층 KV는 `max_ssd_error_per_token` (기본 0.01 L2 norm) 제한으로 압축 강도 상한 설정.
  5. **구현 파일 구조**:
     - `src/cache/kvdrive_tier_compression_codec.py`: `KVDriveTierCompressionCodec` — 계층별 코덱 선택 + 재압축 파이프라인.
     - `src/cache/fp8_vq_transcode.py`: FP8 ↔ VQ 변환 퓨즈드 커널 (PyTorch `torch.compile` 또는 Triton).
     - `src/cache/vq_ssd_codec.py`: VQ + write-time 희소화 + 엔트로피 코딩 결합 SSD 전용 코덱.
     - `configs/kvdrive_tier_compression_policy.yaml`: 계층별 코덱, 압축 강도, 재압축 임계값 외부화.
  6. **이전 C 기법들과의 차별점 정리**:
     - HMAMultiConnectorScheduler + DPAttentionAwareCompressionSelector(05-18 C-1): 요청 특성(요청 유형, GPU 수, DP Attention 상태)에 따라 압축 코덱 선택 → 이 기법: KV가 저장된 계층(HBM/DRAM/SSD)에 따라 압축 강도 자동 결정.
     - PrefillShareAwareCompressionCodec(05-18 C-2): 공유/비공유 KV 분류에 따른 차등 압축 → 이 기법: 계층 위치에 따른 차등 압축.
     - 두 분류 기준(계층 위치 vs. 공유 여부, 요청 유형)은 직교하므로 결합 가능.
- **예상 효과**: 메모리 −75~90% (HBM FP8 2× + DRAM VQ 10× + SSD 30~50× 가중 평균, 계층 분포에 따라), accuracy delta ±0.6% 이내 (HBM FP8 ±0.2%, DRAM VQ ±0.4%, SSD 고압축 오차 per-token 제한), 처리량 +20~35% (계층별 최적 압축으로 각 계층 대역폭 활용 극대화)
- **accuracy-preserving 근거**:
  (1) HBM 계층 FP8: 기존 vLLM v0.21.0 HMA 기본 FP8 양자화 수준 (±0.2% perplexity 변화, 이미 프로덕션 검증됨). 최고 빈도 접근 KV에 가장 낮은 압축 강도 적용으로 핵심 어텐션 패턴 보존.
  (2) DRAM 계층 FibQuant VQ: FibQuantVQCodec(05-14 기구현)의 ±0.5% perplexity 보장 상속 (중간 빈도 접근 KV).
  (3) SSD 계층 고압축: NAtH 이론("정확도는 영구 퇴거 비율에 의존, 오프로딩은 제로 근사 오류") 기반 → SSD 보관은 영구 퇴거가 아니므로 오프로딩 오차만 추가. `max_ssd_error_per_token` 제한으로 SSD 계층 개별 토큰 압축 오차를 0.01 L2 norm 이하로 제한.
  (4) 전체 시스템 정확도: 어텐션 점수 낮은 SSD 계층 토큰의 압축 오차가 어텐션 출력에 미치는 영향 = (SSD 토큰 어텐션 가중치) × (압축 오차) ≈ 작음 (어텐션 점수 낮은 토큰이므로).

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: KVDriveThunderAgentIntegratedStack (A+B+C) — KVDrive 3계층 + ThunderAgent Program 분석 + 계층 차등 압축 완전 통합 스택
- **조합 근거**:
  `KVDriveAttentionPipelineScheduler(A-1)`이 어텐션 점수 기반으로 KV를 HBM/DRAM/SSD에 배치하고, `ThunderAgentStaticSegmentReservationCache(B-1)`이 Program 정적 분석으로 재사용 가능한 세그먼트를 사전 예약해 적절한 계층에 고정하며, `KVDriveTierCompressionCodec(C-1)`이 각 계층의 KV에 자동으로 최적 압축을 적용한다. `ThunderAgentProgramScheduler(A-2)`가 쓰래싱 감지 시 프로그램을 일시 중지해 메모리 압박을 해소하면, 일시 중지된 Program의 KV가 DRAM/SSD로 이동되며 계층 차등 압축이 자동 적용된다.

  **처리 흐름**:
  - **Step 1 (Program 파싱)**: 에이전틱 워크플로우 수신 → `LLMProgramParser`가 KV 의존성 DAG 구성 → `SegmentReservationMap` 생성.
  - **Step 2 (세그먼트 사전 예약)**: `ThunderAgentStaticSegmentReservationCache`가 높은 재사용 확률 세그먼트를 HBM에 예약. `KVDriveAttentionPipelineScheduler`가 나머지 KV를 어텐션 점수 기반으로 3계층에 배치.
  - **Step 3 (압축 적용)**: `KVDriveTierCompressionCodec`이 계층별 자동 압축 — HBM FP8, DRAM VQ, SSD 고압축.
  - **Step 4 (실행 + 쓰래싱 모니터링)**: Program 스텝 실행 중 `ThunderAgentProgramScheduler`의 주기적 모니터가 쓰래싱 감지 시 낮은 우선순위 Program 일시 중지 + KV를 DRAM/SSD로 이동(압축 자동 재적용).
  - **Step 5 (동적 재사용 보완)**: 정적 예약에 없는 세그먼트는 `SegmentedHashCache`/`RelayUShapeLayerSelectiveSegmentCache` 동적 결정으로 fallback.

  이전 사이클의 모든 Cross 아이디어(AMPDPrefillShareNonContiguousStack 05-18, HMAChainedACPipeline 05-17, NAtHRetentionTierDecider 05-16, LookaheadRelaySegmentCache 05-15 등)와 근본적으로 다른 조합 축: 이번 Cross-1은 "3계층 시스템 관리(KVDrive) + 워크플로우 정적 분석(ThunderAgent) + 계층-자동 차등 압축"의 통합으로, 이전 Cross 아이디어들이 "지연 스케줄링 + 팬아웃 라우팅", "HMA 플러그인 체이닝", "이중 신호 티어 결정" 등에 집중했던 것과 설계 축이 완전히 다르다.

- **구체 설계**:
  1. `KVDriveThunderAgentStack` 모듈 (`src/engine/kvdrive_thunder_agent_stack.py`)
  2. 초기화 시 A-1, A-2, B-1, C-1 컴포넌트 등록 + 의존성 연결:
     - A-2의 `SegmentReservationMap` → B-1의 `ThunderAgentStaticSegmentReservationCache`에 주입.
     - A-1의 `KVTierRegistry` → C-1의 `KVDriveTierCompressionCodec`에 계층 위치 조회 인터페이스 제공.
     - A-2의 쓰래싱 이벤트 → A-1의 계층 이동 + C-1의 재압축 자동 트리거.
  3. 모니터링: 각 단계별 지표를 `results/<exp-name>/metrics.json`에 기록 — {Program 파싱 시간, 예약 세그먼트 수 + 예약 히트율, KV 계층 분포(HBM%/DRAM%/SSD%), 쓰래싱 이벤트 횟수, 계층별 압축비, accuracy delta}.
  4. 기존 스택과의 결합: `AMPDPrefillShareNonContiguousStack(05-18 Cross-1 기구현 예정)`과 독립 모듈로 운영하거나, 동일 HMA 커넥터 레지스트리에 등록해 요청 특성에 따라 두 스택 중 하나를 선택.

- **예상 복합 효과**: 처리량 +80~150% (KVDrive 1.74× + ThunderAgent 쓰래싱 방지 + 세그먼트 즉시 히트 복합), 메모리 −80~92% (3계층 차등 압축 + 정적 예약으로 불필요 KV 최소화), accuracy delta ±0.7% 이내 (HBM FP8 정밀도 보존 + 계층별 압축 오차 제어), 비연속 히트율 +30~45% (정적 예약으로 실행 전 세그먼트 HBM 배치)
- **구현 난이도**: high
- **우선순위**: 1

### 아이디어 Cross-2: KVDriveNAtHTierPipelineCompressionAC (A+C) — KVDrive 어텐션 파이프라인 + 계층 차등 압축 A+C 직접 통합
- **조합 근거**: `KVDriveAttentionPipelineScheduler(A-1)`이 어텐션 점수 기반으로 계층 배치를 결정하는 순간, `KVDriveTierCompressionCodec(C-1)`이 목적지 계층에 맞는 압축을 즉시 적용하는 타이트 통합이 가능하다. 이는 A-1의 계층 이동 결정과 C-1의 압축 적용을 단일 연산으로 퓨즈(fuse)해 계층 이동 시 재압축 오버헤드를 최소화한다.

  이전 A+C 조합들(NAtHRetentionTierDecider 05-16, HMAChainedACPipeline 05-17, LazyFetchDPAttnCompressionABPipeline 05-18)과의 차별점:
  - NAtHRetentionTierDecider(05-16): 이중 신호(누적 어텐션 + 전역 리텐션) 기반 티어 결정 → 이 기법: KVDrive 슬라이딩 윈도우 + 어텐션 기반 3계층 배치 + 계층별 자동 압축 퓨즈드.
  - HMAChainedACPipeline(05-17): HMA 플러그인 API + DCP/PCP 이벤트 체이닝 → 이 기법: 계층 이동 결정과 압축 적용의 단일 퓨즈드 연산.
  - LazyFetchDPAttnCompressionABPipeline(05-18): 지연 페치 + DP Attention 환경 인식 압축 → 이 기법: KVDrive 파이프라인 재구성 + SSD 계층 포함 3계층 자동 압축.

- **구체 설계**:
  1. `KVDriveTierPlacementCompressionFuser` 클래스 (`src/cache/kvdrive_tier_fuser.py`)
  2. A-1의 `KVTierRegistry.update_tier(token_id, new_tier)` 호출 시 C-1의 `KVDriveTierCompressionCodec.compress(kv, tier=new_tier)` 자동 호출 — 콜백 등록 패턴.
  3. 퓨즈드 연산: 계층 이동 + 재압축을 단일 `torch.cuda.Stream`에서 실행 → 계층 이동 대역폭과 재압축 GPU 연산을 동시 실행.
  4. 퓨즈드 커널: `fp8_vq_fused_transfer()` — FP8 KV를 DRAM으로 전송하면서 동시에 VQ 인코딩 적용(전송 + 인코딩 오버랩).

- **예상 복합 효과**: 처리량 +40~70% (KVDrive 파이프라인 재구성 + 계층별 압축 대역폭 최적화), 메모리 −75~88% (3계층 차등 압축), accuracy delta ±0.6% 이내
- **구현 난이도**: medium
- **우선순위**: 2

## 이번 사이클 최우선 구현 타겟
**A-1 (KVDriveAttentionPipelineScheduler)** 과 **C-1 (KVDriveTierCompressionCodec)** 및 **Cross-2 (KVDriveNAtHTierPipelineCompressionAC, A+C)** 를 최우선 구현 타겟으로 권장한다.

이유:
1. A-1(KVDriveAttentionPipelineScheduler)은 KVDrive의 핵심 3원칙(어텐션 기반 배치 + 파이프라인 재구성 + 3계층 조율)을 `src/scheduler/`에 구현하는 기법으로, 기구현된 `NAtHDDROffloadingScheduler(05-16)` 대비 SSD 계층 추가 + I/O-컴퓨트 오버랩 파이프라인이라는 명확한 확장이다. 기구현 `torch.cuda.Stream` 비동기 전송 패턴을 재활용하고 `asyncio` SSD I/O를 추가하면 구현 효율이 높다. 처리량 +35~60% (1.74× KVDrive 기반) 효과가 Activity A의 목표 지표(+20% 이상)를 단독으로 달성 가능하다.
2. C-1(KVDriveTierCompressionCodec)은 기구현된 FibQuantVQCodec(05-14), SPKVWriteTimeCodec(05-16), FP8 양자화를 계층별 래퍼로 조합하는 기법으로 새로운 알고리즘 구현 없이 기존 코덱을 재활용할 수 있어 구현 효율이 매우 높다. 계층-자동 차등 압축이라는 새로운 설계 차원을 낮은 구현 비용으로 검증할 수 있다.
3. Cross-2(KVDriveNAtHTierPipelineCompressionAC)는 A-1과 C-1의 자연스러운 A+C 통합으로, 계층 이동 결정 + 압축 적용의 퓨즈드 연산이 별도 구현 없이 콜백 등록 패턴으로 구현 가능하다. 단독 A-1 + 단독 C-1 + 결합 Cross-2의 3방향 비교로 계층별 압축의 추가 효과를 정량화할 수 있다.
4. A-2(ThunderAgentProgramScheduler)는 LLM Program 파서 구현이 필요해 구현 비용이 A-1보다 높지만, 에이전틱 워크플로우 서빙에서 쓰래싱 방지라는 구조적 문제를 해결하는 중요한 기법이다. A-1 완성 후 순차 구현 권장.
5. B-1(ThunderAgentStaticSegmentReservationCache)은 A-2의 `LLMProgramParser`와 `SegmentReservationMap`을 전달받아 구현하므로 A-2 완성 후 순차 구현. Cross-1은 A-1+A-2+B-1+C-1 모두 완성 후 통합 단계에서 구현.

## planner에게 전달할 요청사항
- **A-1(KVDriveAttentionPipelineScheduler) 구현 시**: `src/scheduler/kvdrive_attention_pipeline_scheduler.py` 신규 생성. `BaseScheduler` 상속. `KVTierRegistry` (`src/scheduler/kv_tier_registry.py`) 신규 생성: `Dict[token_id → TierInfo(tier, physical_location, approx_size)]`. 누적 어텐션 점수 EMA(α=0.95): `torch.cuda.Event` 기반 비동기 수집. 슬라이딩 윈도우 `local_window_size=512`, 임계값 `attn_hbm_threshold=0.80 (상위 20%)`, `attn_dram_threshold=0.30 (하위 30%)` YAML 외부화. CUDA Stream 2-way 파이프라인(I/O Stream + Compute Stream). SSD I/O: `asyncio.run_in_executor` + 3 스텝 선행 프리페치. **스케줄링 오버헤드 검증 필수**: TTFT +5% 이내 + 기구현 NAtHDDROffloadingScheduler(05-16) 대비 처리량 비교 + SSD 계층 포함 전후 KV 메모리 사용률 측정. `unnecessary_eviction_rate` (= 0%) 검증 지표 추가.
- **C-1(KVDriveTierCompressionCodec) 구현 시**: `src/cache/kvdrive_tier_compression_codec.py` 신규 생성. `CompressionCodec` 인터페이스 구현. HBM 코덱: 기구현 FP8 양자화 래핑. DRAM 코덱: 기구현 `FibQuantVQCodec(05-14)` 래핑. SSD 코덱: 기구현 `SPKVWriteTimeCodec(05-16)` + `FibQuantVQCodec` 고압축 모드 결합. `fp8_vq_transcode()` 퓨즈드 커널: `torch.compile` 기반 (커스텀 CUDA 없이). 재압축 임계값 `max_ssd_error_per_token=0.01` YAML 외부화. **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 계층 분포(HBM 20%/DRAM 50%/SSD 30% 기준) × 압축 강도 교차 실험 + 계층별 압축 전후 perplexity 비교 + 기구현 FibQuantVQCodec 단독 적용 대비 비교. SSD 계층 고압축 오차가 어텐션 출력에 미치는 영향 측정(낮은 어텐션 가중치 토큰의 압축 오차 기여도 분석) 필수.
- **Cross-2(KVDriveNAtHTierPipelineCompressionAC) 구현 시**: `src/cache/kvdrive_tier_fuser.py` 신규 생성. `KVTierRegistry.update_tier()` 콜백에 `KVDriveTierCompressionCodec.compress()` 등록. `fp8_vq_fused_transfer()` 퓨즈드 전송+압축 커널. **복합 효과 측정 필수**: 단독 A-1 + 단독 C-1 + 결합 Cross-2 처리량/메모리/accuracy 3방향 비교.
- **A-2(ThunderAgentProgramScheduler) 구현 시** (A-1 완성 후): `src/scheduler/thunder_agent_program_scheduler.py` + `src/scheduler/tool_resource_prefetcher.py` 신규 생성. `LLMProgramParser`: JSON/YAML 워크플로우 명세 파싱 + SimHash 64-bit 컨텍스트 해시. `LLMProgram.kv_dependency_graph`: NetworkX DAG 또는 단순 Dict. 쓰래싱 임계값 `thrash_threshold=0.90`, `evict_rate_threshold=0.15` YAML 외부화. 주기적 모니터 `monitor_interval_ms=50`. Program 일시 중지: A-1 `KVTierRegistry`를 통한 DRAM 강등 + 전역 대기 큐 이동. **처리량 검증 필수**: 에이전틱 워크플로우 시뮬레이션(10~50 스텝 Program) + 쓰래싱 이벤트 횟수 측정 + 기구현 PBKVAgentSegmentPreservationScheduler(05-13) 대비 처리량 비교.
- **B-1(ThunderAgentStaticSegmentReservationCache) 구현 시** (A-2 완성 후): `src/cache/thunder_agent_static_segment_reservation.py` 신규 생성. `CacheStore` 인터페이스 상속. `SegmentReservationMap` 자료구조. `reserve_segment(segment_id, min_tier, until_step_id)` 인터페이스. `max_reservation_budget=0.20` (HBM의 20%) YAML 외부화. **비연속 히트율 검증 필수**: 정적 예약 ON/OFF 비교 + 예약 히트율(`reservation_hit_rate = 예약 세그먼트 중 실제 히트 비율`) 측정 + 기구현 AMPDAdapShotLazyLoadPipeline(05-18 B-1) 대비 TTFT 비교.
- **목표 지표 중 이번 사이클에서 측정할 지표**:
  - Inference Throughput (A-1 KVDrive 파이프라인 재구성 + Cross-2 A+C 복합 효과 +35% 이상 목표)
  - KV Cache Memory Reduction (C-1 계층 차등 압축 −75% 이상 목표)
  - Cache Eviction Rate (A-1 SSD 보관으로 영구 퇴거 ≈ 0% 검증)
  - Compression Accuracy Delta (필수 ±1% 이내 — C-1 계층별 압축 강도별 독립 검증)
  - Scheduling Overhead (A-1 I/O-컴퓨트 오버랩 효율 측정 — 오버랩 없을 때 대비 스톨 감소량)
- **Activity C 구현 시 공통 필수 사항**: 모든 C 기법은 `src/metrics/` 내 perplexity 측정 코드 포함 + 압축 전후 perplexity를 `results/<exp-name>/metrics.json`에 기록. C-1은 계층별(HBM/DRAM/SSD) perplexity 변화를 별도로 측정하고 계층 이동 시 재압축 전후 perplexity 비교 필수.
