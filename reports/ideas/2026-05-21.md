## Activity A 아이디어

### 아이디어 A-1: KVServeBayesianCompressionBudgetAwareScheduler (KVServe Bayesian 압축 예산 인식 스케줄러)
- **가설**: KVServe(2605.13734)의 Bayesian Profiling Engine이 압축 전략 공간(양자화 × 퇴거 × 근사)을 탐색해 3차원 Pareto 후보 집합(지연 × 정확도 × 대역폭)을 생성한다는 원리를, P/D 분리 환경의 KV 전송 병목 문제에 적용하면, 스케줄러가 요청 큐 상태(온라인/오프라인 비율, SLO 예산)와 인터커넥트 대역폭을 실시간 입력으로 받아 Pareto 프런트에서 최적 압축 전략을 자동 선택하는 "압축 예산 인식 스케줄러"를 구성할 수 있으며, 기존 단일 정적 압축 전략 대비 P/D 전송 병목을 50~70% 줄이고 처리량을 향상시킬 수 있다.
- **접근 방법**:
  1. `KVServeBayesianScheduler` 클래스 (`src/scheduler/kvserve_bayesian_scheduler.py`): `BaseScheduler` 상속
  2. **Bayesian Profiling Engine 구현**:
     - 오프라인 탐색 단계: 압축 전략 공간 = {양자화: FP8/INT8/INT4} × {퇴거: LRU/H2O/SnapKV} × {근사: None/low-rank} 총 18개 후보 전략. 각 전략에 대해 Bayesian 최적화(Gaussian Process + Expected Improvement)로 탐색 순서를 결정해 50× 오버헤드 감소.
     - 보정 데이터(100 요청 시뮬레이션)에서 각 전략의 (지연, accuracy delta, 전송 대역폭) 3차원 지표를 측정.
     - Pareto 프런트 계산: 3차원 공간에서 지배당하지 않는 전략 집합 `P_pareto`를 추출.
     - 결과를 `configs/kvserve_pareto_front.yaml`에 저장.
  3. **런타임 Pareto 프런트 적응**:
     - 요청 큐 상태 모니터링: `online_ratio = online_requests / total_requests`, `slo_headroom = slo_deadline - current_latency_p99`.
     - 인터커넥트 대역폭 감지: `available_bandwidth_gbps = measure_rdma_bandwidth()` (주기적 1초마다).
     - Pareto 프런트에서 현재 상태에 가장 가까운 전략 선택: `argmin_{s ∈ P_pareto} weighted_distance(s, current_state)`, 가중치 = SLO 긴급도 × 대역폭 압박.
     - 선택된 전략을 `CompressionStrategyCache`에 캐싱해 동일 상태 재진입 시 탐색 없이 재사용.
  4. **온라인 Pareto 프런트 업데이트**:
     - 런타임 실측 (지연, accuracy delta, 대역폭 사용량)으로 오프라인 Pareto 프런트를 점진적 갱신 (EMA 기반).
     - 환경 드리프트(대역폭 변화, 워크로드 변화) 시 GP 사후 분포를 업데이트해 Pareto 프런트 재계산. 재계산 주기: `pareto_update_interval_s` (기본 60초) YAML 외부화.
  5. **이전 A 기법과의 차별점**:
     - KVServeBayesianScheduler: Bayesian 기반 압축 전략 공간 탐색 + 3D Pareto 후보 집합 + 런타임 상태 기반 Pareto 프런트 이동.
     - 기존 A-1 KVDriveAttentionPipelineScheduler(05-19): 어텐션 점수 기반 3계층 물리적 KV 배치 + I/O-컴퓨트 오버랩 파이프라인 재구성.
     - 기존 A-1 HMAMultiConnectorScheduler(05-17): 요청 특성별 압축 커넥터 선택(O(1) 딕셔너리 룩업, 고정 전략 집합).
     - 기존 A-1 CONCURCongestionBasedAgentAdmissionScheduler(05-20): KV 풀 점유율 기반 혼잡 어드미션 제어.
     - KVServeBayesianScheduler는 이들 모두와 다르게 "전략 공간을 Bayesian 탐색으로 자동 축소 + 3D Pareto 프런트 런타임 적응"이라는 메타-최적화 계층을 도입한다.
- **대상 환경**: 멀티 노드 (P/D 분리 환경 필수, 단일 노드에서도 적용 가능)
- **멀티 노드 고려사항**: P 노드와 D 노드 간 KV 전송 대역폭(InfiniBand/RDMA)을 Pareto 탐색의 세 번째 축(대역폭 절감)으로 직접 포함. 오프라인 Pareto 프런트는 클러스터 토폴로지별로 1회 계산. 런타임 적응은 마스터 스케줄러에서 집중 실행 후 각 노드에 선택된 전략을 전파.
- **예상 효과**: 처리량 +20~35% (KVServe 원논문 기반 P/D 전송 병목 해소), KV 전송 대역폭 절감 −40~60% (Pareto 최적 압축 전략 선택), Scheduling overhead TTFT +4% 이내 (오프라인 Pareto 계산 완료 후 런타임 선택은 O(|P_pareto|) 거리 계산 = < 0.2ms)
- **구현 난이도**: medium
- **우선순위**: 2

## Activity B 아이디어

### 아이디어 B-1: CompactAttentionBlockUnionNonContiguousReuseIndex (CompactAttention Block-Union 비연속 세그먼트 재사용 인덱스)
- **가설**: 비연속 KV 세그먼트 재사용 시 물리적으로 분산된 PagedAttention 블록을 연속 KV 텐서로 집결하는 비용(memcpy 오버헤드)이 주요 병목이 되는 문제를, CompactAttention(2605.16839)의 Block-Union KV Selection 아이디어 — 2D 블록-스파스 마스크를 GQA-aware per-group KV 블록 테이블로 변환해 표준 어텐션 커널을 재사용 — 를 비연속 세그먼트 재사용에 적용하면, 비연속 PA 블록들을 "block-union 인덱스"로 집약한 뒤 sparse → dense 변환 없이 직접 표준 어텐션 커널에 투입할 수 있으며, 기존 비연속 재사용 시 필요한 메모리 복사 비용을 제거하고 GQA 모델에서 헤드 간 KV 공유를 비연속 세그먼트에도 적용할 수 있다.
- **접근 방법**:
  1. `BlockUnionNonContiguousReuseIndex` 클래스 (`src/cache/block_union_noncontiguous_index.py`): `CacheStore` 인터페이스 상속
  2. **Block-Union 인덱스 자료구조**:
     - 비연속 세그먼트 집합 `{seg_1, seg_2, ..., seg_k}`를 PA 블록 단위로 표현: `SegmentBlockUnion = List[List[BlockPtr]]` — 세그먼트별 PA 블록 포인터 리스트.
     - GQA-aware per-group 블록 테이블 변환: GQA 그룹 내 모든 헤드가 공유하는 K/V 블록 포인터를 `block_table[group_id][position] = block_ptr` 형태로 통합.
     - Q-block union: 현재 쿼리 블록들을 집합 연산으로 union해 비연속 세그먼트 어디에서도 참조 가능한 통합 쿼리 블록 집합 구성.
     - Intra-group union: 동일 GQA 그룹 내 헤드 간 블록 포인터 공유를 명시적 union으로 표현해 중복 저장 제거.
  3. **표준 커널 재사용을 위한 변환 파이프라인**:
     - `build_block_union_table(segment_ids: List[str]) → BlockUnionTable`: 비연속 세그먼트의 PA 블록 포인터를 GQA-aware 블록 테이블로 변환. 변환 비용: O(k × seg_len / block_size) 포인터 연산 (memcpy 없음).
     - `BlockUnionTable`을 표준 PagedAttention 커널의 `block_tables` 인자로 직접 주입. 커스텀 스파스 커널 불필요.
     - 변환된 블록 테이블은 현재 배치 내에서만 유효 (배치 간 재사용 없음 — 메모리 생명주기 명확화).
  4. **비연속 히트 시 처리 경로**:
     - `get_block_union(query_tokens) → BlockUnionTable`: 쿼리 토큰 집합에 대해 히트되는 세그먼트들의 block-union 테이블 즉시 생성.
     - 히트된 세그먼트의 위치 독립적 접근: block-union 인덱스는 세그먼트의 원본 위치(position)를 보존하므로 RoPE 재인코딩 없이 직접 사용 가능 (CompactAttention의 표준 커널 재사용 원칙 상속).
     - RoPE 재인코딩 필요 여부 판단: 세그먼트 원본 위치와 현재 컨텍스트 위치가 일치하면 재인코딩 불필요, 불일치하면 AdapShot RoPE 재인코딩(05-18 B-1에서 기구현) 적용.
  5. **기존 `src/cache/segmented.py`와 통합**:
     - `SegmentedHashCache`의 `get(segment_id)` 경로에 block-union 변환을 선택적으로 추가: `use_block_union=True` 플래그.
     - 기존 memcpy 기반 경로(fallback)와 block-union 경로를 A/B 비교 실험으로 오버헤드 측정.
  6. **이전 B 기법과의 차별점**:
     - AMPDAdapShotLazyLoadPipeline(05-18 B-1): 세그먼트 로드-재인코딩 파이프라인 비동기 오버랩 (데이터 이동 타이밍 최적화).
     - ThunderAgentStaticSegmentReservationCache(05-19 B-1): Program 정적 분석 기반 세그먼트 사전 예약 (실행 전 계획 수립).
     - CongestionAwareNonContiguousSegmentEvictionPolicy(05-20 B-1): 혼잡도 신호 기반 동적 퇴거율 제어 (메모리 관리 정책).
     - BlockUnionNonContiguousReuseIndex: 비연속 PA 블록을 GQA-aware 블록 테이블로 변환해 memcpy 없이 표준 커널에 투입 (물리적 표현 변환 방식 자체를 변경). 이전 어느 B 기법도 block-union 추상화를 도입하지 않았다.
- **예상 효과**: 비연속 히트율 +25~40% (memcpy 병목 제거로 히트 처리 가능한 요청 비율 증가), 비연속 재사용 레이턴시 −40~60% (block-union 변환 O(포인터 연산) vs. 기존 memcpy O(데이터 복사)), TTFT −20~35% (재사용 처리 병목 해소)
- **구현 난이도**: medium
- **우선순위**: 1

## Activity C 아이디어

### 아이디어 C-1: CompactAttentionBlockUnionKVSelectionCodec (CompactAttention Block-Union KV 선택 압축 코덱)
- **가설**: CompactAttention(2605.16839)이 청크드 프리필 환경에서 2D 블록-스파스 어텐션 마스크를 GQA-aware per-group KV 블록 테이블로 변환해 표준 어텐션 커널을 재사용한다는 원리를, Activity C의 KV 캐시 압축에 적용하면, 스파스 어텐션 마스크를 직접 커스텀 스파스 커널로 처리하지 않고 KV 선택 블록 테이블로 변환해 "중요 KV 블록만 표준 커널로 처리"하는 accuracy-preserving 압축 패턴을 구현할 수 있으며, LLaMA-3.1-8B-Instruct 기준 128K 컨텍스트에서 최대 2.72× 어텐션 속도 향상을 달성하면서 RULER 벤치마크 밀집 어텐션 수준의 정확도를 유지할 수 있다.
- **접근 방법**:
  1. `CompactAttentionBlockUnionCodec` 클래스 (`src/cache/compact_attention_block_union_codec.py`): `CompressionCodec` 인터페이스 구현
  2. **2D 블록-스파스 마스크 → KV 선택 블록 테이블 변환**:
     - 입력: 청크드 프리필의 각 쿼리 청크에 대해 어텐션 중요도 마스크 `M[q_chunk][k_block]` (2D 블록-스파스).
     - Q-block union 연산: 쿼리 청크 내 모든 쿼리 토큰이 참조하는 KV 블록의 합집합(union)을 계산. `selected_blocks = union_{q ∈ q_chunk} {k_block : M[q][k_block] = 1}`.
     - Intra-group union: GQA 그룹 내 헤드별로 Q-block union 결과를 그룹 단위로 통합. `group_blocks[group_id] = union_{h ∈ group} selected_blocks[h]`.
     - 출력: `KVSelectionBlockTable[group_id][position] = block_ptr` — 선택된 KV 블록만 포함하는 희소 블록 테이블.
  3. **표준 어텐션 커널 재사용**:
     - `KVSelectionBlockTable`을 표준 PagedAttention 커널의 `block_tables` 인자로 직접 주입.
     - 커스텀 스파스 커널 불필요: 구현 복잡도 낮음, 기존 최적화된 표준 커널 성능 그대로 활용.
     - KV 블록 선택률: `selection_ratio = |selected_blocks| / |total_blocks|` (기본 목표 30~50%).
  4. **중요도 마스크 생성 방법**:
     - **방법 1 (경량 오프라인 프로파일)**: 보정 데이터(100~200 시퀀스)에서 어텐션 점수 상위 블록 집합의 안정성을 레이어·헤드별로 프로파일링. 안정적인 패턴은 `block_importance_profile.yaml`에 저장.
     - **방법 2 (온라인 빠른 추정)**: 첫 번째 쿼리 청크의 어텐션 점수로 전체 중요도 마스크를 추정 (청크드 프리필의 순차 처리 활용). 추정 오버헤드: 첫 청크 어텐션 계산 시 추가 O(q_chunk × k_total) 연산.
     - 두 방법을 `mask_source: ["profile", "online", "hybrid"]` YAML 파라미터로 선택 가능.
  5. **청크드 프리필과의 통합**:
     - 청크드 프리필 각 단계에서 청크가 처리될 때마다 중요도 마스크를 점진적으로 구축.
     - 마지막 청크 처리 완료 시 최종 `KVSelectionBlockTable` 확정 → 디코딩 단계에 직접 사용.
     - 청크 크기 `chunk_size` (기본 2048 토큰)와 선택률 `kv_selection_ratio` (기본 0.40) YAML 외부화.
  6. **이전 C 기법과의 차별점**:
     - SP-KV write-time codec(05-16): write-time에 KV 기록 결정 (post-hoc이 아닌 write-time 결정, 단일 토큰 단위).
     - SpecAttn verification-guided codec(05-20 C-1): 자기-추측 디코딩 검증 단계 로짓 활용 (디코딩 단계 특화).
     - GlobalRetentionGate(05-16): 전역 경쟁 퇴거 (크로스-레이어 retention score 기반).
     - ABSparseCodec(이전 사이클): block-sparse 어텐션 직접 적용 (커스텀 스파스 커널 필요).
     - CompactAttentionBlockUnionCodec: 2D 블록-스파스 마스크를 GQA-aware 블록 테이블로 변환해 커스텀 스파스 커널 없이 표준 커널 재사용 (청크드 프리필 특화, 구현 복잡도 최소화). ABSparseCodec과 달리 변환 과정에서 표준 커널 재사용이 핵심이다.
- **예상 효과**: 메모리 −35~55% (KV 블록 선택률 30~50% 기준), accuracy delta ±0.5% (CompactAttention 원논문 RULER 벤치마크 기반), 어텐션 속도 +100~170% (CompactAttention 원논문 최대 2.72× 기반)
- **accuracy-preserving 근거**:
  (1) CompactAttention 원논문(2605.16839)에서 Block-Union KV Selection이 LLaMA-3.1-8B-Instruct 기준 128K 컨텍스트 RULER 벤치마크에서 밀집 어텐션에 근접한 정확도를 실험적으로 보임 — accuracy-preserving 특성이 이미 검증됨.
  (2) Q-block union 연산이 쿼리 청크 내 모든 쿼리 토큰이 참조하는 KV 블록의 합집합을 취하므로 쿼리 토큰이 필요로 하는 모든 KV가 포함됨 (false negative 없음). 선택 초과(false positive)만 발생하므로 보수적 선택이 보장됨.
  (3) Intra-group union으로 GQA 그룹 내 헤드들의 실제 참조 패턴을 모두 포함해 GQA 특성으로 인한 정보 손실 방지.
  (4) `kv_selection_ratio = 0.40` (상위 40% KV 블록 유지) 기본 설정: 이전 SpecAttn의 70~85% KV 유지보다 공격적이나, block-level union 보수성으로 중요 KV 누락 방지. 레이어별 동적 조정 가능.
  검증: WikiText-2 perplexity ±1% + RULER 벤치마크(128K 컨텍스트) + LongBench 8개 서브태스크 + 선택률(20%/30%/40%/50%) 별 정확도 곡선 + 청크 크기별(512/1024/2048) 마스크 품질 비교.
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 C-2: KVServeBayesianMixedStrategyCompressionSelector (KVServe Bayesian 혼합 전략 압축 선택기)
- **가설**: KVServe(2605.13734)의 Bayesian Profiling Engine이 단일 압축 기법에 의존하지 않고 "양자화 × 퇴거 × 근사" 혼합 전략 공간을 자동 탐색해 3차원 Pareto 후보 집합을 생성한다는 원리를, Activity C의 독립 압축 코덱 선택 문제에 적용하면, 기존 기구현 코덱들(SpecAttnVerificationGuidedKVSparseCodec, FibQuantVQCodec, GlobalRetentionGateEvictionCodec, SPKVWriteTimeCodec 등)의 혼합 조합 공간을 Bayesian 탐색으로 자동 최적화하는 "메타-압축 전략 선택기"가 가능하며, 단일 기법 적용 대비 accuracy delta를 ±1% 이내로 유지하면서 메모리 절감율을 극대화할 수 있다.
- **접근 방법**:
  1. `KVServeBayesianMixedCompressionSelector` 클래스 (`src/cache/kvserve_bayesian_compression_selector.py`): `CompressionCodec` 인터페이스 구현
  2. **혼합 전략 공간 정의**:
     - 압축 전략 공간: `{codec_1, codec_2, codec_3}` 조합 (기구현 코덱 중 최대 3개 선택), 각 코덱의 강도 파라미터 포함.
     - 구체적 공간: {양자화: FP8/INT8/INT4} × {퇴거: None/GlobalRetention/SpecAttn} × {희소화: None/SPKVWriteTime/BlockUnion(C-1)} = 27개 조합.
     - 각 조합의 "압축 파이프라인 순서"도 파라미터화: 양자화 먼저 vs. 퇴거 먼저 vs. 희소화 먼저.
  3. **Bayesian 최적화 탐색**:
     - 목적 함수: `maximize memory_reduction - λ × accuracy_delta` (λ는 정확도 예산, 기본 λ=10 → ±1% 이내 목표).
     - Gaussian Process 서로게이트 모델로 27개 조합 중 최소 탐색 횟수(기본 15회)로 Pareto 프런트 추출.
     - 탐색 오버헤드: KVServe 원논문 기준 기존 그리드 탐색 대비 50× 감소 (보정 데이터 100 시퀀스 기준 수 분 내 완료).
  4. **런타임 전략 캐싱 및 적용**:
     - 오프라인 탐색 완료 후 최적 혼합 전략 `(codec_sequence, params)`를 `configs/kvserve_optimal_codec_mix.yaml`에 저장.
     - 런타임: 저장된 전략 시퀀스를 파이프라인으로 실행. 각 기구현 코덱의 `compress(kv, **params)` 순차 호출.
     - 환경 변화(컨텍스트 길이 분포 이동, GPU 메모리 압박 증가) 감지 시 재탐색 트리거: `drift_threshold` YAML 외부화.
  5. **이전 C 기법과의 차별점**:
     - HMAMultiConnectorScheduler(05-17 A-1): 요청 특성별 단일 코덱 선택 (O(1) 딕셔너리 룩업, 고정 규칙 기반).
     - DPAttentionAwareCompressionSelector(05-18 C-1): DP Attention 환경 상태 기반 정책 선택 (단일 코덱, 환경 감지).

- **예상 효과**: 메모리 −45~70% (혼합 전략 시너지로 단일 코덱 최대 압축률 초과 가능), accuracy delta ±0.7% 이내 (Bayesian 목적 함수에 정확도 페널티 명시 포함), 탐색 오버헤드 −95% (GP 서로게이트 50× 감소)

### 아이디어 Cross-1: BlockUnionNonContiguousCompressionPipeline (B+C) — Block-Union 비연속 재사용 + KV 선택 압축 공동 파이프라인
- **조합 근거**: `BlockUnionNonContiguousReuseIndex(B-1)`이 비연속 PA 블록을 GQA-aware 블록 테이블로 변환하는 표현 계층을 제공하고, `CompactAttentionBlockUnionCodec(C-1)`이 동일한 블록 테이블 표현을 KV 선택 압축의 입력으로 사용한다. 두 기법이 공통 `KVSelectionBlockTable` 자료구조를 공유하므로, 비연속 세그먼트 재사용(B)과 KV 선택 압축(C)을 단일 블록 테이블 구성 단계에서 통합할 수 있다. "비연속 세그먼트 히트 → block-union 인덱스 구성 → KV 선택(중요 블록만) → 표준 커널 투입"이라는 4단 파이프라인이 B+C 공동 설계의 자연스러운 구현 경로다.
- **구체 설계**:
  1. `BlockUnionBCPipeline` 모듈 (`src/cache/block_union_bc_pipeline.py`)
  2. **통합 처리 흐름**:
     - **Step 1 (세그먼트 히트 감지)**: `BlockUnionNonContiguousReuseIndex.get_block_union(query_tokens)` 호출 → 히트된 비연속 세그먼트들의 PA 블록 포인터 목록 획득.
     - **Step 2 (중요도 마스크 생성)**: `CompactAttentionBlockUnionCodec`의 마스크 생성기가 히트된 블록들에 대해 중요도 마스크 `M[q_chunk][k_block]` 계산. 히트된 블록만 대상이므로 마스크 크기가 전체 컨텍스트 대비 감소.
     - **Step 3 (통합 KV 선택 블록 테이블 구성)**: B-1의 block-union 포인터 집합과 C-1의 중요도 마스크를 결합해 "비연속 히트 블록 중 중요 블록만" 포함하는 최종 `KVSelectionBlockTable` 생성. 이 단계가 B와 C의 감소 효과를 결합: 비연속 히트 비율 × KV 선택률.
     - **Step 4 (표준 커널 투입)**: 최종 블록 테이블을 표준 PagedAttention 커널에 투입. 커스텀 스파스 커널 불필요.
  3. **B+C 감소 효과 결합**:
     - B-1 단독: 비연속 히트 세그먼트의 블록 수 활용 (히트율에 비례).
     - C-1 단독: 전체 KV 블록 중 선택률(기본 40%) 유지.
     - B+C 결합: 비연속 히트 블록 × 선택률 = 곱연산적 감소. 예: 히트율 50% × 선택률 40% → 전체 KV 블록의 20%만 사용 (80% 절감).
  4. **RoPE 재인코딩 통합**: 비연속 세그먼트의 위치 불일치 시 AdapShot RoPE 재인코딩(05-18 B-1 기구현)을 Step 1과 Step 2 사이에 삽입. 재인코딩 후 블록 포인터 갱신.
  5. **모니터링**: `{비연속 히트율, KV 선택률, B+C 결합 감소율, accuracy delta, TTFT}`를 `results/<exp-name>/metrics.json`에 기록.
- **예상 복합 효과**: 처리량 +30~50% (비연속 재사용 처리 병목 해소 + KV 선택 압축 어텐션 속도 향상 복합), 메모리 −50~70% (비연속 히트 블록 + KV 선택 곱연산 감소), accuracy delta ±0.6% 이내 (CompactAttention 원논문 정확도 보존 상속 + 비연속 재사용 오차 최소)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 Cross-2: BayesianKVServeBlockUnionIntegratedStack (A+B+C) — Bayesian 압축 예산 스케줄러 + Block-Union 비연속 재사용 + KV 선택 압축 3-Way 통합
- **조합 근거**: `KVServeBayesianScheduler(A-1)`이 P/D 분리 환경에서 Bayesian Pareto 프런트로 최적 압축 전략을 선택하고, `BlockUnionNonContiguousReuseIndex(B-1)`이 비연속 PA 블록을 GQA-aware 블록 테이블로 변환하며, `CompactAttentionBlockUnionCodec(C-1)`이 표준 커널 재사용으로 accuracy-preserving KV 선택 압축을 적용한다. KVServe의 Bayesian Profiling이 B-1과 C-1의 파라미터(선택률, 청크 크기)를 통합 최적화 공간에 포함하면, P/D 분리 환경에서 비연속 KV를 block-union으로 선택·압축해 D 노드로 전송하는 전 과정이 Bayesian 최적화 하에 놓이게 된다. 특히 P 노드가 비연속 세그먼트를 block-union으로 선택·압축하여 D 노드로 전송하는 시나리오는 세 Activity의 자연스러운 교차점이다.
- **구체 설계**:
  1. `BayesianBlockUnionStack` 모듈 (`src/engine/bayesian_block_union_stack.py`)
  2. **통합 파라미터 공간**:
     - A: P/D 전송 압축 전략 (양자화 × 퇴거 × 희소화)
     - B: block-union 세그먼트 히트 임계값(`reuse_threshold`), RoPE 재인코딩 여부
     - C: KV 선택률(`kv_selection_ratio`), 마스크 생성 방법(`mask_source`)
     - Bayesian 탐색 공간에 A+B+C 파라미터를 모두 포함해 통합 최적화.
  3. **P/D 분리 환경 처리 흐름**:
     - **P 노드**: 비연속 세그먼트 히트 감지(B-1) → block-union 인덱스 구성 → KV 선택 압축(C-1) → 압축된 블록 테이블을 D 노드로 전송 (KVServe 대역폭 인식 전송).
     - **D 노드**: 압축된 블록 테이블 수신 → 표준 어텐션 커널 투입 → 디코딩 실행.
     - P→D 전송 비용: block-union 선택으로 전송 블록 수 감소 + C-1 압축으로 블록 크기 감소 = 곱연산 전송량 감소.
  4. **Bayesian 통합 탐색**:
     - KVServeBayesianScheduler(A-1)의 GP 서로게이트 모델이 A+B+C 통합 파라미터 공간을 탐색.
     - 목적 함수: `maximize (throughput × memory_reduction) - λ × accuracy_delta - μ × transfer_cost`.
     - λ=10 (정확도), μ=5 (전송 비용) 가중치 YAML 외부화.
  5. **기존 스택과의 비교**: Cross-1 BlockUnionBCPipeline과 독립 모듈로 운영. Cross-1이 B+C 통합에 집중한다면, Cross-2는 A를 추가해 P/D 분리 환경에서의 전송 최적화를 포함한다.
- **예상 복합 효과**: 처리량 +40~60% (Bayesian 최적 전략 + block-union 병목 제거 + KV 선택 압축 복합), 메모리 −55~75% (P→D 전송량 감소 + KV 선택 압축 곱연산), accuracy delta ±0.7% 이내 (Bayesian 정확도 제약 + CompactAttention 정확도 보존)
- **구현 난이도**: high
- **우선순위**: 2

**최우선: Cross-1 (BlockUnionNonContiguousCompressionPipeline, B+C)**

이유:
- B-1(BlockUnionNonContiguousReuseIndex)과 C-1(CompactAttentionBlockUnionCodec) 두 아이디어가 공통 자료구조(`KVSelectionBlockTable`)를 공유하므로 단독 구현보다 통합 구현이 효율적이다.
- B-1은 기존 `src/cache/segmented.py`에 block-union 변환을 추가하는 방식으로 구현 가능하며 기존 경로와의 A/B 비교가 용이하다.
- C-1의 표준 커널 재사용 원칙으로 구현 복잡도가 낮다(커스텀 스파스 커널 불필요).
- 두 기법의 곱연산 감소 효과(비연속 히트율 × KV 선택률)가 목표 지표(메모리 -30%, 처리량 +20%) 달성에 가장 가까운 경로다.

**2순위: C-1 단독 구현** (CompactAttentionBlockUnionCodec을 먼저 독립 구현해 KV 선택 압축 효과 측정)

**3순위: B-1 단독 구현** (BlockUnionNonContiguousReuseIndex로 memcpy 병목 제거 효과 독립 측정)

## planner에게 전달할 요청사항

1. **Activity C 구현 시 accuracy-preserving 검증 방법 명시 필수**:
   - C-1 (CompactAttentionBlockUnionCodec): WikiText-2 perplexity ±1% 이내 + RULER 벤치마크(128K 컨텍스트, 밀집 어텐션 대비) + LongBench 8개 서브태스크 + 선택률(20%/30%/40%/50%) 별 정확도 곡선 필수.
   - C-2 (KVServeBayesianMixedCompressionSelector): Bayesian 탐색 과정의 각 후보 조합에서 WikiText-2 perplexity 실측, ±1% 위반 조합 자동 제외 로직 포함.

2. **이번 사이클 측정 지표**:
   - Inference Throughput (tokens/sec): Cross-1 B+C 결합으로 처리량 +20% 이상 목표
   - KV Cache Memory Reduction: B+C 곱연산 감소로 -30% 이상 목표
   - Non-Contiguous Cache Hit Rate: B-1 block-union 변환 후 히트율 변화 측정 (memcpy 병목 제거 전후 비교)
   - Compression Accuracy Delta: C-1 ±1% 이내 (필수, RULER 벤치마크 포함)
   - Scheduling Overhead: A-1 Bayesian 탐색 오프라인 오버헤드 측정 (런타임 TTFT +5% 이내)

3. **구현 우선순위**: Cross-1 → C-1 → B-1 → A-1 → C-2 → Cross-2 순서로 구현 계획 수립.

4. **B-1 구현 시**: `src/cache/block_union_noncontiguous_index.py` 신규 생성. `CacheStore` 인터페이스 상속. `build_block_union_table(segment_ids) → BlockUnionTable` 인터페이스. GQA-aware per-group 블록 테이블 자료구조 (`Dict[group_id, Dict[position, BlockPtr]]`). 기존 `SegmentedHashCache.get()` 경로에 `use_block_union=True` 플래그 추가. **비연속 히트율 및 레이턴시 검증 필수**: block-union 경로 vs. memcpy 경로 A/B 비교, 세그먼트 수 k에 따른 스케일링 측정.

5. **C-1 구현 시**: `src/cache/compact_attention_block_union_codec.py` 신규 생성. `CompressionCodec` 인터페이스 구현. Q-block union 연산 구현 (`torch.unique` 기반 블록 포인터 합집합). Intra-group union 구현. `mask_source: ["profile", "online", "hybrid"]` YAML 파라미터. 청크 크기 `chunk_size=2048`, 선택률 `kv_selection_ratio=0.40` YAML 외부화. **accuracy-preserving 검증 필수**: 위 1항의 검증 방법 적용.

6. **Activity C 구현 시 공통 필수 사항**: 모든 C 기법은 `src/metrics/` 내 perplexity 측정 코드 포함 + 압축 전후 perplexity를 `results/<exp-name>/metrics.json`에 기록. C-1은 RULER 벤치마크(128K 컨텍스트) 추가 측정 필수.
