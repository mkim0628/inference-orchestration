# 트렌드 리포트 — 2026-05-19

## 전체 요약

1. **Activity A (스케줄링)**: 오늘의 핵심 신규 항목은 두 편이다. KVDrive(2605.18071, 2026-05-18 제출)는 GPU HBM·호스트 DRAM·SSD 3계층을 어텐션 행동 기반으로 동적 조율하는 최초의 "홀리스틱(holistic)" KV 캐시 관리 시스템으로, 최신 기준선 대비 1.74× 처리량 향상을 달성했다. ThunderAgent(2602.13692)는 에이전틱 워크플로우를 LLM Program 추상화로 모델링해 KV 캐시·CPU·외부 도구 자원을 통합 스케줄링하는 프로그램-인식(program-aware) 추론 시스템으로, vLLM 대비 1.5~3.6× 처리량, RL 롤아웃 환경에서 1.8~3.9× 향상을 보였다.

2. **Activity B (비연속 재사용)**: 이번 주(2026-05-12 ~ 2026-05-19)에 Activity B 범주의 완전히 새로운 논문은 확인되지 않았다. 현재 최전선(RelayCaching, ICaRus, QKVShare, KV-Fold, KVM, AdapShot, Louver)이 유효하다. IBM Research/Red Hat의 KubeCon EU 2026 llm-d 튜토리얼(이미 보고)이 KV 캐시 인식 라우팅의 쿠버네티스 레벨 표준화를 계속 진행 중임을 확인했다.

3. **Activity C (압축)**: 이번 주 Activity C에 해당하는 완전히 새로운 논문도 발견되지 않았다. 기존 최전선(SP-KV, Make Each Token Count, LookaheadKV, FibQuant, AB-Sparse, Fluxion)이 유효하다. KVDrive(Activity A 등재)의 어텐션-기반 캐시 관리가 슬라이딩 윈도우로 핵심 항목을 GPU HBM에 유지하는 방식이 Activity C 관점에서도 주목할 만한 "암묵적 토큰 퇴거 + 계층 이동" 전략이다.

4. **A+C 시너지**: KVDrive의 HBM↔DRAM↔SSD 3계층 파이프라인 스케줄링과 Activity C의 FP8/SP-KV 압축이 결합되면, 각 계층에 저장되는 KV를 계층별 다른 정밀도로 압축("HBM: FP8, DRAM: FP4/VQ, SSD: 양자화+엔트로피 코딩")하는 계층-차별 압축 스케줄러가 가능하다.

5. **전반적 관찰**: NeurIPS 2026 제출 마감(6월)을 앞둔 소강 상태가 2주째 지속되고 있다. 신규 논문 수는 적으나, KVDrive처럼 전일 제출된 시스템 논문이 등장해 "이론 논문 소강 + 시스템 구현 지속"의 교차 패턴이 유지되고 있다. 다음 주 NeurIPS 투고 마감 이후 arXiv 제출량이 급증할 가능성이 있다.

## Activity A — KV Cache-aware Scheduling / Orchestration

### 새로운 논문·구현체

#### 1. [KVDrive: A Holistic Multi-Tier KV Cache Management System for Long-Context LLM Inference](https://arxiv.org/abs/2605.18071)
- **출처**: arXiv 2605.18071 (2026-05-18 제출 — 오늘 기준 1일 전, 저자 미상 — 소속 미공개)
- **핵심 아이디어**: 기존 KV 캐시 오프로딩 시스템은 전체 캐시를 호스트 메모리에 저장하고 디코딩 중 중요한 항목만 선택적으로 가져오지만, 스파스성을 더 이상 높일 수 없게 되면 컨텍스트 길이와 배치 크기가 증가함에 따라 KV 전송량이 급증해 디코딩 지연의 주 원인이 되는 구조적 한계가 있다. KVDrive는 이 문제를 "시스템 관점"에서 캐시 배치(placement), 파이프라인 스케줄링(pipeline scheduling), 계층 간 조율(cross-tier coordination)을 공동 최적화해 해결한다. 세 가지 핵심 기술: (1) **어텐션 기반 캐시 관리(Attention-Based Cache Management)** — 트랜스포머 아키텍처에 특화된 어텐션-인식 캐시 관리로 재사용을 극대화하고 불필요한 데이터 이동을 최소화; 슬라이딩 윈도우로 핵심 항목을 GPU HBM에 유지, (2) **파이프라인 스케줄 재구성(Pipeline Restructuring)** — I/O 바운드 단계와 CPU/GPU 컴퓨트 바운드 단계를 오버랩해 이종 자원 간 스톨(stall)을 제거, (3) **3계층 데이터 이동 조율(Cross-Tier Coordination)** — GPU HBM·호스트 DRAM·SSD를 가로지르는 데이터 이동을 조율해 GPU와 DRAM 한계를 훨씬 넘어선 장문 컨텍스트 추론 지원.
- **연구 목표와의 관련성**: high
- **주목할 점**: 최신 기준선 대비 **1.74× 처리량 향상**을 정확도 보존하면서 달성. GPU HBM → DRAM → SSD의 3계층을 전부 아우르는 KV 캐시 관리 시스템이라는 점에서, 기존 Fluxion(이미 보고, Activity C)이 CPU-상주 KV의 스파스 접근에 집중했다면 KVDrive는 SSD까지 포함한 3계층 전체를 통합 스케줄링하는 상위 수준의 시스템이다. Activity A의 "KV 오프로딩 스케줄러 설계"와 "캐시 퇴거율 최소화" 목표를 단일 노드 내에서 가장 완전하게 구현한 최근 사례다. vLLM v0.21.0 HMA(이미 보고)가 스케줄러-HMA 통합을 프레임워크 레벨에서 공식화했다면, KVDrive는 동일한 문제를 스탠드얼론 시스템으로 더 완전하게 구현한 연구 수준의 대응이다 — v0.21 HMA 위에 KVDrive의 어텐션 기반 배치 정책과 파이프라인 재구성 원칙을 이식하는 것이 Activity A의 vLLM 6단계 이식의 자연스러운 방향이다. NAtH(이미 보고, Activity A)의 "퇴거 비율에만 정확도가 의존"하는 이론과 KVDrive의 "어텐션 기반 슬라이딩 윈도우로 핵심 항목 HBM 유지" 원칙이 결합하면, "어텐션 점수가 임계값 이하인 항목을 DDR→SSD 순으로 이동하되 퇴거 비율을 3% 미만으로 제한"하는 정확도-보장 3계층 스케줄러 정책이 설계 가능하다.

#### 2. [ThunderAgent: A Simple, Fast and Program-Aware Agentic Inference System](https://arxiv.org/abs/2602.13692)
- **출처**: arXiv 2602.13692 (2026-02-14, Weili Xu 외 — ThunderAgent-org, 오픈소스: github.com/ThunderAgent-org/ThunderAgent)
- **핵심 아이디어**: 기존 LLM 추론 시스템이 에이전틱 워크플로우에서 개별 요청 단위로 자원을 스케줄링해 워크플로우 전체에 대한 KV 캐시·CPU·원격 도구 환경의 통합 관리가 불가능한 문제를 해결한다. ThunderAgent는 에이전틱 워크플로우를 **LLM Program** 추상화로 모델링한다 — 이 추상화가 KV 캐시, 시스템 상태, 디스크 메모리, 네트워크 포트 같은 외부 도구 자원을 통합된 시각으로 관리할 수 있는 인터페이스를 제공한다. 두 가지 핵심 구성 요소: (1) **프로그램-인식 스케줄러(Program-Aware Scheduler)** — 워크플로우 전체를 에이전트-토큰 단위의 LLM Program으로 분석해 KV 캐시 히트율을 극대화하는 방식으로 스케줄링; 주기적 모니터가 각 GPU 노드에서 활성 프로그램 수를 확인하고 과부하 노드의 프로그램을 일시 중지해 전역 대기 큐로 이동시켜 KV 캐시 쓰래싱 방지, (2) **도구 자원 관리자(Tool Resource Manager)** — 에이전트가 사용할 외부 도구 실행 환경을 비동기적으로 사전 준비(async environment prefetching)해 에이전트 실행 시작과 도구 준비 완료 대기 시간을 겹침.
- **연구 목표와의 관련성**: high
- **주목할 점**: 에이전틱 서빙 시나리오에서 vLLM 대비 **1.5~3.6× 처리량**, 에이전틱 RL 롤아웃 환경에서 **1.8~3.9× 향상**. 프로그램 추상화로 미래 KV 캐시 수요를 사전 인식해 선점적 예약·배치가 가능하다는 점에서, AMPD(이미 보고)의 "어느 노드에서 증분 프리필을 수행할 것인가" 결정과 ThunderAgent의 "어떤 순서로 LLM Program의 스텝을 실행해 KV 히트율을 극대화할 것인가" 결정이 상보적으로 결합 가능하다. PBKV(이미 보고)가 "에이전트 호출 패턴을 예측해 KV를 보존"했다면, ThunderAgent는 "워크플로우 구조 자체를 Program으로 파싱해 스케줄링에 직접 활용"하는 근본적으로 다른 접근이다 — PBKV의 확률적 예측과 ThunderAgent의 정적 Program 분석을 결합하면 "Program이 제공하는 결정적 구조 정보 + 실행 시 불확실성은 PBKV로 보완"하는 하이브리드 스케줄러가 가능하다. KV 캐시 쓰래싱을 방지하기 위해 과부하 노드의 프로그램을 일시 중지하는 메커니즘이 Activity A의 "캐시 퇴거율 최소화"와 직결된다 — 퇴거 직전에 프로그램을 일시 중지하고 메모리를 확보한 후 재개하는 "선점-대기" 패턴은 Valve(이미 보고)의 프로덕션 선점 보장과 동일 원리를 에이전틱 워크플로우로 확장한 것이다.

## Activity B — Non-Contiguous KV Cache Reuse

### 새로운 논문·구현체

이번 주(2026-05-12 ~ 2026-05-19) 기간에 Activity B 범주의 완전히 새로운 논문·구현체는 확인되지 않았다. 직전 주까지 보고된 RelayCaching(2603.13289), ICaRus(2603.13281), QKVShare(2605.03884), KV-Fold(2605.12471), KVM(2605.09877), AdapShot(2605.03644), Louver(2605.06763)가 현재 Activity B의 최전선이다.

ThunderAgent(Activity A 등재)가 LLM Program 추상화로 에이전트 간 KV 캐시 재사용 계획을 사전 수립한다는 점에서, 비연속 세그먼트 재사용 관점에서도 주목할 만하다. ThunderAgent의 Program 파서가 워크플로우 내 어느 스텝에서 이전 스텝의 KV를 재사용할 수 있는지를 정적으로 식별하면, RelayCaching(B)의 동적 재사용 결정 대신 정적 Program 분석으로 비연속 재사용 가능 세그먼트를 사전 예약하는 "정적-동적 하이브리드 비연속 재사용 스케줄러"가 가능하다.

KVDrive(Activity A 등재)의 슬라이딩 윈도우로 핵심 KV 항목을 HBM에 유지하는 원칙이 Activity B의 비연속 세그먼트 재사용에서 "히트 가능성이 높은 세그먼트는 HBM에, 낮은 세그먼트는 DRAM이나 SSD로 이동"하는 히트율-인식 계층적 비연속 세그먼트 스토리지 아키텍처로 확장 가능하다.

## Activity C — KV Cache Compression

### 새로운 논문·구현체

이번 주(2026-05-12 ~ 2026-05-19) 기간에 Activity C 범주의 완전히 새로운 논문·구현체는 확인되지 않았다. 직전 주까지 보고된 SP-KV(2605.14037), "Make Each Token Count"(2605.09649), LookaheadKV(2603.10899), FibQuant(2605.11478), AB-Sparse(2605.12110), Fluxion(2605.07719)가 현재 Activity C의 최전선이다.

KVDrive(Activity A 등재)의 어텐션-기반 캐시 관리 전략이 Activity C 압축 관점에서 갖는 함의는 주목할 만하다. KVDrive의 "어텐션 점수 기반 슬라이딩 윈도우로 핵심 항목 HBM 유지"는 암묵적으로 낮은 어텐션 점수 항목을 DRAM/SSD로 이동시키는 "사실상의 어텐션-가중 토큰 퇴거"이다 — "Make Each Token Count"(C)의 전역 리텐션 게이트와 KVDrive의 슬라이딩 윈도우 관리를 결합하면, 리텐션 게이트 점수로 슬라이딩 윈도우 포함 항목을 결정하는 학습-기반 3계층 스케줄러가 가능하다.

## Activity 간 시너지 포인트

**A + B (ThunderAgent + Activity B 비연속 재사용)**
- ThunderAgent가 LLM Program 정적 분석으로 워크플로우 내 어느 스텝이 이전 스텝의 KV를 재사용할 수 있는지를 사전에 식별하면, Activity B의 비연속 세그먼트 재조합이 실행 시점이 아닌 Program 파싱 시점에 재사용 계획을 수립할 수 있다. RelayCaching(B)의 레이어 범위 프로파일러가 "어떤 레이어의 KV가 재사용 가능한가"를 동적으로 결정한다면, ThunderAgent Program 분석이 "어떤 워크플로우 스텝 간에 KV 재사용이 가능한가"를 정적으로 결정하는 두 수준의 역할 분리가 완성된 "정적-동적 하이브리드 에이전틱 비연속 KV 재사용 파이프라인"이 가능하다.

**A + C (KVDrive + Activity C 압축)**
- KVDrive의 HBM·DRAM·SSD 3계층에 Activity C의 계층별 차등 압축을 결합하면, HBM에는 FP8(정확도 우선), DRAM에는 FibQuant VQ(공간 효율), SSD에는 FibQuant 고압축(10×+)을 각각 적용하는 "계층-차별 압축 + 3계층 KVDrive 스케줄링" 통합이 가능하다. KVDrive의 어텐션-기반 슬라이딩 윈도우가 HBM 잔류 항목을 결정하고, SP-KV의 write-time 희소화가 처음부터 DRAM/SSD행 항목을 줄이면 총 이동 데이터량이 두 요소의 곱으로 절감된다.
- "Make Each Token Count"(C)의 전역 리텐션 게이트가 KVDrive의 슬라이딩 윈도우에서 어떤 항목을 HBM에 유지할지 결정하는 역할을 맡으면, 학습-기반 글로벌 리텐션 정책과 시스템-수준 3계층 관리가 결합된 "학습-기반 KVDrive 캐시 정책"이 가능하다.

**A + B + C (ThunderAgent + KVDrive + 비연속 재사용 + 압축 통합)**
- ThunderAgent Program 분석(A) → 비연속 재사용 가능 세그먼트 사전 식별(B) → KVDrive 3계층 배치 스케줄링(A) → 계층별 FibQuant/SP-KV 차등 압축(C) → RelayCaching 동적 재사용 갱신(B). ThunderAgent가 워크플로우 구조에서 재사용 가능 세그먼트를 식별하고, KVDrive가 해당 세그먼트를 히트 가능성에 따라 HBM/DRAM/SSD에 배치하며, 압축 정책이 계층별 메모리 예산에 맞게 세그먼트를 압축하고, RelayCaching이 실행 중 실제 편차를 검출해 최소 재계산으로 갱신하는 완전한 에이전틱 비연속 KV 최적화 스택.

## idea-generator에게 전달할 포인트

### Activity A
- **KVDrive(2605.18071) — 3계층 어텐션-인식 KV 관리**: Activity A의 vLLM 이식(6단계)에서 KVDrive의 "어텐션-기반 캐시 배치 + 파이프라인 재구성 + 계층 간 조율" 세 원칙이 vLLM v0.21.0 HMA 멀티-커넥터 아키텍처의 자연스러운 구현 목표가 된다. 구체적으로: (1) HMA 커넥터로 DRAM/SSD 각각의 연결 관리, (2) DCP/PCP 오프로딩 커넥터로 KVDrive의 파이프라인 재구성 구현, (3) 어텐션 점수 기반 슬라이딩 윈도우로 HBM 잔류 항목 결정. src/engine/batch_runner.py에 KVDrive 스타일 어텐션-기반 캐시 배치 정책을 프로토타입으로 구현한 후 vLLM v0.21 HMA로 이식하는 2단계 전략을 권장.
- **ThunderAgent(2602.13692) — Program-인식 에이전틱 KV 스케줄링**: Activity A의 에이전틱 서빙 시나리오에서 요청 단위 스케줄링 대신 LLM Program 단위 스케줄링으로 전환하는 것이 KV 히트율 극대화와 캐시 쓰래싱 방지의 핵심 원칙이다. ThunderAgent의 오픈소스 구현(github.com/ThunderAgent-org/ThunderAgent)을 참조해 src/engine/runner.py에 에이전틱 워크플로우를 Program으로 파싱하는 기본 인터페이스를 추가하면, Activity A의 "요청 재정렬 + 배치 우선순위" 스케줄러를 Program-인식 방향으로 발전시키는 단기 구현 경로가 열린다. PBKV(이미 보고)의 확률적 예측 + ThunderAgent의 Program 정적 분석 결합이 에이전틱 KV 스케줄링의 강력한 하이브리드 전략이다.
- **KVDrive + NAtH 결합 기회**: NAtH(이미 보고)의 "정확도는 퇴거 비율에만 의존" 이론 + KVDrive의 어텐션-기반 슬라이딩 윈도우를 결합하면, 슬라이딩 윈도우에서 제외된 항목을 "퇴거(영구 제거) vs DDR 오프로딩 vs SSD 오프로딩" 중 어느 경로로 보낼지를 어텐션 점수와 퇴거 비율 제약으로 결정하는 "정확도-보장 3계층 KVDrive+NAtH 스케줄러"를 Activity A 단기 구현 목표로 설정할 것.
