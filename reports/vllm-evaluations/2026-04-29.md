## Activity A — Cache-Hit-Aware Scheduling (vLLM port, NEW)

| Metric | Target | Measured | Status |
|--------|--------|----------|--------|
| Warm request scheduled before cold | Yes | **Yes** | ✅ PASS |
| 100-request queue construction time | < 200ms | **4.9ms** | ✅ PASS |

`CacheHitAwareRequestQueue` correctly prioritises a warm request (tokens
matching a cached template) over a cold request. Queue construction for
100 requests takes 4.9ms — well within the scheduling overhead budget.

**Priority formula** (same as standalone):
```
priority = hit_rate × (1 − min(wait_steps / fairness_max_wait, 1.0))
```

Fairness: cold requests that wait ≥ `fairness_max_wait` rounds are promoted
to the front via tie-breaking on wait steps.

## Limitations and Production Notes

1. **Non-contiguous reuse is informational in this port**: `get_computed_blocks`
   reports NC hits but does not yet skip prefill for cached chunks. Full prefill
   skipping requires cooperation from the attention kernel (model-specific).
   The `NonContiguousAttentionWrapper` provides the API hooks but is not wired
   into any specific model.

2. **HadamardInt4Codec per-row scales**: Each encode call stores per-row scale
   tensors in memory (`_scales` dict). In long-running production servers, this
   dict grows proportionally to `num_unique_(layer_idx, tensor_id)` pairs
   processed. Periodic pruning may be needed for very long-running deployments.

3. **`CacheHitAwareRequestQueue.pop_request` re-sorts**: The current
   implementation re-sorts the queue on every pop to update wait steps. This is
   O(n log n) per pop, acceptable for typical batch sizes (< 256 requests) but
   should be replaced with a proper heap for high-concurrency workloads.
