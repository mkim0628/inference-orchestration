## 미해결 이슈

### [FAIL] KV Memory Reduction 실측 0% (Activity C — 높음)

**파일**: `vllm_integration/compression_codec.py`, `vllm_integration/attention_backend_patch.py`

**문제**: `DPAttentionAwareVllmCodec.compression_hook()`과 `DPAttentionAwareCompressionAttentionHook.write_to_cache()`가 INT8 양자화 결과를 FP16으로 재패킹(`(q.float() * scale).half()`)해 반환한다. 텐서 shape·dtype이 원본과 동일하므로 실제 저장 바이트가 줄지 않는다.

**수정 방향**:
1. (권장) INT8 텐서 직접 저장 + scale을 별도 side-channel 저장: `return q.to(torch.int8), scale` 형태로 반환하여 실제 2× 바이트 감소 구현.
2. (대안) 토큰 수 기반 eviction (budget_ratio 방식) 추가로 실제 stored token 수를 줄여 실측 30%+ 달성.

**관련 항목**: `memory_reduction_ratio()` 메서드가 현재 항상 0.0을 반환함.

---

### [경고] load_and_reencode_b18 인자 오용 위험 (Activity B — 중간)

**파일**: `vllm_integration/block_manager_patch.py`, 함수 `load_and_reencode_b18`

**문제**: 시그니처가 `source_position: int, target_position: int`를 기대하지만, 평가 중 리스트를 전달하면 `TypeError: unsupported operand type(s) for -: 'list' and 'list'`가 발생한다. 기존 `_adapshot_rope_reencode_b18`은 단일 정수 스칼라 위치만 처리한다. 실제 사용자가 position range(시작·끝 위치)를 전달하거나 배치 처리를 시도할 때 오류가 발생할 수 있다.

**수정 방향**: 함수 시그니처에 타입 힌트를 명확히 하고, docstring에 `int` 스칼라임을 명시. 배치 처리가 필요하면 `vmap` 또는 루프 처리 추가.

---

### [경고] memory_reduction_ratio() 0.0 반환 (Activity C)

**파일**: `vllm_integration/attention_backend_patch.py`, line 4974-4978

**문제**: `_total_bytes_stored == _total_bytes_original` (FP16 재패킹)이어서 항상 0.0 반환. 내부 smoke test에서 "expect ~0.0 for FP16 shape-preserving"으로 명시적으로 0.0을 기대하고 있어 의도된 동작이지만, 평가 항목 KV Memory Reduction >= 30%에는 미달.

## 개선 권고사항

1. **INT8 실제 저장 구현** (Priority: High): `write_to_cache()`/`compression_hook()`이 INT8 텐서를 그대로 반환하도록 변경. block table 호환성 유지를 위해 FP16-as-INT8 표현 대신 int8 dtype과 scale을 별도 저장하는 두-버퍼 패턴 사용. 이를 통해 실측 50% KV 메모리 감소 달성 가능.

2. **load_and_reencode_b18 인자 검증 추가** (Priority: Medium): 함수 진입부에서 `isinstance(source_position, int)` 검증 추가.

3. **통합 smoke test 업데이트** (Priority: Low): `test_integration_smoke.py`가 2026-05-02 사이클 기준으로 작성됨. 2026-05-18 A+B+C 컴포넌트용 테스트 추가 권장.

4. **다음 vLLM 버전 주의사항**: vLLM 0.21.0에서 `vllm.v1.core.sched.scheduler.Scheduler.waiting` 속성이 현재 유지되고 있으나, vLLM v1 API가 alpha 상태이므로 0.22.x에서 구조 변경 가능성 있음. `scheduler_patch.py`의 `self.waiting` 직접 접근 부분에 try/except 방어 코드 추가 권장.
